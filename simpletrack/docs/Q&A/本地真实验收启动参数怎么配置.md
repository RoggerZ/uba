# 本地真实验收启动参数怎么配置

## 问题

这次在本机对 `SimpleTrack SaaS` 做真实验收时，`localhost:3005`、`analytics-service`、Redis、MySQL、ClickHouse 分别是怎么启动的？参数应该怎么配？

## 短答案

这次真实验收不是只启动 `src/simpletrack-saas` 一个前端服务，而是把下面 4 层一起接起来：

1. `simpletrack-saas`：SaaS 控制面，跑在 `http://localhost:3005`
2. `analytics-service`：collect + readback 运行时，跑在 `http://127.0.0.1:8080`
3. Redis / MySQL / ClickHouse：给 `analytics-service` 做 durable queue、checkpoint 和 readback 存储
4. 真实 browser + 真实 collect：用真实账号登录 SaaS，创建真实 website source 和 goal，再发真实事件，最后在 Realtime / Events / Goals 页面验收

如果只启动 SaaS，不启动 `analytics-service` 和存储依赖，页面会退回到：

- `Dashboard readback partially unavailable`
- `Realtime readback unavailable`
- `Events readback unavailable`

这不算 P1 里的“数据管道活了”。

## 这次真实验收用到的 SaaS 本地配置

文件：`src/simpletrack-saas/.env.local`

本次实际使用的是下面这些关键项：

```env
DATABASE_URL="postgresql://postgres:postgres@127.0.0.1:5432/simpletrack"
DIRECT_URL="postgresql://postgres:postgres@127.0.0.1:5432/simpletrack"

NEXT_PUBLIC_SAAS_URL="http://localhost:3005"
NEXT_PUBLIC_MARKETING_URL="http://localhost:3001"
NEXT_PUBLIC_DOCS_URL="http://localhost:3001"

AUTH_REQUIRE_EMAIL_VERIFICATION="0"
AUTH_RATE_LIMIT_ENABLED="0"

SIMPLETRACK_ANALYTICS_RUNTIME_SOURCE_TOKEN="runtime-service-token"
SIMPLETRACK_ANALYTICS_SERVICE_URL="http://127.0.0.1:8080"
SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN="query-service-token"
```

### 这些参数分别是干什么的

- `NEXT_PUBLIC_SAAS_URL`
  - 必须和这次实际访问的端口一致
  - 这次验收是 `3005`，所以必须是 `http://localhost:3005`
- `AUTH_REQUIRE_EMAIL_VERIFICATION="0"`
  - 本地真实验收时关闭邮箱验证，否则真实登录可能卡在 `EMAIL_NOT_VERIFIED`
- `AUTH_RATE_LIMIT_ENABLED="0"`
  - 本地多轮真实登录 / 注册 / 回归时，避免 Better Auth 本地 rate limit 干扰验收
- `SIMPLETRACK_ANALYTICS_RUNTIME_SOURCE_TOKEN`
  - 给 `analytics-service` 调 SaaS 内部 `runtime-source` API 用
- `SIMPLETRACK_ANALYTICS_SERVICE_URL`
  - 告诉 SaaS 的 readback helper 去哪里读 Realtime / Events / Goals
- `SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN`
  - SaaS 调 `analytics-service` 内部 readback 接口时带的 bearer token

## 第 3 步：从 SaaS 取 runtime-source 配置

为什么要多做这一步？

因为当前 `analytics-service` 在 `INGESTION_ENABLED=true` 时，不仅要走 HTTP resolver，还要求启动时有一份 `ANALYTICS_SERVICE_SOURCES_JSON` 作为 schema surface。换句话说：

- `ANALYTICS_SERVICE_SOURCE_RESOLVER='http'` 负责运行时按 `write_key` 去 SaaS 查 source
- `ANALYTICS_SERVICE_SOURCES_JSON` 负责启动时告诉服务“当前允许哪些 source 表结构进入 ingestion / readback 面”

### 取法

先在 SaaS 里创建真实 source，拿到真实 `write_key`，然后调用：

```powershell
$body = @{ write_key = 'wk_xxx' } | ConvertTo-Json -Compress
Invoke-RestMethod `
  -Method Post `
  -Uri 'http://127.0.0.1:3005/api/internal/analytics/runtime-source' `
  -Headers @{ Authorization = 'Bearer runtime-service-token' } `
  -ContentType 'application/json' `
  -Body $body
```

返回结果里会有：

- `tenant_id`
- `project_id`
- `source_id`
- `allowed_origins`
- `readback_policy`
- `session_salt`
- `visit_salt`
- `visit_window_seconds`
- `client_hash_salt`

把这整条 source 配置塞进 `ANALYTICS_SERVICE_SOURCES_JSON`。

## 第 5 步：为什么这些参数能对齐 P1

P1 这里要验证的不是“页面能打开”，而是：

1. 真实登录能进控制面
2. 真实 website source 能创建
3. 真实 write key 能拿到
4. 真实 collect 能成功
5. 真实 readback 能在 Realtime / Events / Goals 页面看到

这次启动参数就是围绕这 5 件事配的。

其中最关键的是：

- `NEXT_PUBLIC_SAAS_URL=http://localhost:3005`
  - 保证本地 browser / auth / redirect / runtime origin 一致
- `AUTH_REQUIRE_EMAIL_VERIFICATION=0`
  - 保证本地真实验收不会被邮箱验证流程卡住
- `SIMPLETRACK_ANALYTICS_SERVICE_URL=http://127.0.0.1:8080`
  - 保证 SaaS 读侧不是假数据，而是去真实 analytics-service
- `ANALYTICS_SERVICE_QUERY_ENABLED=true`
  - 保证 `analytics-service` 真能提供 `/v1/realtime`、`/v1/events`、`/v1/goals`
- `ANALYTICS_SERVICE_INGESTION_ENABLED=true`
  - 保证 `/collect` 不是只返回 200，而是真的把事件推进到存储里

## 常见坑位

### 1. `NEXT_PUBLIC_SAAS_URL` 还是 `3000`

现象：

- 登录或跳转异常
- 本地真实验收和运行时 origin 不一致

处理：

```env
NEXT_PUBLIC_SAAS_URL="http://localhost:3005"
```

### 2. 登录时报 `EMAIL_NOT_VERIFIED`

现象：

- `POST /api/auth/sign-in/email` 返回 `403`
- 响应体里有 `EMAIL_NOT_VERIFIED`

处理：

```env
AUTH_REQUIRE_EMAIL_VERIFICATION="0"
AUTH_RATE_LIMIT_ENABLED="0"
```

### 3. 页面一直显示 `Live analytics is not connected for this workspace yet`

现象：

- Dashboard / Realtime / Events 全是 unavailable

处理方向：

1. 先看 `analytics-service` 是否启动
2. 再看 `healthz` 是否返回 `ok`
3. 再看 SaaS 是否配置了：
   - `SIMPLETRACK_ANALYTICS_SERVICE_URL`
   - `SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN`
4. 再看 `analytics-service` 是否同时拿到了：
   - `ANALYTICS_SERVICE_QUERY_ENABLED=true`
   - `ANALYTICS_SERVICE_QUERY_TOKEN`
   - `ANALYTICS_SERVICE_SOURCES_JSON`
   - Redis / MySQL / ClickHouse

### 4. `analytics-service` 启动时报 `ANALYTICS_SERVICE_SOURCES_JSON is required when ingestion is enabled`

现象：

- 虽然配了 HTTP resolver，服务还是起不来

原因：

- 当前实现里，同进程 ingestion 启动时仍要求一份启动 schema surface

处理：

- 先从 SaaS 的 `runtime-source` API 取当前 source 配置
- 再把返回结果写进 `ANALYTICS_SERVICE_SOURCES_JSON`

3. 记录这个 source 的真实 `write key`

```bash
WRITE_KEY='wk_xxx'
```

### 2. 通过 SaaS 的 runtime-source API 拉取 source 配置

```bash
export WRITE_KEY='wk_xxx'

RUNTIME_SOURCE_JSON="$(
  curl -sS \
    -X POST 'http://127.0.0.1:3005/api/internal/analytics/runtime-source' \
    -H 'Authorization: Bearer runtime-service-token' \
    -H 'Content-Type: application/json' \
    -d "{\"write_key\":\"${WRITE_KEY}\"}"
)"

echo "${RUNTIME_SOURCE_JSON}" | jq .
```

如果这里返回的不是一条带 `tenant_id / project_id / source_id / readback_policy / session_salt / visit_salt` 的 JSON，就不要继续往下启动 `analytics-service`。

## Linux 版本里最容易踩的坑

### 1. 没装 `jq`

现象：

- `RUNTIME_SOURCE_JSON` 拿到了，但后续脚本不好拼

处理：

```bash
sudo apt-get update
sudo apt-get install -y jq
```

### 2. `ANALYTICS_SERVICE_SOURCES_JSON` 没包成数组

错误写法：

```bash
export ANALYTICS_SERVICE_SOURCES_JSON="${RUNTIME_SOURCE_JSON}"
```

当前这套启动最好写成：

```bash
export ANALYTICS_SERVICE_SOURCES_JSON="[$(echo "${RUNTIME_SOURCE_JSON}" | jq -c .)]"
```

### 3. `allowed_origins` 和 collect 的 `Origin` 不匹配

如果 runtime-source 返回的 `allowed_origins` 不包含你发 collect 时带的 `Origin`，请求会被拒绝。

最稳的做法是：

- 先看 `runtime-source` 返回的 `allowed_origins`
- collect 时就用同样的 origin

### 4. MySQL 端口和本机已有服务冲突

如果 `23306` 已被占用，就像这次 Windows 验收一样：

- `analytics-core` 启动时把容器端口改到 `23307`
- `analytics-service` 的 `ANALYTICS_SERVICE_MYSQL_DSN` 也同步改成 `23307`

不要只改一边。
