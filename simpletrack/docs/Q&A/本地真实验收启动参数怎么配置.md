# 本地真实验收启动参数怎么配置

## 问题

这次在本机对 `SimpleTrack SaaS` 做真实验收时，`localhost:3005`、`analytics-service`、Redis、MySQL、ClickHouse 分别是怎么启动的？参数应该怎么配？

## 短答案

这轮已经不再推荐“手工分别起 SaaS + analytics-service + analytics-core 依赖”的旧方式作为主路径，而是优先使用 `src/deploy/local/docker/` 下的 **layered Docker Compose** 栈。

当前本机真实验收实际跑通的是下面这一整套：

1. `saas`：SaaS 控制面，当前宿主机入口 `http://localhost:33105`
2. `marketing`：公开产品入口，当前宿主机入口 `http://localhost:33104`
3. `docs`：公开文档入口，当前宿主机入口 `http://localhost:33102`
4. `analytics-service`：collect + readback 运行时，当前宿主机入口 `http://localhost:38180`
5. Redis / MySQL / ClickHouse / Kafka / Postgres / MinIO：都由同一个 Docker 网络里的独立容器承接
6. 真实 browser + 真实 collect：真实注册、真实 organization、真实 website source、真实 goal、真实 collect、真实 Realtime / Events / Goals / Dashboard 页面验收

如果只起 SaaS，不起 `analytics-service` 和底层依赖，页面仍然会退回到：

- `Dashboard readback partially unavailable`
- `Realtime readback unavailable`
- `Events readback unavailable`

这不算 P1 里的“数据管道活了”。

## 这轮 Docker 真实验收里实际发生了什么

这轮不是截图验收，也不是只做 HTTP 探活，而是完成了下面这条真实闭环：

1. 真实浏览器访问 `http://localhost:33105/login`
2. 真实注册新账号
3. 真实创建 organization
4. 真实创建 website source，拿到真实 `write key`
5. 真实创建 goal
6. 真实发送 3 条 collect 事件：
   - `pageview`
   - `cta_click`
   - `ultraqa_activation_<timestamp>`
7. 真实调用 readback：
   - `/v1/realtime`
   - `/v1/events`
   - `/v1/goals`
8. 真实浏览器重新打开：
   - Dashboard
   - Websites
   - Realtime
   - Events
   - Goals

最终页面证据是：

- Dashboard：`Active visitors=1`、`Events seen=3`、`Pageviews=1`、`Activation goal=1`
- Realtime：`Install health=Connected`
- Events：3 条真实 raw events 可见
- Goals：目标从 `NO DATA` 变成 `HAS DATA`

本轮还真实测出并修掉了一个应用问题：新建 source 在尚无 routed table 时，ClickHouse 读侧会直接报 `Unknown table expression identifier ... code: 60`，从而把 Dashboard / Realtime / Events / Goals 打成 `internal server error`。现在 `analytics-core` 已补读侧 fail-soft：缺 routed table 时按空结果 / 零计数处理，不再把“新 source 还没数据”炸成 500。

## 当前 Docker 方案下的关键环境口径

Docker 分层栈当前统一读 `src/deploy/local/docker/.env.dev`。

这台机器上真实验收通过的关键 public URL 是：

```env
SIMPLETRACK_PUBLIC_SAAS_URL=http://localhost:33105
SIMPLETRACK_PUBLIC_MARKETING_URL=http://localhost:33104
SIMPLETRACK_PUBLIC_DOCS_URL=http://localhost:33102
```

`analytics-service` 运行时的关键 token 口径是：

```env
SIMPLETRACK_RUNTIME_SOURCE_TOKEN=runtime-service-token
SIMPLETRACK_QUERY_TOKEN=query-service-token
```

对应地，容器内应用互联路径是：

- `saas -> http://analytics-service:8080`
- `analytics-service -> http://saas:3000/api/internal/analytics/runtime-source`

这轮 Docker 栈还做了两件很关键的收口：

1. `analytics-service` 在 `ANALYTICS_SERVICE_SOURCE_RESOLVER=http` 且 `INGESTION_ENABLED=true` 时，不再要求人为先准备静态 source 列表才能启动本地 Docker 开发闭环。
2. `analytics-core` 读侧在新 source 尚无 routed table 时，不再把 readback 打成 500，而是返回 empty/no-data 路径，让页面保持可验收的空态。

## 手工方式为什么现在降级为次选

之前这份文档记录的是 Windows 上一套“非 Docker、逐个进程手工起”的验收方法。它仍然能帮助理解参数结构，但现在不再是主推荐路径，原因有两个：

1. 端口更容易和本机已有服务冲突
2. `analytics-service`、SaaS、marketing、docs、依赖库之间的 host/container 地址口径更容易漂

当前已经有了可复用的 Docker 网络、分层 compose 和 helper 脚本，所以本地真实验收优先直接复用 Docker 栈。

下面保留旧的手工启动说明，主要作为参数含义参考，而不是当前首选运行方式。

SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN="query-service-token"

### 这些参数分别是干什么的

- `NEXT_PUBLIC_SAAS_URL`
  - 必须和这次实际访问的端口一致
  - 这次验收是 `3005`，所以必须是 `http://localhost:3005`
- `AUTH_REQUIRE_EMAIL_VERIFICATION="0"`
  - 本地真实验收时关闭邮箱验证，否则真实登录可能卡在 `EMAIL_NOT_VERIFIED`
- `AUTH_RATE_LIMIT_ENABLED="0"`
  - 本地多轮真实登录 / 注册 / 回归时，避免 Better Auth 本地 rate limit 干扰验收
- `SIMPLETRACK_ANALYTICS_RUNTIME_SOURCE_TOKEN`
  - 给 `analytics-service` 调 SaaS 内部 `runtime-source` API 用
- `SIMPLETRACK_ANALYTICS_SERVICE_URL`
  - 告诉 SaaS 的 readback helper 去哪里读 Realtime / Events / Goals
- `SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN`
  - SaaS 调 `analytics-service` 内部 readback 接口时带的 bearer token

## 第 3 步：从 SaaS 取 runtime-source 配置

为什么还会提到这一步？

因为它能解释 runtime-source 是怎么接到 `analytics-service` 上的，但要注意：

- **旧口径**：需要手工把 runtime-source 结果包成 `ANALYTICS_SERVICE_SOURCES_JSON`
- **当前 Docker 口径**：HTTP resolver + 本地 Docker ingestion 已经不再要求你先手工准备这一份静态 schema surface 才能形成开发闭环

也就是说，runtime-source API 现在主要负责运行时真源配置，而不是要求操作者手工回填启动材料。

### 取法

先在 SaaS 里创建真实 source，拿到真实 `write_key`，然后调用：

```powershell
$body = @{ write_key = 'wk_xxx' } | ConvertTo-Json -Compress
Invoke-RestMethod `
  -Method Post `
  -Uri 'http://127.0.0.1:3005/api/internal/analytics/runtime-source' `
  -Headers @{ Authorization = 'Bearer runtime-service-token' } `
  -ContentType 'application/json' `
  -Body $body
```

返回结果里会有：

- `tenant_id`
- `project_id`
- `source_id`
- `allowed_origins`
- `readback_policy`
- `session_salt`
- `visit_salt`
- `visit_window_seconds`
- `client_hash_salt`

旧的手工启动方式里，会把这整条 source 配置塞进 `ANALYTICS_SERVICE_SOURCES_JSON`。

当前 Docker 主路径下，不再把这一步作为必须人工执行的前置条件。

## 常见坑位

### 1. `NEXT_PUBLIC_SAAS_URL` 还是 `3000`

现象：

- 登录或跳转异常
- 本地真实验收和运行时 origin 不一致

处理：

```env
NEXT_PUBLIC_SAAS_URL="http://localhost:3005"
```

### 2. 登录时报 `EMAIL_NOT_VERIFIED`

现象：

- `POST /api/auth/sign-in/email` 返回 `403`
- 响应体里有 `EMAIL_NOT_VERIFIED`

处理：

```env
AUTH_REQUIRE_EMAIL_VERIFICATION="0"
AUTH_RATE_LIMIT_ENABLED="0"
```

### 3. 页面一直显示 `Live analytics is not connected for this workspace yet`

现象：

- Dashboard / Realtime / Events 全是 unavailable

处理方向：

1. 先看 `analytics-service` 是否启动
2. 再看 `healthz` 是否返回 `ok`
3. 再看 SaaS 是否配置了：
   - `SIMPLETRACK_ANALYTICS_SERVICE_URL`
   - `SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN`
4. 再看 `analytics-service` 是否同时拿到了：
   - `ANALYTICS_SERVICE_QUERY_ENABLED=true`
   - `ANALYTICS_SERVICE_QUERY_TOKEN`
   - `ANALYTICS_SERVICE_SOURCES_JSON`
   - Redis / MySQL / ClickHouse

### 4. `analytics-service` 启动时报 `ANALYTICS_SERVICE_SOURCES_JSON is required when ingestion is enabled`

现象：

- 虽然配了 HTTP resolver，服务还是起不来

原因：

- 这是旧口径或旧运行方式下的症状；当前 Docker 主路径已经不应再以这条报错作为常态前置

处理：

- 如果你还在沿用旧的非 Docker 手工启动链路，可以先从 SaaS 的 `runtime-source` API 取当前 source 配置，再把返回结果写进 `ANALYTICS_SERVICE_SOURCES_JSON`
- 如果你走的是当前推荐的 Docker 分层栈，就不应该把这一步当作必须人工回填的日常前置

### 5. 新 source 刚创建就出现 `internal server error`

现象：

- Goals / Realtime / Events / Dashboard 刚建完 source 时显示 `internal server error`
- `analytics-service` 日志里有 `Unknown table expression identifier ... code: 60`

原因：

- 旧行为下，新 source 还没有对应 ClickHouse routed table，读侧直接把缺表抛成 500

当前状态：

- 这轮已经在 `analytics-core` 里补了 fail-soft 修复
- 缺 routed table 现在应该走 empty / no-data 路径，而不是把页面炸成 500

### 6. `/collect` 返回 202 但没有出数

现象：

- `/collect` 返回 `202`
- 但 body 里带 `filtered: true`
- 页面仍然没有数据

原因：

- 这通常不是“已经写入成功”，而是 collect 在 pre-queue stage 被过滤了
- 这轮真实验收里就遇到过一次，服务端日志明确写的是 `reason=bot user agent`

处理方向：

- 先看 `analytics-service` 日志里 collect 是否被 `bot/internal traffic` 过滤
- 验收流量要尽量贴近真实浏览器 UA / Origin，不要拿会被默认 bot token 命中的流量去冒充真实接入成功

## Linux 可直接复制执行的启动脚本版

下面给一套更适合 Linux 的直接复制版。

为了避免把多个长跑进程塞进一个脚本后不好看日志，这里推荐用 **3 个终端** 启动：

1. 终端 A：起 `analytics-core` 依赖容器
2. 终端 B：起 `simpletrack-saas`
3. 终端 C：起 `analytics-service`

### 使用前提

这套脚本默认你已经具备：

- `docker` / `docker compose`
- `node` / `corepack` / `pnpm`
- `go`
- `curl`
- `jq`

并且工作区根目录是：

```bash
~/src/uba
```

如果你的本地目录不是这个，请把下面脚本里的 `~/src/uba` 换成你自己的实际路径。

---

## 终端 A：启动 Redis / MySQL / ClickHouse

### 默认端口版

```bash
cd ~/src/uba/src/analytics-core
docker compose up -d redis mysql clickhouse
```

### 如果 `23306` 被占用

这次 Windows 机器上就是因为 `23306` 被占用，才临时改成了 `23307`。  
Linux 上如果也遇到同样问题，可以这样启动：

```bash
cd ~/src/uba/src/analytics-core
export ANALYTICS_CORE_MYSQL_PORT=23307
docker compose up -d mysql clickhouse
docker compose up -d redis
```

### 健康检查

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}' | grep analytics-core
```

预期至少能看到：

- `analytics-core-redis`
- `analytics-core-mysql`
- `analytics-core-clickhouse`

## 终端 B：启动 simpletrack-saas

### 1. 写入本地验收用 `.env.local`

```bash
cd ~/src/uba/src/simpletrack-saas

cat > .env.local <<'EOF'
DATABASE_URL="postgresql://postgres:postgres@127.0.0.1:5432/simpletrack"
DIRECT_URL="postgresql://postgres:postgres@127.0.0.1:5432/simpletrack"

NEXT_PUBLIC_MARKETING_URL="http://localhost:3001"
NEXT_PUBLIC_SAAS_URL="http://localhost:3005"
NEXT_PUBLIC_DOCS_URL="http://localhost:3001"
NEXT_PUBLIC_AVATARS_BUCKET_NAME="avatars"

BETTER_AUTH_SECRET="simpletrack-local-dev-secret"
AUTH_REQUIRE_EMAIL_VERIFICATION="0"
AUTH_RATE_LIMIT_ENABLED="0"

SIMPLETRACK_ANALYTICS_RUNTIME_SOURCE_TOKEN="runtime-service-token"
SIMPLETRACK_ANALYTICS_SERVICE_URL="http://127.0.0.1:8080"
SIMPLETRACK_ANALYTICS_SERVICE_QUERY_TOKEN="query-service-token"

MAIL_FROM="noreply@example.com"
PRICE_ID_PRO_MONTHLY="price_local_monthly"
PRICE_ID_PRO_YEARLY="price_local_yearly"
PRICE_ID_LIFETIME="price_local_lifetime"
EOF
```

### 2. 生产构建

```bash
cd ~/src/uba/src/simpletrack-saas
corepack pnpm exec dotenv -e .env.local -- corepack pnpm --filter saas run build
```

### 3. 生产启动

```bash
cd ~/src/uba/src/simpletrack-saas/apps/saas
corepack pnpm exec dotenv -e ../../.env.local -- next start --port 3005
```

### 4. 健康检查

```bash
curl -I http://127.0.0.1:3005/login
```

如果能看到 `200` 或 `307/308` 之类可接受跳转，说明 SaaS 已经起来了。

## 终端 C：启动 analytics-service

### 1. 先在浏览器里做一次控制面准备

先打开：

```text
http://localhost:3005/login
```

登录真实账号：

- Email: `simpletrack.local@example.com`
- Password: `CodexTest!123`

然后：

1. 创建一个真实 organization
2. 在 `Websites` 页面创建一个真实 source
3. 记录这个 source 的真实 `write key`

下面脚本里把它记成：

```bash
WRITE_KEY='wk_xxx'
```

### 2. 通过 SaaS 的 runtime-source API 拉取 source 配置

```bash
export WRITE_KEY='wk_xxx'

RUNTIME_SOURCE_JSON="$(
  curl -sS \
    -X POST 'http://127.0.0.1:3005/api/internal/analytics/runtime-source' \
    -H 'Authorization: Bearer runtime-service-token' \
    -H 'Content-Type: application/json' \
    -d "{\"write_key\":\"${WRITE_KEY}\"}"
)"

echo "${RUNTIME_SOURCE_JSON}" | jq .
```

如果这里返回的不是一条带 `tenant_id / project_id / source_id / readback_policy / session_salt / visit_salt` 的 JSON，就不要继续往下启动 `analytics-service`。

### 3. 启动 analytics-service

#### MySQL 用默认 `23306` 的版本

```bash
cd ~/src/uba/src/analytics-service

export ANALYTICS_SERVICE_ADDR='127.0.0.1:8080'
export ANALYTICS_SERVICE_REDIS_ADDR='127.0.0.1:26379'

export ANALYTICS_SERVICE_SOURCES_JSON="[$(echo "${RUNTIME_SOURCE_JSON}" | jq -c .)]"

export ANALYTICS_SERVICE_SOURCE_RESOLVER='http'
export ANALYTICS_SERVICE_CONTROL_PLANE_URL='http://127.0.0.1:3005/api/internal/analytics/runtime-source'
export ANALYTICS_SERVICE_CONTROL_PLANE_TOKEN='runtime-service-token'
export ANALYTICS_SERVICE_CONTROL_PLANE_ALLOW_INSECURE_LOOPBACK='true'

export ANALYTICS_SERVICE_QUERY_ENABLED='true'
export ANALYTICS_SERVICE_QUERY_TOKEN='query-service-token'
export ANALYTICS_SERVICE_QUERY_TOKEN_SCOPES='realtime,events,properties,goals'

export ANALYTICS_SERVICE_INGESTION_ENABLED='true'
export ANALYTICS_SERVICE_MYSQL_DSN='analytics_core:analytics_core@tcp(127.0.0.1:23306)/analytics_core?parseTime=true'
export ANALYTICS_SERVICE_MYSQL_AUTO_MIGRATE='true'

export ANALYTICS_SERVICE_CLICKHOUSE_ADDR='127.0.0.1:29000'
export ANALYTICS_SERVICE_CLICKHOUSE_DATABASE='analytics_core'
export ANALYTICS_SERVICE_CLICKHOUSE_USER='analytics_core'
export ANALYTICS_SERVICE_CLICKHOUSE_PASSWORD='analytics_core'
export ANALYTICS_SERVICE_CLICKHOUSE_AUTO_MIGRATE='true'

go run ./cmd/simpletrack-anaysitics-service
```

#### MySQL 临时改到 `23307` 的版本

如果你像这次验收一样把 `analytics-core` 的 MySQL 临时起在 `23307`，就只改这一行：

```bash
export ANALYTICS_SERVICE_MYSQL_DSN='analytics_core:analytics_core@tcp(127.0.0.1:23307)/analytics_core?parseTime=true'
```

其余参数保持不变。

### 4. 健康检查

```bash
curl -sS http://127.0.0.1:8080/healthz
```

预期返回：

```text
ok
```

## Linux 下的真实 collect 验证脚本

当 SaaS 和 `analytics-service` 都起来以后，可以直接发 3 条真实事件：

```bash
NOW="$(date -u +%Y-%m-%dT%H:%M:%SZ)"

curl -sS -X POST 'http://127.0.0.1:8080/collect' \
  -H 'Origin: https://example.com' \
  -H 'Content-Type: application/json' \
  -d "{
    \"write_key\":\"${WRITE_KEY}\",
    \"id\":\"evt_linux_pageview_1\",
    \"event_name\":\"pageview\",
    \"distinct_id\":\"visitor_linux_1\",
    \"event_time\":\"${NOW}\",
    \"properties\":{\"path\":\"/pricing\"}
  }" | jq .

curl -sS -X POST 'http://127.0.0.1:8080/collect' \
  -H 'Origin: https://example.com' \
  -H 'Content-Type: application/json' \
  -d "{
    \"write_key\":\"${WRITE_KEY}\",
    \"id\":\"evt_linux_click_1\",
    \"event_name\":\"cta_click\",
    \"distinct_id\":\"visitor_linux_1\",
    \"event_time\":\"${NOW}\",
    \"properties\":{\"button\":\"start-trial\"}
  }" | jq .

curl -sS -X POST 'http://127.0.0.1:8080/collect' \
  -H 'Origin: https://example.com' \
  -H 'Content-Type: application/json' \
  -d "{
    \"write_key\":\"${WRITE_KEY}\",
    \"id\":\"evt_linux_goal_1\",
    \"event_name\":\"ultraqa_activation\",
    \"distinct_id\":\"visitor_linux_1\",
    \"event_time\":\"${NOW}\",
    \"properties\":{\"plan\":\"pro\"}
  }" | jq .
```

## Linux 下的 readback 验证脚本

```bash
FROM="$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ)"
TO="$(date -u -d '1 hour' +%Y-%m-%dT%H:%M:%SZ)"

curl -sS \
  -H 'Authorization: Bearer query-service-token' \
  "http://127.0.0.1:8080/v1/realtime?write_key=${WRITE_KEY}" | jq .

curl -sS \
  -H 'Authorization: Bearer query-service-token' \
  "http://127.0.0.1:8080/v1/events?write_key=${WRITE_KEY}&from=${FROM}&to=${TO}" | jq .

curl -sS \
  -H 'Authorization: Bearer query-service-token' \
  "http://127.0.0.1:8080/v1/goals?write_key=${WRITE_KEY}&event_name=ultraqa_activation&from=${FROM}&to=${TO}" | jq .
```

如果这 3 个接口都能看到刚才那几条事件，再回浏览器看：

- 首页
- Realtime
- Events
- Goals

就能验证“数据管道活了”这件事不是假 ready。

## Linux 版本里最容易踩的坑

### 1. 没装 `jq`

现象：

- `RUNTIME_SOURCE_JSON` 拿到了，但后续脚本不好拼

处理：

```bash
sudo apt-get update
sudo apt-get install -y jq
```

### 2. `ANALYTICS_SERVICE_SOURCES_JSON` 没包成数组

错误写法：

```bash
export ANALYTICS_SERVICE_SOURCES_JSON="${RUNTIME_SOURCE_JSON}"
```

当前这套启动最好写成：

```bash
export ANALYTICS_SERVICE_SOURCES_JSON="[$(echo "${RUNTIME_SOURCE_JSON}" | jq -c .)]"
```

### 3. `allowed_origins` 和 collect 的 `Origin` 不匹配

如果 runtime-source 返回的 `allowed_origins` 不包含你发 collect 时带的 `Origin`，请求会被拒绝。

最稳的做法是：

- 先看 `runtime-source` 返回的 `allowed_origins`
- collect 时就用同样的 origin

### 4. MySQL 端口和本机已有服务冲突

如果 `23306` 已被占用，就像这次 Windows 验收一样：

- `analytics-core` 启动时把容器端口改到 `23307`
- `analytics-service` 的 `ANALYTICS_SERVICE_MYSQL_DSN` 也同步改成 `23307`

不要只改一边。
