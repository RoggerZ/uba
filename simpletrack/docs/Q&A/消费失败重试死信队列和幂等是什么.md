# 消费失败、重试、死信队列和幂等是什么

## Q：你说的“重试”具体是什么意思？

A：这里的“重试”不是说 Kafka 或 Redis 自己凭空把同一条消息复制很多份，而是指：**消费端没有确认成功处理，系统之后会再次让消费端处理这条消息**。

对 Kafka 来说，如果消费端业务事务没有成功，就不要 commit offset。进程重启或重新分配分区后，会从未提交的 offset 继续消费。看起来像“又消费了一次”，本质是“上一次没有完成确认”。

对 Redis Stream 来说，如果消费者拿到消息后没有 `XACK`，消息会留在 PEL（Pending Entries List）里。后续可以通过 `XAUTOCLAIM` / `XCLAIM` 把超时未 ack 的消息重新分配给消费者处理。

## Q：所以 Kafka 有没有“重复消费”？

A：严格说，Kafka 的语义是基于 offset 的。只要 offset 没提交，这条消息就仍然是“未确认完成”的消息；重启后再次读到它是正常行为。

但从业务处理角度看，消费端确实可能多次执行同一条消息的处理逻辑。因此工程上仍然要考虑“幂等”。

## Q：什么是幂等？

A：幂等就是同一条事件处理一次和处理多次，最终结果一致。

在 `analytics-core` 里，幂等可以靠这些方式保证：

- 每条事件有稳定 `event_id`。
- ClickHouse 明细表中保留 `event_id`、`tenant_id`、`project_id`、`source_id`。
- 写入侧必须识别重复到达，不应该把同一条事件在数据库里存两份。
- `ingestion_status` 记录事件处理状态，避免同一事件被重复计入关键聚合。

## Q：入库需要去重吗？

A：需要。你的判断是对的：不能只靠查询时 `distinct event_id` 掩盖重复数据。查询时去重可以作为兜底，但主路径应该在入库前或入库事务边界做幂等控制。

推荐做法：

1. collect 阶段要求或生成稳定 `event_id`，同一次客户端重试必须带同一个 `event_id`。
2. ingestion worker 写入前先检查 `ingestion_status` 或轻量幂等表，唯一键使用 `(tenant_id, project_id, source_id, event_id)`。
3. 如果状态已经是 `accepted`，说明事件已经入库，直接 ack 队列消息，不再写 ClickHouse。
4. 如果状态是 `processing` 且未超时，避免多个 worker 同时写同一条事件。
5. 如果状态是 `failed` 且错误可恢复，按重试策略重新处理。
6. ClickHouse 事件表仍保留 `event_id`，并可用 ReplacingMergeTree、insert dedup token 或离线校验做兜底。

因此 `analytics-core` 的目标是：**队列可以重复投递，消费可以重复执行，但事件明细最终不重复入库。**

## Q：Redis Stream 是否也能按 Kafka ack 思路设计？

A：可以。P1 推荐按 Kafka 的消费确认思路设计 Redis Stream：

1. consumer 读取消息。
2. 执行业务事务：校验、补字段、写 ClickHouse、写 `ingestion_status`。
3. 业务事务成功后才 `XACK`。
4. 如果进程崩溃或处理失败，不 ack。
5. 后续通过 pending 检查和 claim 机制重新处理。

核心原则：**先保证业务写入成功，再确认队列消费成功。**

## Q：死信队列是什么？

A：死信队列（Dead Letter Queue，DLQ）是放“多次处理仍然失败”的消息的地方。

例如一条事件 JSON 格式坏了、字段永远不合法、或者某个下游写入长期失败。如果它一直留在主队列里反复处理，会阻塞正常事件。此时可以把它转移到死信队列，并记录失败原因。

## Q：P1 要不要做死信队列？

A：建议 P1 做一个轻量版，不做复杂平台。

推荐：

- Redis Stream 主队列：`events:stream`。
- Redis Stream 死信队列：`events:deadletter`。
- 每条消息记录 `attempt_count`、`last_error`、`last_attempt_at`。
- 超过最大尝试次数后写入死信队列，并同步写 `ingestion_status=failed`。

## Q：推荐的重试策略是什么？

A：P1 推荐：

- 可恢复错误：例如 ClickHouse 短暂不可用，保留 pending，稍后 claim 重试。
- 不可恢复错误：例如事件缺少必填字段，直接写 `ingestion_status=failed`，不进入无限重试。
- 最大重试次数：例如 3 到 5 次。
- 重试间隔：先简单固定间隔，后续再做指数退避。
- 死信保存：超过次数后进入 dead letter stream。

## Q：这和 xwl_bi 的思路冲突吗？

A：不冲突。xwl_bi 里“消费成功才提交 offset”的思路是对的。`analytics-core` 应该继承这个原则，只是把 Kafka 的 offset commit 抽象成统一的 checkpoint / ack 语义，让 Redis Stream 和 KafkaBus 都能使用同一套消费成功判断。
