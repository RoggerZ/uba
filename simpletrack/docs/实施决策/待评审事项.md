## R1：Supastarter 1 天 SimpleTrack spike

状态：模板已确定，接入细节待评审

已确定：

- 生产 SaaS 模板先选择 Supastarter for Next.js。
- MakerKit 只保留为 B2B 企业控制面对照和备选。
- ShipAny、MkSaaS、ShipFast、Mkdirs 作为本地参考模板，不再作为第一梯队生产底座。

当前判断：

- `template-src/ai-supastarter-template` 已有本地源码，可核验 marketing、saas、docs、mail-preview、多支付、organizations、admin 等模块。
- 不直接修改 `template-src/ai-supastarter-template`；SimpleTrack 产品实现进入 `src/simpletrack-saas` 工作副本。
- `src/simpletrack-saas` 已从本地 Supastarter 模板初始化，当前等待远端仓库创建或权限修复后推送。
- Supastarter 同时承接产品官网、docs、SaaS 控制面、邮件预览和多支付 provider，更适合 P1 先落地。
- MakerKit 如果未来在组织/RBAC、Super Admin、测试、文档、企业控制台 UI 上明显领先，可以作为备选重开评审，但不阻塞当前路线。

评审问题：

- Supastarter 中新增 Website 资源是否顺畅。
- Realtime / Events 页面能否自然放进 `apps/saas` 的 dashboard shell。
- subscription gate、组织隔离、Admin、邮件模板是否容易维护。
- `apps/marketing` 和 `apps/docs` 是否能承接产品介绍、定价/订阅入口和 docs/quickstart。
- 许可证是否允许 SimpleTrack 商业 SaaS、私有仓库、闭源修改和团队协作。

下一步：

- 在 `src/simpletrack-saas` 工作副本中做 1 天 SimpleTrack spike。
- 创建或授权 `simpletrack/simpletrack-saas` 远端仓库后推送当前工作副本。
- MakerKit 保留资料级跟踪，不作为当前阻塞项。

## R2：支付与 Merchant of Record 路线

状态：后置评审，上线前逐项处理

已确定：

- 先按 Supastarter 已支持的支付 provider 接入。
- 当前本地 Supastarter provider 包含 Stripe、Lemon Squeezy、Polar、Creem、Dodo Payments。
- Paddle 暂不作为 P1 必接，除非后续模板或商业计划明确需要。

当前倾向：

- P1 阶段优先顺着 Supastarter 的支付抽象跑通订阅入口和 subscription gate。
- KYC/KYB、退款、拒付、发票、税务处理、费用结构等不作为 P0/P1 早期阻塞，放到真实上线收费前逐项确认。
- Stripe 技术能力强，Lemon Squeezy / Polar / Dodo Payments / Creem 等更接近 MoR 或出海友好路线，最终以实际开通能力和主体条件为准。

上线前后置检查：

- 当前主体是否能通过 KYC/KYB。
- 是否支持订阅、退款、拒付、发票、税务处理。
- 是否能与候选 SaaS 模板顺畅集成。
- 费用结构是否适合早期产品。
- 支付 provider 是否支持目标销售地区和个人开发者主体。

说明：

- KYC / KYB 解释见 `simpletrack/docs/Q&A/KYC-KYB是什么.md`。
- 个人开发者发票和税务处理解释见 `simpletrack/docs/Q&A/个人开发者如何处理发票和税务.md`。

## R3：`analytics-core` 实施设计细节

状态：已确定要做，设计细节待评审，高优先级

当前判断：

- 已确定 P1 新建独立核心仓库 `analytics-core`。
- 本地独立仓库已挂载在 `src/analytics-core`，并已提交最小 Go module、事件契约、EventBus、DirectBus、RedisStreamBus 和 KafkaBus 包边界。
- 远端 `simpletrack/analytics-core` 推送当前被 GitHub SSH 身份权限阻塞。
- `github.com/1340691923/xwl_bi` 是原作者上游仓库，不能通过整仓改名混淆上游来源、旧业务后台、SimpleTrack 新业务和核心能力边界。
- `analytics-core` 只关注分析数据面能力：事件分析、漏斗分析、留存分析、路径分析、LTV 分析、归因分析、分群、会话、元数据、实时数据、事件属性、用户属性。
- `analytics-core` 支持 Kafka、MySQL、Redis、ClickHouse 技术栈。
- 前期使用 Redis Stream 替代 Kafka，但不删除 Kafka 代码，保留 KafkaBus 作为后续高吞吐实现。
- 代码层面需要剔除 `xwl_` 变量、函数、包和业务命名前缀，按更专业、业务无关的方向修改命名和结构。
- 旧的 xwl_bi Vue2 后台界面不考虑复用。
- 初版实施方案见 `simpletrack/docs/实施决策/analytics-core实施方案.md`。
- 参考产品边界：Umami 参考分析对象体系，Litlyx 参考接入教育、Raw Events 和 Show test data。

评审问题：

- `analytics-core` 的模块目录如何划分。
- 哪些 xwl_bi 代码保留、重命名、重构、删除或重写。
- ClickHouse、MySQL、Redis、Kafka 的依赖边界如何抽象。
- Redis Stream 和 KafkaBus 的接口如何保持一致。
- 核心仓库如何保持业务无关，不引入定价、团队、订阅、账单等 SimpleTrack 控制面概念。
- SimpleTrack 产品层如何把核心仓库能力映射到 `Workspace + Site`。
- 是否能保证站点级数据隔离和外部 SaaS 安全边界。

下一步：

- 修复 `simpletrack/analytics-core` 远端写权限后推送本地提交。
- 用 Redis Stack 容器验证 RedisStreamBus 的 publish / consume / ack。
- 做 xwl_bi repo 级模块映射和最小抽取 spike。

## R4：企业分析控制台 UI 是否可落地

状态：待评审

当前判断：

- 可落地，但不能只看模板首页截图。
- 必须确认模板的 dashboard shell、layout、table、filter、form、nav、empty state、charts 是否足够可定制。
- 已确定先选择 Supastarter for Next.js，因此 UI 评审优先围绕 Supastarter 的 SaaS app、admin、organizations、payments、settings 和 shared components 展开。
- MakerKit 仍作为 B2B 企业控制面对照，重点看未来是否有明显领先证据。
- MkSaaS 的 data-table 组件可作为 Events table 参考；ShipAny 更偏 AI SaaS 快速模板；ShipFast/Mkdirs 不作为第一底座。

初步评审：

| 模板 | 企业分析控制台 UI 支撑判断 | 说明 |
| --- | --- | --- |
| Supastarter for Next.js | 已确定先选 | 本地有 `apps/saas`、`apps/marketing`、`apps/docs`、`apps/mail-preview`，并有 admin、organizations、payments、settings、shared 组件和多支付 provider。 |
| MakerKit Next.js | B2B 对照/备选 | 可能强在更细组织/RBAC、Super Admin、按席位/用量计费、测试和文档；但当前没有本地源码，不阻塞 Supastarter 路线。 |
| MkSaaS | 组件参考 | 本地有 Next 16、Better Auth、Stripe、Fumadocs 和较完整 data-table 组件，可参考 Events table，但组织/多支付/企业治理证据弱于 Supastarter。 |
| ShipAny | 快速 AI SaaS 参考 | 本地 template-two 有 Better Auth、RBAC 脚本、Stripe/Creem/PayPal 和 Fumadocs，更偏 AI SaaS 快速启动。 |
| ShipFast | 非第一选择 | Next 14、NextAuth、Mongo、Stripe、Mailgun/Crisp 方向，企业分析控制台证据弱。 |

评审问题：

- 模板是否使用 shadcn/ui、Tailwind、Radix、Ant Design 或其他成熟组件系统。
- 页面密度是否可调。
- 是否有稳定的 sidebar、breadcrumbs、tabs、table、filter bar、settings form。
- 颜色、圆角、阴影、营销式卡片是否容易收敛。

下一步：

- 对候选模板做 UI 可改造性截图评审。

## R5：免费方案与付费方案差异

状态：待评审

当前判断：

- 免费方案通常不一定功能更少，但经常在功能深度、产品化完整度、文档、升级支持、企业控制面和维护确定性上弱于成熟付费模板。

评审问题：

- 免费方案是否只是代码参考，还是能直接承接生产控制面。
- 免费方案缺失的能力需要多少开发成本补齐。
- 付费方案的许可证、更新、支持是否值得成本。
