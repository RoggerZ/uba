# SimpleTrack 分阶段实施计划

> 状态：已确定基础方向，细节持续更新  
> 最近更新：2026-04-29  
> 目标：先圈定功能，再按阶段逐步细化后端和生产实现。

## 总纲

SimpleTrack 分成两条主线：

1. **商业控制面**：登录、组织、订阅支付、邮件、Admin、AI 基础设施，优先由成熟 SaaS 模板或成熟服务承接。
2. **分析数据面**：tracker、collect API、事件校验、实时写入、事件存储、查询聚合、目标、漏斗、路径、归因，属于 SimpleTrack 的核心自研价值，P1 起通过独立仓库 `analytics-core` 承接。

阶段推进原则：

- P0 确认底座和原型，避免选型反复。
- P1 先把数据管道和公开产品入口跑通，不扩成完整分析平台。
- P2 再补分析增强能力。
- P3 再补企业治理和高级商业化能力。

## P0：产品与底座确认

状态：已确定要做，正在推进

目标：

- 建立可评审、可交互、可迁移到生产方向的 Next.js 原型。
- 确认 Supastarter 接入 spike 和必要许可证核验清单。
- 确认 SimpleTrack 的商业控制面和分析数据面边界。

已确定范围：

- 原型继续走 Next.js。
- 不在原型内继续自建登录、支付、邮件、组织、后台、AI 基础设施。
- 付费 SaaS 模板已确定先选择 Supastarter for Next.js；MakerKit 保留为 B2B 企业控制面对照和备选。
- ShipAny、MkSaaS、ShipFast、Mkdirs 作为 `template-src` 本地参考，不再作为第一梯队生产底座。
- 视觉方向采用低装饰、高密度、信息层次稳定的企业分析控制台。

交付物：

- `simpletrack/prototype/simpletrack-enterprise-mvp/`：可交互原型。
- `simpletrack/docs/SimpleTrack技术栈与SaaS模板选型评估.md`：技术栈与模板选型评估。
- `simpletrack/docs/实施决策/`：已确定和待评审决策库。
- `simpletrack/docs/Q&A/`：概念解释和评审问答。
- `simpletrack/docs/实施决策/付费SaaS模板本地对比.md`：本地模板源码对比。
- `simpletrack/docs/实施决策/analytics-core实施方案.md`：`analytics-core` P1 抽取和实施方案。

验收标准：

- 评审时能点击主要页面和流程。
- 页面结构能迁移到成熟 SaaS 模板的 dashboard shell。
- 已明确哪些能力暂时 mock，哪些能力未来交给模板或服务。
- 已列出购买或采用模板前的一天 spike 验收项。

## P1：数据管道活了 + 公开产品入口

状态：已确定为第一阶段核心目标

一句话目标：

- 客户站点上的页面浏览和自定义事件，能够被 SDK 采集，经 collect API 写入数据面，并在 Realtime 与 Events 中被看到；同时用户能通过产品官网了解产品、查看定价/订阅入口，并按 docs/quickstart 完成接入。

P1 必做：

- `analytics-core` 独立核心仓库：从 xwl_bi 抽取分析数据面核心，仓库名不带 `simpletrack` 或 `xwl`。
- Tracker SDK：支持页面浏览、基础自定义事件、站点标识。
- Collect API：接收事件、校验字段、拒绝明显非法请求。
- Event validation：校验 site、event name、timestamp、session/user 标识。
- Event storage：写入事件存储，优先围绕 ClickHouse 或可替换事件存储设计。
- Realtime view：能看到最近事件和基础实时指标。
- Events table：能查询事件列表、时间、页面、来源、用户标识、属性摘要。
- Website / Project 设置：能创建站点、生成安装 snippet、查看接入状态。
- Goal 最小闭环：定义一个关键事件，并在 dashboard 中显示是否有数据。
- 产品官网 / Marketing Site / 公开站点：类似 `https://umami.is/` 的公开入口，包含产品介绍、定价/订阅入口、docs/quickstart。

P1 明确不做：

- Team / RBAC。
- Revenue / Attribution 产品化页面。
- Session Replay / Performance。
- Boards / Share / API Key。
- Funnels / Journeys 产品化页面。
- 复杂自定义报表编辑器。

P1 后端细化方向：

- 数据模型先围绕 `workspace`、`site`、`event`、`goal`、`usage` 设计；`analytics-core` 内部保持业务无关，不写入定价、团队、订阅、账单概念。
- `analytics-core` 的能力边界包含：事件分析、漏斗分析、留存分析、路径分析、LTV 分析、归因分析、分群、会话、元数据、实时数据、事件属性、用户属性。
- `analytics-core` 支持 Kafka、MySQL、Redis、ClickHouse 技术栈。
- 事件队列必须抽象为 interface，避免第一版被 Kafka 绑定。
- 前期可以使用 Redis Stream 替代 Kafka，但不删除 Kafka 代码，后续保留 KafkaBus 作为高吞吐实现。
- 从 xwl_bi 抽取时不做整仓改名；应剔除 `xwl_` 变量、函数、包和业务命名前缀，按更专业、业务无关的方向重命名和重构代码结构。
- SimpleTrack 引用该核心仓库，并在自己的产品层处理定价、团队、订阅、账单、onboarding 和企业控制台。
- 旧的 xwl_bi Vue2 后台界面不考虑复用。
- 产品参考采用“双参考”：Umami 参考分析对象体系、事件语义、Realtime/Events/Funnels/Journeys/Retention/Segments 边界；Litlyx 参考短接入链路、Raw Events 验收、Product 空态/示例态/真实态和 `Show test data` 教育方式。
- 产品官网和 docs 先使用 Supastarter 的 `apps/marketing`、`apps/docs`；如果效果满足，直接使用；如果不满足，再轻量定制。

验收标准：

- 新站点复制 snippet 后能上报 pageview。
- 手动触发自定义事件后能进入 Events。
- Realtime 能显示最近事件，不需要刷新很久才看到。
- 关闭或删除站点后，上报被拒绝或标记为不可用。
- P1 页面不出现 P2/P3 功能入口造成范围膨胀。
- `analytics-core` 已建立独立仓库边界、模块目录、命名规范和 EventBus 抽象。
- `analytics-core` 实施方案中的 xwl_bi 代码映射、风险清单、命名映射和验收标准已被评审。
- 产品官网能展示产品价值、定价/订阅入口和接入文档入口。

## P2：分析能力增强

状态：已确定为第二阶段方向，具体范围待评审

目标：

- 从“能收数”推进到“能回答关键产品问题”。

候选范围：

- Funnels。
- Journeys / Paths。
- Segment filtering。
- Custom event schema。
- Email report。
- Basic alerts。
- Saved views。
- 留存分析，是否纳入 P2 需结合 xwl_bi 复用成本评审。

P2 原则：

- 不做全量 BI 平台。
- 每个分析页面只回答一个核心问题。
- 高级能力可以来自 xwl_bi，但必须经过产品化裁剪。

## P3：企业能力

状态：已确定为后续阶段，不进入 P1

目标：

- 支撑更高客单价、更复杂团队和更稳定的商业化运营。

候选范围：

- Team / RBAC。
- Audit log。
- API key。
- Shared dashboards。
- Revenue / Attribution。
- Admin support tools。
- SSO 或企业认证扩展。
- 更细的套餐、用量、超额、账单管理。

P3 原则：

- 只有当 P1/P2 证明产品价值后再展开。
- 优先使用成熟 SaaS 模板和成熟服务的控制面能力。
- 自研重点仍然放在分析数据面，而不是重写企业账号体系。
