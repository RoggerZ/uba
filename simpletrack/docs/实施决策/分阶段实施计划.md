# SimpleTrack 分阶段实施计划

> 状态：已确定基础方向，细节持续更新  
> 最近更新：2026-05-01
> 目标：先圈定功能，再按阶段逐步细化后端和生产实现。

## 总纲

SimpleTrack 分成两条主线：

1. **商业控制面**：登录、组织、订阅支付、邮件、Admin、AI 基础设施，优先由成熟 SaaS 模板或成熟服务承接。
2. **分析数据面**：tracker、collect API、事件校验、实时写入、事件存储、查询聚合、目标、漏斗、路径、归因，属于 SimpleTrack 的核心自研价值，P1 起通过独立仓库 `analytics-core` 承接。

阶段推进原则：

- P0 确认底座和原型，避免选型反复。
- P1 先把数据管道和公开产品入口跑通，不扩成完整分析平台。
- P2 再补分析增强能力。
- P3 再补企业治理和高级商业化能力。

## P1：数据管道活了 + 公开产品入口

状态：已确定为第一阶段核心目标

一句话目标：

- 客户站点上的页面浏览和自定义事件，能够被 SDK 采集，经 collect API 写入数据面，并在 Realtime 与 Events 中被看到；同时用户能通过产品官网了解产品、查看定价/订阅入口，并按 docs/quickstart 完成接入。

P1 必做：

- `analytics-core` 独立核心仓库：从 xwl_bi 抽取分析数据面核心，仓库名不带 `simpletrack` 或 `xwl`。
- Tracker SDK：支持页面浏览、基础自定义事件、站点标识。
- Collect API：接收事件、校验字段、拒绝明显非法请求。
- Event validation：校验 site、event name、timestamp、session/user 标识。
- Event storage：写入事件存储，优先围绕 ClickHouse 或可替换事件存储设计。
- Realtime view：能看到最近事件和基础实时指标。
- Events table：能查询事件列表、时间、页面、来源、用户标识、属性摘要。
- Website / Project 设置：能创建站点、生成安装 snippet、查看接入状态。
- Goal 最小闭环：定义一个关键事件，并在 dashboard 中显示是否有数据。
- 产品官网 / Marketing Site / 公开站点：类似 `https://umami.is/` 的公开入口，包含产品介绍、定价/订阅入口、docs/quickstart。

P1 明确不做：

- Team / RBAC。
- Revenue / Attribution 产品化页面。
- Session Replay / Performance。
- Boards / Share / API Key。
- Funnels / Journeys 产品化页面。
- 复杂自定义报表编辑器。

P1 后端细化方向：

- 数据模型先围绕 `workspace`、`site`、`event`、`goal`、`usage` 设计；`analytics-core` 内部保持业务无关，不写入定价、团队、订阅、账单概念。
- `analytics-core` 的能力边界包含：事件分析、漏斗分析、留存分析、路径分析、LTV 分析、归因分析、分群、会话、元数据、实时数据、事件属性、用户属性。
- `analytics-core` 支持 Kafka、MySQL、Redis、ClickHouse 技术栈。
- collect 层已确定把输入请求标准化为 `EventEnvelope`，并校验 `id`、`tenant_id`、`project_id`、`source_id`、`source_type`、`event_name`、`distinct_id` 和时间戳；`collect.Handler` 已负责标准化后发布到 EventBus。
- collect HTTP API 已确定使用 fasthttp 作为事件上报热路径 HTTP 库；fasthttp 只做 JSON 解码、状态码和响应适配，不把框架上下文传入 `collect.Handler` 或分析核心逻辑。
- 事件队列必须抽象为 interface，避免第一版被 Kafka 绑定。
- 前期可以使用 Redis Stream 替代 Kafka，但不删除 Kafka 代码，后续保留 KafkaBus 作为高吞吐实现。
- Redis Stream 消费语义已确定为 pending 优先：只有 ingestion 写入成功后才 ack；失败消息保留 pending 并重试；达到 `MaxAttempts` 后进入死信队列并 ack 原消息。
- ingestion 入库必须按事件唯一键保持幂等，重复消费同一事件时视为成功处理，不能写出两份事件明细。
- ClickHouse 表策略直接采用方案 B：按 `tenant_id / project_id / source_id` 路由到物理事件表，但上层仍通过统一 `events` 逻辑模型和 query builder 查询。
- ClickHouse 事件明细高吞吐写入优先使用 `clickhouse-go/v2 PrepareBatch` 原生批量写入；GORM `CreateInBatches` 作为低频写入或压测对照，不作为默认事件入库热路径。
- 入库必须做幂等去重，同一 `(tenant_id, project_id, source_id, event_id)` 重复消费时不能在数据库存两份事件；当前已落地 `EventWriteGuard` 接口边界和 GORM/MySQL `IngestionStatusGuard` 真实状态守卫，后续 worker 必须复用这条链路。
- Realtime 与 Events 查询已建立 `storage.EventQueryBuilder` 契约，ClickHouse adapter 通过 GORM ClickHouse dry-run query builder 输出 query plan；后续查询执行器必须复用这套字段白名单、表路由和过滤边界。
- Realtime 与 Events 查询已建立 `storage.EventReader` 契约，ClickHouse adapter 通过 GORM Raw 执行 query plan 并返回 `EventRecord`，产品层不得直接扫描 ClickHouse 动态表。
- `ingestion.Processor` 已明确为 P1 worker 边界：EventBus adapter 负责 ack/nack，Processor 负责调用 `storage.EventWriter`，真实 worker 入口不得复制一套消费写入逻辑。
- 本地运行依赖已固定在 `src/analytics-core/docker-compose.yml`：Redis 使用 `redis/redis-stack:latest`，MySQL 使用 `mysql:8.4`，ClickHouse 使用 `clickhouse/clickhouse-server:25.3`；默认映射到高位端口以避开本机已有 Redis/MySQL。
- 从 xwl_bi 抽取时不做整仓改名；应剔除 `xwl_` 变量、函数、包和业务命名前缀，按更专业、业务无关的方向重命名和重构代码结构。
- SimpleTrack 引用该核心仓库，并在自己的产品层处理定价、团队、订阅、账单、onboarding 和企业控制台。
- 旧的 xwl_bi Vue2 后台界面不考虑复用。
- 产品参考采用“双参考”：Umami 参考分析对象体系、事件语义、Realtime/Events/Funnels/Journeys/Retention/Segments 边界；Litlyx 参考短接入链路、Raw Events 验收、Product 空态/示例态/真实态和 `Show test data` 教育方式。
- Umami 源码参考采用“只读实现对照”：可参考 `tracker`、`api/send`、`event_data`、Realtime/Events 查询和 ClickHouse schema；但 P1 仍保持 `analytics-core` 的 Go collect、EventBus、ingestion worker、`EventWriter`、`EventReader` 和 `EventQueryBuilder` 边界。
- 产品官网和 docs 先使用 Supastarter 的 `apps/marketing`、`apps/docs`；如果效果满足，直接使用；如果不满足，再轻量定制。

### Umami 源码启发的 `analytics-core` 优化排期

这些优化来自 `simpletrack/docs/umami/docs/源码实现参考/` 的源码结构梳理，落地原则是：P1 只补数据管道闭环必须项，P1.5/P2 再做性能和高级分析增强。

| 排期 | 优化项 | 给 SimpleTrack 的启发 | 给 `analytics-core` 的启发 | 状态 |
| --- | --- | --- | --- | --- |
| P1-002A | 事件属性与用户属性模型 | Events 页面不能只显示事件名，还要能看到 custom event 属性和 identify 用户属性摘要 | 已在 collect 阶段落地属性入口契约，并在 storage 层新增 `EventPropertyRecord` / `FlattenEventProperties` typed row 逻辑展开；继续参考 Umami `event_data` / `session_data` 评审 ClickHouse typed rows、Map/JSON 或混合模型，并接入 `EventWriter` 与 `EventQueryBuilder` | 进行中 |
| P1-002B | client info enrich 与 bot/IP 过滤 | 产品上可以提供 internal traffic / bot 过滤开关，避免接入验收被无效流量污染 | IP、UA、browser、os、device、geo 和 bot/IP 过滤应是 collect/ingestion stage，不进入 ClickHouse writer | 待评审后实现 |
| P1-002C | session/visit resolver | 用户看到的是 visitors/sessions/active users，不需要理解内部匿名 hash | 用可替换 resolver 派生 `session_id`、`visit_id`，支持 salt 轮换、cookie/no-cookie、业务 id 和匿名 hash | 待评审后实现 |
| P1-002D | 查询字段白名单与过滤构建 | Events / Realtime 的过滤体验要稳定，非法字段不应该打到数据库 | `EventQueryBuilder` 已先落地类型化排序字段/方向白名单、过滤字段/operator 白名单、filter 数量上限和 `ErrInvalidEventQuery`；后续在 P1-002A 属性模型确定后补属性白名单与属性过滤 | 进行中 |
| P1-002E | Realtime/Events 最小端到端验收 | Realtime 是“活性检测”，Events 是“原始证据”，P1 必须可复验 | 已用本地 Redis/MySQL/ClickHouse 跑通 collect -> EventBus -> ingestion -> ClickHouse -> `EventReader`，覆盖 pageview、自定义事件属性和 user properties | 已完成 |
| P1-004 | Web tracker SDK 最短链路 | docs/quickstart 应提供 snippet、auto pageview、manual track、identify 和 debug 示例 | `collect.Request` / `EventEnvelope` 要稳定承接 Web SDK payload，SDK 不反向决定核心协议 | 待实现 |
| P1.5-001 | ClickHouse 读侧优化与属性治理 | 数据量变大后，Realtime、Events、Breakdown 不能靠全表扫维持体验 | 参考 Umami materialized view、小时聚合表、projection 和高频属性索引，作为高性能 ClickHouse 组件基础 | P1 闭环后评审 |
| P2-001 | Performance metrics | LCP、INP、CLS、FCP、TTFB 对产品体验有价值，但不阻塞 P1 收数 | 可作为事件类型或属性组承接，P2 再决定是否独立建模 | 暂缓 |

验收标准：

- 新站点复制 snippet 后能上报 pageview。
- 手动触发自定义事件后能进入 Events。
- Realtime 能显示最近事件，不需要刷新很久才看到。
- 自定义事件属性和 identify 用户属性能以明确模型进入存储，并至少在 Events 中可见或可确认已入库。
- Realtime 与 Events 查询必须复用 `EventQueryBuilder` / `EventReader`，并验证字段白名单、分页上限和非法 filter 处理。
- session/visit、client enrich、bot/IP 过滤如果进入 P1，必须作为可测试 stage 实现，不允许绕过 EventBus 或直接在 HTTP route 写库。
- 关闭或删除站点后，上报被拒绝或标记为不可用。
- P1 页面不出现 P2/P3 功能入口造成范围膨胀。
- `analytics-core` 已建立独立仓库边界、模块目录、命名规范和 EventBus 抽象。
- `analytics-core` Redis Stream 已覆盖 pending 优先重试、死信队列和幂等 ingestion processor，并通过自动化测试。
- `analytics-core` collect 标准化、`collect.Handler`、fasthttp `POST /collect` 入口、storage `EventWriter` 接口、ClickHouse `TableRouter`、native batch `BatchWriter`、GORM/MySQL `IngestionStatusGuard`、Events/Realtime query plan builder、ClickHouse query reader、ingestion worker 边界、本地运行依赖和 opt-in e2e 测试已落地，并通过自动化测试；Redis Stream 集成测试和完整 P1 数据管道 e2e 已在 Docker Redis/MySQL/ClickHouse 上通过。
- `analytics-core` 实施方案中的 xwl_bi 代码映射、风险清单、命名映射和验收标准已被评审。
- Umami 官方源码快照、源码审阅文档和实施映射已就位，可为 P1-002 的事件属性、Realtime/Events 查询和 ClickHouse schema 验收提供对照。
- 产品官网能展示产品价值、定价/订阅入口和接入文档入口。
- 产品官网、docs quickstart 和 mail-preview 已完成浏览器截图级验证；后续只做轻量文案和视觉微调，不再阻塞 P1 数据管道。

## P2：分析能力增强

状态：已确定为第二阶段方向，具体范围待评审

目标：

- 从“能收数”推进到“能回答关键产品问题”。

候选范围：

- Funnels。
- Journeys / Paths。
- Segment filtering。
- Custom event schema。
- Email report。
- Basic alerts。
- Saved views。
- 留存分析，是否纳入 P2 需结合 xwl_bi 复用成本评审。

P2 原则：

- 不做全量 BI 平台。
- 每个分析页面只回答一个核心问题。
- 高级能力可以来自 xwl_bi，但必须经过产品化裁剪。

## P3：企业能力

状态：已确定为后续阶段，不进入 P1

目标：

- 支撑更高客单价、更复杂团队和更稳定的商业化运营。

候选范围：

- Team / RBAC。
- Audit log。
- API key。
- Shared dashboards。
- Revenue / Attribution。
- Admin support tools。
- SSO 或企业认证扩展。
- 更细的套餐、用量、超额、账单管理。

P3 原则：

- 只有当 P1/P2 证明产品价值后再展开。
- 优先使用成熟 SaaS 模板和成熟服务的控制面能力。
- 自研重点仍然放在分析数据面，而不是重写企业账号体系。
