# SimpleTrack 技术栈底座决策

> 状态：部分已确定，部分待评审  
> 最近更新：2026-04-29

## 已确定

### D1：前端生产主线优先 Next.js

状态：已确定

原因：

- 当前 SaaS 模板和独立工具 starter 大量集中在 Next.js。
- Next.js 同时覆盖营销页、认证页、控制台、API routes、部署和生态集成。
- 原型若使用 Next.js，更容易迁入 Supastarter、MakerKit、ShipAny、MkSaaS 等候选底座。

影响：

- 新原型继续保持 Next.js App Router 方向。
- 不再沿纯静态 HTML 或孤立 Vite demo 方向扩展生产骨架。

### D2：商业控制面不自研

状态：已确定

原因：

- 登录、组织、支付、邮件、Admin、AI 基础设施是成熟 SaaS 模板最擅长解决的问题。
- SimpleTrack 的独特价值在行为分析数据面，不在重复造一个弱版 SaaS 壳。

影响：

- P0/P1 不自建完整 auth、billing、email、organization、admin。
- 原型只保留入口、mock 或页面草案。

### D3：P1 收窄为“数据管道活了 + 公开产品入口”

状态：已确定

原因：

- 这是 SimpleTrack 最早能证明产品价值的地方。
- 先证明事件能进来、能被看到，再做漏斗、路径、归因、团队。

影响：

- P1 必做 tracker、collect、validation、Realtime、Events、Website settings、Goal 最小闭环，以及产品官网 / Marketing Site / docs/quickstart。
- P1 产品层不做 Team/RBAC、Revenue/Attribution 页面、Replay/Performance、Boards/Share/API Key、Funnels/Journeys；但 `analytics-core` 底座会预留归因、漏斗、路径等分析能力边界。

### D4：P1 新建 `analytics-core` 独立核心仓库

状态：已确定

原因：

- xwl_bi 的 ClickHouse + MySQL + Redis + Kafka 技术栈适合分析产品，但 xwl_bi 的旧业务后台、命名和产品边界不适合直接成为 SimpleTrack 产品。
- SimpleTrack 需要的是业务无关、高性能、可复用的分析数据面核心，而不是把定价、团队、订阅、账单等 SaaS 控制面写入分析引擎。
- 独立仓库可以让 SimpleTrack 引用核心能力，同时保持未来复用、压测和重构空间。

影响：

- 仓库名固定为 `analytics-core`，不带 `simpletrack` 或 `xwl`。
- P1 先创建独立核心仓库，抽取事件分析、漏斗分析、留存分析、路径分析、LTV 分析、归因分析、分群、会话、元数据、实时数据、事件属性、用户属性等能力边界。
- 代码层面需要剔除 `xwl_` 变量、函数、包和业务命名前缀，按更专业、业务无关的模块命名重构。
- 技术栈支持 Kafka、MySQL、Redis、ClickHouse；前期 Redis Stream 替代 Kafka，KafkaBus 保留。
- 旧 Vue2 后台界面不复用。

### D5：P1 包含产品官网 / Marketing Site / 公开站点

状态：已确定

原因：

- SimpleTrack 不是只给内部评审看的原型，P1 需要用户能理解产品、看到定价/订阅入口，并按文档完成接入。
- 这类页面更准确叫产品官网、Marketing Site 或公开站点；如果只有单页宣传才叫 landing page。
- Supastarter 本地源码已经有 `apps/marketing` 和 `apps/docs`，适合承接这部分。

影响：

- P1 增加产品介绍、定价/订阅入口、docs/quickstart。
- 公开站点和控制台分离，避免营销页视觉污染企业分析控制台。
- 先使用 Supastarter 的 `apps/marketing` 和 `apps/docs` 看效果是否满足；效果好就直接使用，否则轻量定制。

### D6：生产 SaaS 模板先选择 Supastarter for Next.js

状态：已确定

原因：

- `template-src` 已有 Supastarter 本地源码，可核验到 `apps/saas`、`apps/marketing`、`apps/docs`、`apps/mail-preview`。
- 本地源码已看到 organizations、admin、payments、settings、mail、storage、AI 等模块，覆盖 SimpleTrack P1 需要的商业控制面底座。
- `packages/payments` 已覆盖 Stripe、Lemon Squeezy、Polar、Creem、Dodo Payments，比单 Stripe 模板更适合出海支付探索。
- MakerKit 当前没有本地源码，不能像 Supastarter 一样做源码级 spike。

影响：

- P0 不再把 Supastarter 和 MakerKit 作为同等候选反复摇摆，先按 Supastarter 进入 spike。
- MakerKit 保留为 B2B 企业控制面对照和备选，只在后续证明明显领先时再重开选型。
- ShipAny、MkSaaS、ShipFast、Mkdirs 仅作为组件、docs、营销页和支付接法参考。

### D7：支付先按 Supastarter 已支持 provider 接入

状态：已确定

原因：

- 当前目标是尽快跑通商业控制面，不把支付平台合规细节前置成 P0 阻塞。
- Supastarter 本地 provider 已包含 Stripe、Lemon Squeezy、Polar、Creem、Dodo Payments，可先顺着模板能力接入。

影响：

- P1 前不急着逐项深挖 KYC/KYB、退款、拒付、发票、税务和费用结构。
- 上线收费前再按实际选择的 provider 做逐项检查。
- Paddle 暂不作为 P1 必接，除非后续确认 Supastarter 或业务计划需要。

- 需要验证的是它承接 SimpleTrack P1 控制面和企业分析控制台的实际开发手感。
- MakerKit 只作为 B2B 对照，不阻塞 Supastarter spike。
