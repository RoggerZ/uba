# 执行与复验手册

## 1. 启动本地站点

```powershell
cd C:\Users\admin\Documents\src\uba\simpletrack\docs\umami\tracking-demo
python -m http.server 4173
```

入口：

- 调试页：`http://localhost:4173/debug.html`
- SaaS 仿真站：`http://localhost:4173/site/index.html?websiteId=YOUR_WEBSITE_ID&utm_source=producthunt&utm_medium=launch&utm_campaign=producthunt_launch&cohort=spring_launch&plan=trial&performance=1`

## 2. 跑真实浏览器流量

```powershell
npx --yes -p playwright node run-browser-flows.mjs --website-id YOUR_WEBSITE_ID --base-url http://localhost:4173/site
```

默认会跑 72 个 persona，每人 3 个独立 browser context，对应 216 个真实浏览器 session。其中 6 个 paid persona 会走到 checkout-success。

## 3. 批量灌入业务事件

先 dry run：

```powershell
node bulk-send.mjs --website-id YOUR_WEBSITE_ID --dry-run
```

真实发送：

```powershell
node bulk-send.mjs --website-id YOUR_WEBSITE_ID --hostname localhost --base-url http://localhost:4173/site
```

默认批量脚本会生成 576 个用户、1728 个 session、约 17,280 条业务事件和 48 条收入转化。与浏览器脚本合计后达到 648 用户、1944 session、约 19,440 条业务事件和 54 条收入转化。

## 4. Cloud 页面复验顺序

1. Realtime：确认当前 pageview 和 CTA 事件进入。
2. Sessions：确认 persona 会话和页面路径形成。
3. Performance：确认启用 performance 参数后的页面性能视图是否有数据。
4. Events / Properties：确认事件名和 plan、campaign、cohort、role、workspaceSize 等属性。
5. Compare / Breakdown / Filter：按 plan、campaign、cohort、source、medium 拆分。
6. Goals / Funnels / Journeys：配置 signup、first event、checkout 路径。
7. Segments / Cohorts / Retention：检查命名切片和三组 cohort。
8. UTM / Revenue / Attribution：验证 campaign 到 checkout revenue 的链路。

高级报告对象可以先用 UI 自动化脚本幂等配置：

```powershell
npx --yes -p playwright node configure-cloud-reports.mjs --website-id YOUR_WEBSITE_ID --user-data-dir "%TEMP%\\simpletrack-umami-playwright-profile"
```

该脚本只通过已登录浏览器 UI 操作 Cloud；对象已存在时返回 `exists`，不会重复创建。
9. Replays：按账号能力确认真实浏览器 session 是否可回放。

## 5. 截图采集顺序

- Phase 07：Sessions、Realtime、Performance、Compare、Breakdown、Goals、Filter。
- Phase 08：Funnels、Journeys、Retention、Replays、Segments、Cohorts、UTM、Revenue、Attribution。

每张截图新增后必须同步更新 `快照索引.md`、对应 phase 的 `flow.md`、对应 phase 的 `README.md`、`快照进度.md` 和 `Umami功能深度分析.md`。

登录态恢复后，先用脚本幂等确认 Cloud 报告对象存在：

```powershell
npx --yes -p playwright node configure-cloud-reports.mjs --website-id YOUR_WEBSITE_ID --user-data-dir "%TEMP%\\simpletrack-umami-playwright-profile"
```

这个脚本会确认或创建 `Checkout Completed Goal`、`Growth Baseline Checkout Funnel`、`Producthunt Launch Segment`、`Paid Checkout Cohort`。如果只想检查对象是否存在，可以追加 `--dry-run`；如果只想处理部分对象，可以追加 `--only goal,funnel`。

对象准备好后，再用脚本辅助采集截图：

```powershell
npx --yes -p playwright node capture-cloud-snapshots.mjs --website-id YOUR_WEBSITE_ID --headed --login-wait-ms 120000
```

脚本只接受命令行传入的 website id，不会把账号状态写入仓库。若需要复用 Playwright storage state，必须把 `--storage-state` 指向仓库外路径。
如果检测到 Cloud 登录页，脚本会停止并不写入 P07-Sxx / P08-Sxx 正式截图，避免把登录页误归档为功能页证据。对于 P08-S05A、P08-S06A、P08-S09 这类依赖配置态的截图，如果准备动作失败，截图脚本会停止并提示先运行 `configure-cloud-reports.mjs`。

补充说明：

- 真实浏览器里的 `https://cloud.umami.is/script.js` 当前会把事件发到 `https://api-gateway.umami.dev/api/send`。
- 因此 `send-event.mjs` 和 `bulk-send.mjs` 的默认直发端点也已改成 `https://api-gateway.umami.dev/api/send`。
- `https://cloud.umami.is/api/send` 的 `200 {"beep":"boop"}` 不再作为“已经入库”的充分证据。

如需进一步抓读侧证据，可运行：

```powershell
npx --yes -p playwright node probe-cloud-read-apis.mjs --website-id YOUR_WEBSITE_ID --user-data-dir "%TEMP%\\simpletrack-umami-playwright-profile"
```

该脚本会按页面逐项抓取 Cloud 读接口响应，并输出 `0 / 空数组 / No data available` 等状态摘要。
截至 2026-04-25 晚间，最新复验已经确认：使用普通 Chrome UA 后，脚本上报可以在 Cloud 读侧 materialize；如果再次出现全 0，优先检查是否误用了自定义 User-Agent。
