# 01-安装与接入

> 说明：`官方原话` 只放短英文摘录；`关联现有证据` 只写本地已验证内容。没有本地截图直证的安装细节，会明确标成“官方文档说明”。

## 这个能力解决什么问题

安装与接入解决的是“如何让 Umami 真正开始收数据”。

这里包含两件事：

1. 把应用本身装起来
2. 把网站接入点配置好，让 tracker script 能落到页面里

如果这一步没打通，后面的 Events、Sessions、Realtime 都不会有意义。

## 官方原话

> "There are several different ways to install Umami."

> "Install the application"

> "Copy the code and insert it into the `<head>` section"

> "Add the tracking code into your website HTML"

> "Then visit your website and data should immediately appear in your Umami dashboard."

Tracker 配置还提供这些接入控制：
> "data-domains" / "data-do-not-track" / "data-before-send"

## 中文解读

Umami 的接入链路很短：

1. 先决定怎么安装应用本体
2. 再登录
3. 再新增 website
4. 再把 tracking code 放进 HTML

Cloud 场景下，前两步通常已经被托管掉，用户最常做的是“创建站点 + 拿代码 + 放代码 + 验证数据”。

真正落地时，接入页不能只给一段 `<script>`。它还要回答这几个问题：这段代码应该在哪些域名生效、是否尊重浏览器 Do Not Track、是否允许发送前改写或拦截 payload、是否要开启 performance 采集。

## 通俗例子

你给一个新站点做接入时，最常见动作就是：

- 在 Umami 里新建 `localhost` 或正式域名
- 把 tracking code 放到页面 `<head>`
- 打开网站
- 回到 Umami 看 Overview 或 Realtime 是否有数据

这就像先把水管接上，再确认水压有没有上来。

## 它和相邻能力的区别

- `安装` 解决应用部署本身
- `接入` 解决某个网站怎么开始发数据
- `采集` 解决发什么数据、怎么发
- `Realtime` 解决“有没有马上进来”

不要把“安装成功”和“页面已经有分析结果”混为一谈。

## 落地动作

1. 先确认部署方式：源码、Docker compose、Docker image、或 Cloud
2. 新建 website，填好域名和名称
3. 复制 tracking code
4. 把代码插到页面 `<head>`
5. 刷新网站并观察 Umami 的 dashboard 或 Realtime
6. 生产环境建议显式配置允许域名，避免 staging、preview 或错误域名污染数据
7. 如果涉及合规要求，提前决定是否开启 Do Not Track、排除 query/hash、或用 before-send 过滤敏感字段

## 对 SimpleTrack 的启发

SimpleTrack 的接入页最好只保留三件事：

- 站点标识
- 代码片段
- 验证结果
- 域名白名单和隐私开关

不要把安装、权限、报表、模板、埋点校验都塞进一个页面。这样会让用户分不清自己到底完成了哪一步。
