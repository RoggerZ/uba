# LCP、INP、CLS、FCP、TTFB 是什么？

> 问题来源：Umami Performance 页面会展示 LCP、FCP、TTFB 等性能指标；官方 Web Vitals 体系里又经常提到 LCP、INP、CLS。这里用通俗语言解释这些指标分别在看什么、怎么判断好坏、以及在排查页面体验时应该怎么用。

## 先记一个总框架

这几个指标都在回答同一个大问题：用户打开页面后，体验到底顺不顺。

可以把一次访问想成一次进餐：

| 指标 | 通俗理解 | 用户感受 |
| --- | --- | --- |
| `TTFB` | 服务员多久开始回应你 | 网站是不是半天没反应 |
| `FCP` | 桌上多久先上了一点东西 | 白屏多久结束 |
| `LCP` | 主菜多久端上来 | 页面主体内容多久可见 |
| `INP` | 你按铃后服务员多久响应 | 点击、输入、切换是不是卡 |
| `CLS` | 桌子和盘子会不会突然移位 | 页面会不会乱跳、误点 |

再粗略按时间顺序看：

```text
请求页面 -> TTFB -> FCP -> LCP -> 持续交互中的 INP / CLS
```

其中 `LCP`、`INP`、`CLS` 是当前 Core Web Vitals 的三大核心指标；`FCP` 和 `TTFB` 通常用来辅助定位问题。

## 快速阈值表

这些阈值来自 web.dev 当前 Web Vitals 文档，实际产品评估通常看真实用户数据的 p75，也就是 75% 访问体验达到或优于这个值。

| 指标 | 好 | 需要改进 | 差 | 记忆方式 |
| --- | --- | --- | --- | --- |
| `LCP` | `<= 2.5s` | `2.5s - 4s` | `> 4s` | 主体内容 2.5 秒内出来比较稳 |
| `INP` | `<= 200ms` | `200ms - 500ms` | `> 500ms` | 点击后 0.2 秒内有反应比较顺 |
| `CLS` | `<= 0.1` | `0.1 - 0.25` | `> 0.25` | 页面越不乱跳越好 |
| `FCP` | `<= 1.8s` | `1.8s - 3s` | `> 3s` | 白屏 1.8 秒内结束较好 |
| `TTFB` | `<= 800ms` | `800ms - 1800ms` | `> 1800ms` | 服务器第一口响应别太慢 |

## LCP：页面主体多久出现

`LCP` 全称是 Largest Contentful Paint，可以理解成“最大一块主要内容什么时候画出来”。

它关心的不是页面全部资源都加载完，而是用户最在意的那块内容什么时候出现，比如：

- 首页的大图
- 首屏主标题
- 商品详情页的主图
- 文章页的正文首屏

如果 `LCP` 慢，用户常见感受是：

- 页面好像开了，但关键内容迟迟不出来
- 图片框、骨架屏、空白区域停留太久
- 首屏看起来拖沓

常见原因：

- 首屏大图太大、没有压缩或没有合适尺寸
- 关键 CSS / JS 阻塞渲染
- 接口或服务端响应慢，导致首屏内容拿不到
- 字体、图片、第三方脚本影响首屏渲染

一句话：`LCP` 看“主内容上桌有多快”。

## INP：交互响应是不是灵敏

`INP` 全称是 Interaction to Next Paint，可以理解成“用户操作后，页面多久给出下一次可见反馈”。

它关注用户在页面生命周期里的交互，例如：

- 点击按钮
- 打开下拉框
- 输入表单
- 切换 tab
- 选择筛选条件

如果 `INP` 差，用户常见感受是：

- 点了按钮没反应
- 输入框打字卡顿
- 页面像卡住了一下
- 筛选、弹窗、菜单打开很慢

常见原因：

- 主线程被大段 JS 占满
- 点击后同步执行太多计算
- 渲染列表太大，没有分页或虚拟滚动
- 第三方脚本抢占主线程
- React / Vue 等框架一次状态更新牵动太多组件重渲染

一句话：`INP` 看“用户动手后，页面回不回应得上”。

## CLS：页面会不会突然乱跳

`CLS` 全称是 Cumulative Layout Shift，可以理解成“页面累计发生了多少意外布局偏移”。

它没有秒或毫秒单位，数值越低越稳定。页面元素突然移动时，CLS 就会变差。

典型场景：

- 图片没有预留宽高，加载后把正文挤下去
- 广告或推荐位突然插入
- 字体加载后文字宽度变化，导致布局重排
- 顶部提示条、弹窗、横幅突然出现，把原内容推走

如果 `CLS` 差，用户常见感受是：

- 刚要点按钮，按钮突然跑了
- 正在读文字，段落突然下移
- 页面看起来不稳、不专业

注意：不是所有移动都算坏。用户主动点击后出现的展开、收起、切换，一般不属于“意外”的布局偏移。CLS 主要惩罚用户没有预期的乱跳。

一句话：`CLS` 看“页面有没有稳定地站住”。

## FCP：白屏多久结束

`FCP` 全称是 First Contentful Paint，可以理解成“页面第一次画出内容的时间”。

这里的“内容”可以是：

- 文字
- 图片
- SVG
- canvas
- 非白色背景之外的可见内容

如果 `FCP` 慢，用户常见感受是：

- 打开页面后一直白屏
- 不知道页面是不是坏了
- 浏览器转了半天才看到一点东西

但是 `FCP` 快不代表页面已经好用。它只说明“有东西出来了”，不代表主内容已经出现，也不代表按钮已经能顺畅点击。

一句话：`FCP` 看“白屏什么时候结束”。

## TTFB：服务器第一口响应有多快

`TTFB` 全称是 Time to First Byte，可以理解成“浏览器发出请求后，多久收到服务器返回的第一个字节”。

它不只包含服务端代码执行时间，也可能包含：

- DNS 查询
- TCP / TLS 连接
- 重定向
- CDN 回源
- 服务器排队
- 后端渲染或接口查询
- 网络传输延迟

如果 `TTFB` 慢，用户常见感受是：

- 地址打开后很久才开始有东西
- 不是页面渲染慢，而是请求起步就慢
- 同一个页面在不同地区或网络下差异很大

常见原因：

- 服务端接口慢
- SSR 页面渲染慢
- 数据库查询慢
- CDN 缓存没命中
- 重定向链路太长
- 用户离服务器太远

一句话：`TTFB` 看“后端和网络起步是不是拖后腿”。

## 五个指标怎么配合排查

不要孤立看单个指标，最好按用户抱怨来选入口。

| 用户反馈 | 优先看 | 常见判断 |
| --- | --- | --- |
| “打开页面一直白屏” | `TTFB` + `FCP` | 先判断是服务端没开始回，还是前端没开始画 |
| “页面主体出来很慢” | `LCP` | 看首屏大图、主内容、关键接口和渲染阻塞 |
| “点按钮很卡” | `INP` | 看主线程、事件处理、组件重渲染和第三方脚本 |
| “页面总是跳来跳去” | `CLS` | 看图片尺寸、广告位、字体、动态插入内容 |
| “整体感觉慢，但不知道哪里慢” | `TTFB -> FCP -> LCP -> INP / CLS` | 先定位阶段，再拆具体原因 |

## 对 Umami / SimpleTrack 的读法

在 Umami Performance 页面里看到这些指标时，不要把它们理解成“服务器监控指标”，它们更接近真实用户体验指标。

建议这样读：

1. 先看 p75，不要只看平均值。平均值容易被极快或极慢的少数访问拉偏。
2. 先看页面维度。通常不是整个站都慢，而是某些 URL 或页面类型慢。
3. 再看环境维度。移动端、某些浏览器、某些国家或网络环境可能更差。
4. 把指标和 Sessions、Realtime、Replays 结合起来看。性能指标告诉你“哪里慢”，用户路径和回放更容易告诉你“慢造成了什么影响”。

对 SimpleTrack 来说，可以把这五个指标拆成三层产品能力：

| 层级 | 指标 | 产品问题 |
| --- | --- | --- |
| 起步速度 | `TTFB` `FCP` | 页面是不是很久没有反馈 |
| 首屏体验 | `LCP` | 用户真正要看的内容是否足够快 |
| 使用过程 | `INP` `CLS` | 页面操作是否顺，布局是否稳 |

## 常见误解

| 误解 | 更准确的理解 |
| --- | --- |
| `FCP` 快就说明页面快 | 只能说明白屏结束，主内容可能还没出来 |
| `LCP` 是整页加载完成时间 | 不是，它看的是首屏最大主要内容的出现时间 |
| `TTFB` 慢一定是前端问题 | 不一定，通常要查网络、CDN、服务端、数据库或 SSR |
| `INP` 只看第一次点击 | 不是，它关注页面生命周期内的交互响应表现 |
| `CLS` 只要页面动了就扣分 | 主要看用户没有预期的布局移动，用户主动触发的变化通常不是问题重点 |
| 分数好就代表所有用户都好 | p75 好只能说明大多数访问达标，仍要看慢页面和差环境 |

## 一句话记忆版

- `TTFB`：服务器多久开始回应
- `FCP`：白屏多久结束
- `LCP`：主内容多久出现
- `INP`：点击输入多久响应
- `CLS`：页面会不会乱跳

更短一点：

```text
TTFB 看后端起步
FCP 看白屏结束
LCP 看首屏主内容
INP 看交互卡不卡
CLS 看布局稳不稳
```
