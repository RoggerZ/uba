# SimpleTrack 落地评审清单

> 用途：把 Umami 官方能力解读转成 SimpleTrack 产品评审时可以逐项判断的清单。这里不新增官方能力解释，只引用模块文档和 playbook，重点回答“现在该不该做、做到什么程度、怎么证明它是真的”。

## 使用方式

每次评审先回答三类问题：

1. `是否需要进入本阶段`：这项能力是否服务当前产品目标。
2. `是否有可验证证据`：有截图、操作流、脚本结果，还是只有官方文档说明。
3. `是否有清晰口径`：事件名、字段、时间窗口、身份、权限、模型是否已经定义。

建议评审顺序：

1. 先过 `MVP 准入清单`。
2. 再按业务目标选择 `分析链路`、`增长链路` 或 `排障链路`。
3. 最后过 `证据边界检查`，避免把未复验能力写成已完成。

## MVP 准入清单

| 能力 | 是否应进 MVP | 准入标准 | 关联文档 |
| --- | --- | --- | --- |
| Website 创建 | 是 | 能创建站点、拿到 tracking code、识别 website id | [01-安装与接入](./01-安装与接入.md) |
| Tracking code | 是 | 页面能加载 tracker，能限制生效域名，能说明隐私开关 | [01-安装与接入](./01-安装与接入.md) |
| Pageview | 是 | 打开页面后 Overview 或 Realtime 能出现访问数据 | [02-采集与事件](./02-采集与事件.md) |
| Event | 是 | 至少一个关键事件能通过 DOM、JS 或 API 上报 | [02-采集与事件](./02-采集与事件.md) |
| Realtime | 是 | 能作为接入验收页，显示最近页面、事件、来源或地区 | [05-Realtime](./05-Realtime.md) |
| Filters | 是 | 能按页面、来源、UTM、事件属性或会话属性切片 | [14-Filters](./14-Filters.md) |
| Simple Goal | 是 | 能定义一个页面型或事件型成功动作，并解释分母 | [09-Goals](./09-Goals.md) |
| Breakdown | 增强版预留 | MVP 只保证事件属性可被后续拆分；正式结构分析放增强版 | [08-Breakdown](./08-Breakdown.md) |
| Default Dashboard | 可选 | 若要展示首屏概览，只做系统默认工作台，不做自定义 Board | [20-Boards-Links-Pixels-Teams-API](./20-Boards-Links-Pixels-Teams-API.md) |
| Boards / Share URL | 增强版预留 | 有专题看板、跨团队复用或公开分享需求时再进入 | [20-Boards-Links-Pixels-Teams-API](./20-Boards-Links-Pixels-Teams-API.md) |

MVP 不建议默认包含：

- `Attribution`：需要稳定 UTM、转化事件和模型口径。
- `Revenue`：需要真实收入事件、币种、订单状态。
- `Replays`：需要采样、遮罩、保留周期和隐私策略。
- `Teams`：需要权限模型，不适合只补一个成员列表。

## 接入与采集评审

| 检查项 | 通过标准 | 常见风险 |
| --- | --- | --- |
| 域名范围 | tracking code 只在预期域名生效 | staging、preview、localhost 污染正式数据 |
| Do Not Track | 是否尊重浏览器 DNT 有明确策略 | 隐私承诺与实际采集不一致 |
| before-send | 是否能过滤或改写敏感 payload | token、cookie、表单内容进入分析系统 |
| 事件字典 | 有事件名、触发条件、必带属性、可选属性 | `button_click_01` 这类无业务含义事件泛滥 |
| 属性边界 | 属性数量、长度和字段类型有约束 | 把日志、订单明细、原始对象塞进 event data |
| API send | 服务端上报和浏览器上报落到同一事件模型 | 前后端各用一套命名，后续无法汇总 |

## 数据模型评审

| 对象 | 应该表达什么 | 不应该承载什么 |
| --- | --- | --- |
| `event` | 一次业务动作，例如 `signup_completed` | 长期用户标签、完整表单内容 |
| `event data` | 这次动作的上下文，例如 `surface`、`plan`、`variant` | token、cookie、订单明细 |
| `session data` | 这次访问的上下文，例如 `plan`、`cohort`、`role` | 跨会话永久身份 |
| `distinct id` | 稳定业务用户标识 | 邮箱、手机号、姓名等可识别个人信息 |
| `cohort label` | 事件或会话上的启动批次、生命周期标签 | 被直接等同为 Umami 的 Cohort 分析对象 |
| `cohort feature` | 某时间范围内做过某动作的一群人 | 普通临时筛选条件或静态标签 |
| `segment` | 可复用过滤条件 | 带时间承诺的人群定义 |

关键判断：

1. 事件层、会话层、身份层有没有分开。
2. 匿名访客是否被误写成实名用户。
3. `Distinct ID` 是否可控、稳定、不可直接反查个人信息。

## 分析链路评审

| 问题 | 推荐能力顺序 | 通过标准 |
| --- | --- | --- |
| 有没有成功 | `Goals` | 成功动作和分母说得清楚 |
| 卡在哪一步 | `Goals -> Funnels` | 每个 step 的页面或事件条件稳定 |
| 用户真实怎么走 | `Funnels -> Journeys` | 起点、终点和高频路径可解释 |
| 哪类流量表现不同 | `Filters -> Breakdown` | 字段来源和匹配方式清楚 |
| 优化后是否变好 | `Compare -> Breakdown` | 对比周期固定，涨跌比例可解释 |
| 用户会不会回来 | `Cohorts -> Retention` | cohort 时间窗口和观察周期足够 |

评审时不要跳过的问题：

1. Goal 的分母是谁。
2. Funnel 是否强顺序。
3. Journey 是否只是用来探索路径，不替代 Funnel。
4. Retention 的观察窗口是否已经足够长。
5. Compare 是否用同口径日期范围。

## 增长与商业化评审

| 能力 | 前置条件 | 不满足前置条件时的风险 |
| --- | --- | --- |
| UTM | 有 source、medium、campaign 命名规范 | 同一活动被拆成多个 campaign |
| Revenue | 有收入事件、币种、订单状态 | 把点击结账误算成收入 |
| Attribution | 有 conversion step 和归因模型 | 把归因贡献误当因果 ROI |
| Links | 有外链点击或重定向追踪需求 | 和 UTM 生成器职责混乱 |
| Pixels | 有轻量曝光或外部嵌入需求 | 被误用成完整 tracker 替代品 |

第一版建议：

- UTM 只强制 `source / medium / campaign`。
- Revenue 只承认支付成功或订阅确认后的收入事件。
- Attribution 只做 `First-Click / Last-Click`，并在报表里明示模型。
- 不把 ROI 结论写成因果证明，除非有实验或更严格对照。

## 体验诊断评审

| 问题 | 推荐能力 | 通过标准 |
| --- | --- | --- |
| 刚发布后数据有没有断 | Realtime | 能看到最近数据时间和事件样例 |
| 页面是不是慢 | Performance | 有 Core Web Vitals、页面和环境拆分 |
| 用户到底怎么操作 | Replays | 有采样率、遮罩、最长时长、保留周期 |
| 单个访问者做过什么 | Sessions | 能从访客列表进入活动历史和属性 |
| 异常集中在哪类用户 | Filters / Segments | 能按设备、来源、页面或属性缩小范围 |

上线前必须明确：

1. Performance 需要启用 `data-performance="true"`。
2. Replays 不是历史回填，只能解释启用后采集到的会话。
3. Replays 必须先有采样、遮罩和保留周期策略。
4. Sessions 默认更接近匿名访问上下文，不等同于实名用户。

## 协作与扩展评审

| 能力 | 评审问题 | 建议阶段 |
| --- | --- | --- |
| Boards | 是否需要专题看板或公开分享 | 增强版 |
| Share URL | 是否允许外部访问，公开范围是什么 | 增强版 |
| Teams | 是否已经有团队、角色和网站归属模型 | 后置 |
| API Key | 是否有创建、轮换、撤销和权限边界 | 后置 |
| Links | 是否有外链点击追踪需求 | 后置 |
| Pixels | 是否有外部嵌入或轻量曝光追踪需求 | 后置 |

权限边界建议：

1. 公开分享和团队协作必须是两套语义。
2. API Key 不应默认拥有所有能力。
3. Board 类型要带语义，例如 `Website`、`Mixed`、`Pixel`、`Link`。
4. Teams 不能只做成员列表，要和网站归属、角色、邀请状态一起设计。

## 证据边界检查

写评审结论前，逐项确认：

| 要写的结论 | 最低证据要求 | 建议措辞 |
| --- | --- | --- |
| 已验证 | 有本地截图、操作流或脚本执行结果 | “已在本地 Umami Cloud 实操中验证” |
| 部分验证 | 有入口、空态、配置壳或输入侧证据 | “已验证入口或输入侧，结果态仍需复验” |
| 官方能力说明 | 只有官方文档，没有本地复验 | “基于官方文档解读，当前尚未本地完整复验” |
| 待复验 | 有计划或数据准备，但没有 UI 结果 | “待 Cloud UI 复验和截图” |

当前特别要谨慎的能力：

- `Performance`：已有独立页面截图和结果态，可作为性能分析参考。
- `Replays`：已有独立页面截图，但当前账号受 `Business plan` 限制，未验证回放播放态。
- `Attribution`：已在 `Triggered event / checkout_completed` 口径下显示非零结果；评审时仍要避免把归因贡献直接写成因果 ROI。
- `Revenue`：已有 Revenue 报表页结果态；当前数值包含多轮重跑累积，引用时不能等同于单次样本。
- `Links / Pixels / Teams`：当前更多依赖官方文档和原型/入口证据。

## 最小评审结论模板

可以直接按下面格式写到 PRD 或评审纪要里：

```text
本阶段建议进入：<能力名称>
业务目标：<要回答的问题>
前置条件：<事件、身份、字段、样本量、权限等>
当前证据：<已验证 / 部分验证 / 官方说明 / 待复验>
落地范围：<MVP / 增强版 / 后置>
不做范围：<明确排除的高级能力>
下一步验证：<截图、操作流、脚本或 Cloud UI 复验>
```
