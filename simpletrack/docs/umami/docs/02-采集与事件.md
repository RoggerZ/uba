# 02-采集与事件

> 说明：`官方原话` 只放短英文摘录；`关联现有证据` 只写本地已验证内容。凡是本地 demo 已经跑通的部分，会优先写成“已验证”。

## 这个能力解决什么问题

采集与事件解决的是“用户做了什么，系统如何记录下来”。

这里不是单一入口，而是三种记录方式：

1. 自动采集 pageview
2. 用 HTML data 属性声明事件
3. 用 JavaScript 或服务端 API 主动上报

这样才能同时覆盖按钮点击、页面跳转、服务端补数和批量灌数。

## 官方原话

> "Besides pageviews, Umami is also able to track events"

> "There are two ways to record events in Umami"

> "By default everything is automatically collected"

> "POST /api/send"

> "For Umami Cloud send a POST to `https://cloud.umami.is/api/send`."

Event data 的边界也很明确：
> "Maximum of 50 properties"

> "Maximum length of 500 characters"

## 中文解读

Umami 的事件采集思路很清楚：

- 页面浏览是默认底座
- DOM 事件适合低成本埋点
- `window.umami.track()` 适合动态交互
- `POST /api/send` 适合服务端或批量数据

它把“写埋点”变成了几层不同成本的选择。

事件数据不是越多越好。官方对属性数量和长度都有限制，这背后的产品逻辑是：事件属性应该服务分析，而不是把整段业务对象、日志或表单内容塞进 analytics。

## 通俗例子

一个注册按钮可以这样理解：

- 不点按钮时，只会有 pageview
- 点了按钮但不想写 JS 时，用 `data-umami-event`
- 按钮背后还有业务状态时，用 `track('signup', { ... })`
- 后端要补记一条转化时，用 `POST /api/send`

## 它和相邻能力的区别

- `Track Events` 更偏“页面交互如何声明”
- `Tracker functions` 更偏“JS 层如何主动控制”
- `Sending stats` 更偏“服务端如何直接发数据”
- `Collect data` 更偏“把代码真正接到页面里”

不要把“事件定义”与“数据传输”混成一个问题。

## 落地动作

1. 先统一事件命名，比如 `signup_click`、`checkout_submit`
2. 再定义哪些事件走 DOM 属性，哪些事件走 JS
3. 对服务端补数单独保留 `api/send` 路径
4. 给 event data 加长度和字段数量约束，默认不允许上报原始表单内容、token、cookie 或订单明细
5. 用 Realtime 先验收，再去看 Events 和 Properties
6. 对关键事件保留一张“事件名、触发条件、必带属性、可选属性”的字典

## 对 SimpleTrack 的启发

SimpleTrack 可以把采集层拆成四个明确入口：

- 自动 pageview
- DOM 埋点
- JS 埋点
- 服务端补数

然后把这些入口统一落到同一个事件模型里，避免前端、后端、批量工具各说各话。

事件设计最好从少量核心动作开始：注册、安装、首个事件、支付。等这些动作能稳定回答业务问题，再逐步加属性和新事件。
