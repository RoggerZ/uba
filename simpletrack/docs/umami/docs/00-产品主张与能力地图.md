# 00-产品主张与能力地图

> 说明：`官方原话` 只放短英文摘录；`关联现有证据` 只写本地已验证内容。没有本地截图或 demo 直证的地方，会明确标成“官方文档补充”。

## 这个能力解决什么问题

Umami 的核心问题不是“多一个分析面板”，而是把“网站有没有数据、数据是不是可信、数据应该怎么看”这三件事拆开。

它先解决接入和采集，再解决身份和会话，再解决报表和对比。这样产品可以用很短的链路回答三个问题：

1. 数据有没有进来
2. 是谁在访问
3. 这些访问说明了什么

## 官方原话

> "open-source web analytics platform that respects user privacy"

> "No cookies, no fingerprinting, no personal data."

> "The tracking script is under 2KB."

> "Deploy with Docker in minutes."

## 中文解读

这套产品主张可以拆成四层：

1. 先把数据接进来
2. 再把事件、会话、身份串起来
3. 然后把实时、报表、对比放到不同分析层
4. 最后再决定是否扩展到更重的报表能力

它不是“什么都做的重型分析平台”，而是“默认轻、接入短、可解释”的分析系统。

## 通俗例子

一个新站上线后，运营只想知道三件事：

- 代码是不是已经生效
- 注册按钮有没有被点
- 这一周和上周相比有没有变好

Umami 的能力地图刚好对应这三件事：

- `Realtime` 看“现在有没有数据”
- `Events` 和 `Properties` 看“按钮和属性有没有被记录”
- `Compare` 看“这一段时间表现有没有变化”

## 它和相邻能力的区别

- `安装与接入` 解决“怎么把站点接起来”
- `采集与事件` 解决“怎么把行为记下来”
- `指标对象与Distinct-ID` 解决“怎么把同一个人串起来”
- `Sessions` 解决“怎么查看一个访问者或会话”
- `Realtime` 解决“怎么验证数据正在进入”
- `Performance` 解决“页面是不是慢、慢在哪里”
- `Compare` 解决“这一段时间和过去相比有没有变化”

## 落地动作

1. 先把产品语言定成“接入短、验证快、报告轻”
2. 再把能力分成接入、采集、识别、实时、报表五层
3. 每一层都保留一个最小可验证入口
4. 让“是否有数据”优先于“数据是否很全”

## 对 SimpleTrack 的启发

SimpleTrack 可以直接学这个分层方式：

- 用一个轻量的接入页解决初始化
- 用一个事件采集层解决 pageview、click、track、identify
- 用一个身份层承接 `Distinct ID` 和 `Session` 视角
- 用一个实时验收页确认管道通了
- 用一个报表层再去做趋势和对比

这比把所有分析能力都塞进同一个首页更清晰。
