# 从采集到事件分析

## 目标

把“数据进来了”推进到“我知道用户点了什么、哪些事件值得继续分析”。

## 适用场景

- 已经能收到 pageview，但还看不清关键交互
- 想从按钮点击、提交动作、业务事件开始搭建分析体系

## 涉及能力

- [02-采集与事件](../02-采集与事件.md)
- [08-Breakdown](../08-Breakdown.md)
- [14-Filters](../14-Filters.md)
- [17-UTM](../17-UTM.md)

## 推荐顺序

1. 先定义关键事件命名规则
2. 对最重要的按钮和提交动作接 `data-umami-event` 或 `track()`
3. 给事件加最必要的属性，例如 `plan`、`surface`、`variant`
4. 如果要分析渠道效果，先确认 `utm_source / utm_medium / utm_campaign` 或 `campaign` 已进入可过滤字段
5. 用 Filters 先切最核心的场景
6. 用 Breakdown 看不同维度的结构差异

## 每步判断点

1. 事件名应该表达业务动作，而不是技术实现细节
2. 属性数量要克制，优先保留会参与分析和决策的字段
3. 如果一个问题只靠 event name 就能回答，先不要再补太多属性
4. UTM 和 `campaign` 字段要先统一命名，否则后续 Breakdown 会把同一活动拆散
5. 先确认事件列表稳定，再往 Goals、Funnels 延伸

## 常见坑

- 把 `button_click_01` 这种没有业务含义的名字放进正式分析
- 同一个动作在不同页面用不同事件名，导致后续不能汇总
- 没有先想清楚“想回答什么问题”，就先埋几十个属性

## SimpleTrack 可直接借鉴的决策

- 提供“事件命名模板”和属性建议，而不是只给一个空文本框
- 让事件调试页能同时看到请求是否成功、事件名、属性载荷
- 把 Breakdown 设计成事件分析的自然下一步，而不是埋得很深的功能
