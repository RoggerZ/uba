# 从渠道到营收到归因

## 目标

把“流量从哪里来”一路串到“带来了多少收入、该把功劳分给谁”。

## 适用场景

- 做投放、邮件、社媒、内容增长
- 想比较不同 campaign 的实际效果
- 想把收入和渠道效果放到同一张分析链路里

## 涉及能力

- [17-UTM](../17-UTM.md)
- [18-Revenue](../18-Revenue.md)
- [19-Attribution](../19-Attribution.md)
- [09-Goals](../09-Goals.md)

## 推荐顺序

1. 先保证 UTM 参数规范
2. 再定义能代表业务结果的关键事件或 Goal
3. 如果业务能上报金额，再让收入事件携带 `revenue` 和 `currency`
4. 最后选择 conversion step 和 attribution model 给渠道分功

## 每步判断点

1. UTM 回答“流量来自哪里”
2. Goal 回答“这些流量有没有带来结果”
3. Revenue 回答“结果值多少钱”
4. Attribution 回答“功劳应该怎么分”
5. 归因模型回答的是“按哪条规则分功”，不是自动证明因果

## 常见坑

- UTM 命名不统一，最后同一活动拆成好几份
- 只有流量，没有关键事件，最后归因只能停留在访问量
- 还没想清楚归因模型，就急着解释 ROI
- 收入事件只在前端写一个展示金额，但没有和真实订单或订阅状态对齐

## SimpleTrack 可直接借鉴的决策

- 先把 UTM 和 Goal 打通，再考虑 Revenue 和 Attribution
- 在产品里给出 UTM 命名规范提示，减少运营侧写错参数
- 归因模型要明确写给用户看，不然结论容易被误解
- 第一版归因只做 `First-Click / Last-Click` 就够了，重点是把模型口径、conversion step 和收入字段解释清楚
