# 从 0 到首批数据

## 目标

把一个全新的站点从“还没接入”推进到“Overview、Realtime、Events 开始出第一批可解释的数据”。

## 适用场景

- 新产品第一次接入 Umami
- 想验证埋点链路是否真的通了
- 想为 SimpleTrack 设计最短 onboarding 路径

## 涉及能力

- [01-安装与接入](../01-安装与接入.md)
- [02-采集与事件](../02-采集与事件.md)
- [03-指标对象与Distinct-ID](../03-指标对象与Distinct-ID.md)
- [05-Realtime](../05-Realtime.md)

## 推荐顺序

1. 创建 website，先拿到 `website id`
2. 用官方 tracker snippet 接入页面
3. 打开页面，确认 pageview 已发送
4. 补一个最小事件，例如 `signup_click`
5. 先看 Realtime，再看 Overview，最后看 Events

## 每步判断点

1. 如果连 tracker 都没挂上，不要急着看高级报告
2. 如果请求已经成功但 Overview 还空，不代表接入失败，先去看 Realtime
3. 如果 pageview 有了但事件没有，优先排查 `data-umami-event` 或 `track()` 调用
4. 如果需要跨 session 识别同一个人，再引入 Distinct ID，不要一开始就把身份建模做复杂

## 常见坑

- 一上来就去看 Funnel、Retention，结果因为基础事件还没稳定，页面自然空白
- 只看 Overview，不看 Realtime，容易把“聚合延迟”误判成“接入失败”
- 在没有统一命名规范前就大量堆事件，后面很难做 Breakdown、Goals、Funnels

## SimpleTrack 可直接借鉴的决策

- 把“创建站点 -> 复制代码 -> 验证首批数据”做成一条连续 onboarding，不要拆得过散
- 把 Realtime 当成接入验收页，把 Overview 当成稳定聚合后的首页
- 在新手链路里只要求用户先完成 pageview 和一个关键事件，不强迫先学会所有高级分析
