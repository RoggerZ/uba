# 从实时异常到性能与回放排查

## 目标

当团队发现“数据突然异常”或“用户体验可能有问题”时，快速找到排查顺序。

## 适用场景

- 刚发布新版本，怀疑埋点断了
- 某页面加载变慢，转化突然下滑
- 用户反馈操作混乱，但传统报表解释不清

## 推荐顺序

1. 先看 Realtime，确认现在是否仍有流量和事件进入
2. 再看 Performance，判断是否存在页面级性能回退
3. 如果怀疑是交互问题，继续看 Replays
4. 需要追单个访问过程时，再回到 Sessions
5. 如果异常只出现在某类用户或渠道，再回到 Filters / Segments 缩小范围；这一步是下钻，不是第一入口

## 每步判断点

1. Realtime 适合做“现在是不是坏了”的第一判断
2. Performance 适合确认“是不是慢导致体验差”
3. Replays 适合看“用户到底怎么点的”
4. Sessions 适合补足单次访问的上下文
5. Replays 只能解释采集到的会话，启用前的历史行为不要指望回填

## 常见坑

- 一出问题就先看长期报表，结果错过实时异常
- 性能问题没有和页面、设备、浏览器维度一起看
- 把回放当成唯一真相，而忽略聚合指标和上下文
- 回放没有采样、遮罩和保留策略，后面会同时遇到隐私、成本和噪音问题

## SimpleTrack 可直接借鉴的决策

- 把 Realtime 设计成发布后验收和异常巡检的默认入口
- 如果将来做性能监控，要和页面维度、版本维度一起出现
- 回放必须和 Sessions、事件、性能放在同一条排查链路里，而不是孤立存在
- 排障页最好天然带着当前时间范围、页面、设备和渠道筛选条件，避免每跳一次页面就丢上下文
