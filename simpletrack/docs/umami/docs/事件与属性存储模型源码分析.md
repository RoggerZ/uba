# Umami 事件与属性存储模型源码分析

> 目标：只回答一个问题。Umami 到底把“事件本体、事件属性、identify / session 属性”分别存到哪里，为什么这样分。

## 一句话结论

Umami 不是把所有上下文塞进一个不可查询的 JSON 列。它把：

- 事件骨架存进 `website_event`
- 事件属性展开成多行 `event_data`
- identify / session 上下文展开成多行 `session_data`

这样做的直接收益是：`Events` 页面可以继续按“一行一个事件”展示，而属性筛选、Breakdown、Revenue 这类分析又不用在读侧反复解析整块 JSON。

## 三张核心表分别装什么

| 层 | 作用 | 关键字段 | 源码证据 |
| --- | --- | --- | --- |
| `website_event` | 事件事实表，一行代表一次 pageview / custom event / performance | `website_id`、`session_id`、`visit_id`、`event_type`、`event_name`、URL、UTM、device、性能指标 | 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `prisma/schema.prisma:99-149`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `db/clickhouse/schema.sql:2-55` |
| `event_data` | 事件属性表，一次事件会被展开成多条属性记录 | `event_id`、`data_key`、`string_value`、`number_value`、`date_value`、`data_type` | 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `prisma/schema.prisma:152-172`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `db/clickhouse/schema.sql:57-74` |
| `session_data` | identify / session 上下文表，更像“这趟访问或这个 distinct id 的上下文 KV” | `session_id`、`data_key`、`distinct_id`、typed value columns | 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `prisma/schema.prisma:174-194`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `db/clickhouse/schema.sql:76-91` |

这里最关键的不是“三张表”本身，而是边界：

- `website_event` 保存“这次事件是什么”
- `event_data` 保存“这次事件带了哪些动态属性”
- `session_data` 保存“这个 session / distinct id 当前有什么上下文”

## 写入入口怎么把三层分开

`/api/send` 在 collect 入口就把自定义事件和 identify 分成了两条写入分支：

- 自定义事件 / pageview / performance 走 `saveEvent(...)`
- identify 走 `saveSessionData(...)`

源码里不是一套大 payload 同时写三张表，而是根据 `COLLECTION_TYPE` 进入不同写入函数。`type === customEvent/pageView/performance` 时，`route.ts` 会把 `data` 当成 `eventData` 传给 `saveEvent`；`type === identify` 时，才把 `data` 当成 `sessionData` 传给 `saveSessionData`。源码证据：仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/app/api/send/route.ts:217-280`

这意味着 Umami 的“用户属性”并不是“跟着每个事件一起重复写入的第二个属性袋”，而是更接近：

- 事件上的动态属性：`eventData`
- identify / session 上下文：`sessionData`

## 动态属性为什么能拆成独立行

核心动作有两步：

1. `flattenJSON` 递归展开嵌套对象，把 `{ plan: { tier: "pro" } }` 变成 `plan.tier`
2. `saveEventData` / `saveSessionData` 再把每个 key 写成一行 typed KV

`flattenJSON` 的行为非常直接：

- 普通对象继续递归
- number / string / boolean / date 进入对应类型
- array 被转成 JSON 字符串，同时记录 `data_type`

源码证据：仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/lib/data.ts:4-24`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/lib/data.ts:53-79`

`saveEventData` 会把这些扁平化结果批量落入 `event_data`，而 ClickHouse 路径则把它们组装成 `event_data` message 继续发 Kafka 或直接插表。源码证据：仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/queries/sql/events/saveEventData.ts:20-47`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/queries/sql/events/saveEventData.ts:50-79`

`saveSessionData` 也做同样的扁平化，但关系型路径会先尝试按 `sessionId + dataKey` 更新，更新不到再创建，所以它更像“最近状态覆盖”，不是“无限追加的历史属性流”。源码证据：仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/queries/sql/sessions/saveSessionData.ts:25-70`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/queries/sql/sessions/saveSessionData.ts:73-104`

## 同一个例子在 Umami 里会变成什么

假设业务想表达的是：

```json
{
  "event_name": "signup_clicked",
  "event_data": {
    "button": "hero",
    "plan": "pro"
  },
  "identify_data": {
    "role": "founder"
  }
}
```

在 Umami 里，它不会变成一个“大 JSON 事件”。它会拆成三层：

| 层 | 结果 |
| --- | --- |
| `website_event` | 一条事件行，`event_name=signup_clicked`，并带 `session_id`、`visit_id`、URL、UTM 等基础上下文 |
| `event_data` | 两条属性行：`button=hero`、`plan=pro` |
| `session_data` | 一条或多条 session/identify 行：`role=founder` |

如果这是 ClickHouse 路径，`saveEvent` 先写 `website_event` message，然后在 `eventData` 存在时继续调用 `saveEventData`；如果事件里还有 `revenue` / `currency`，还会额外触发收入写入。源码证据：仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/queries/sql/events/saveEvent.ts:72-168`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `src/queries/sql/events/saveEvent.ts:170-276`; 仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `db/clickhouse/schema.sql:280-289`

## 这个模型为什么对 Umami 合适

### 1. 事件列表和属性分析可以同时成立

如果所有属性都只放在 `website_event.properties_json` 一类字段里，`Events` 好做，但属性筛选、属性分布、Revenue 二次聚合都会变重。Umami 把明细和属性拆开之后，`website_event` 保持干净，`event_data` 负责动态属性。

### 2. identify 不会污染事件事实表

`session_data` 让 identify 上下文不必塞回每一条事件。这样做的代价是：如果你要问“某条历史事件对应当时用户的 profile 是什么”，需要明确到底看事件时刻快照，还是看当前 session/identify 状态。

### 3. ClickHouse 可以继续向聚合层延伸

`website_event` 是事实层，`website_event_stats_hourly` 是上面的聚合层。也就是说 Umami 并不是只有“事件表 + 属性表”，它还进一步做了物化视图和小时聚合。源码证据：仓库: `umami`, commit: `c78ff36db0c82e13c86e5073020472c6546313a3`, file: `db/clickhouse/schema.sql:93-170`

## 代价和限制

| 代价 | 说明 |
| --- | --- |
| 写入链路更长 | 一次自定义事件可能先写 `website_event`，再写多行 `event_data`，收入场景还会再多一步 |
| identify 语义不是“全局用户画像表” | `session_data` 更像 session / distinct id 上下文，不是独立的用户主数据中心 |
| 属性键治理仍然重要 | 扁平化能解决查询问题，但解决不了“乱命名”“同义字段并存”问题 |

## 给 SimpleTrack / analytics-core 的直接启发

1. `events` 明细和“可过滤属性索引”应该分开建模，不要只押一个 JSON 大字段。
2. “用户属性”最好先定义语义边界，再决定是做独立 profile 表，还是像 Umami 一样先做 session/identify KV。
3. 读侧如果要支持 Revenue、Breakdown、Filters，typed KV 比原地 JSON 解析更稳。
4. Umami 的写入边界值得借鉴，但不能原样搬。`analytics-core` 已经有 `collect -> EventBus -> ingestion -> storage` 分层，应该吸收的是字段分层思路，不是 Next.js route 里直接写库的结构。
