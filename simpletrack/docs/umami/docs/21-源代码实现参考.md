# 21-源代码实现参考

> 状态：已确定
> 来源：`references/umami/` 只读源码快照，来自 `https://github.com/umami-software/umami.git` 的 `master` 分支 `c78ff36db0c82e13c86e5073020472c6546313a3`
> 用途：把 Umami 的实现结构翻译成 SimpleTrack 可复用的设计输入，不把第三方源码直接搬进生产实现。

## 结论

Umami 源码适合作为 SimpleTrack 分析产品和数据面的强参考，尤其是四个方向：

1. `tracker -> /api/send -> event/session write` 的最短采集链路。
2. `Website / Session / WebsiteEvent / EventData / SessionData` 的分析对象拆分。
3. `Realtime / Events / Reports` 共用过滤器、时间范围和 SQL 查询层的读侧组织方式。
4. PostgreSQL 与 ClickHouse 双后端的字段口径、索引和聚合表设计。

但 SimpleTrack 不能照搬 Umami 的运行时边界。Umami 把采集 HTTP 入口、会话计算、URL/UTM 解析、bot/IP 过滤和写库分发都压在 Next.js API route 里；SimpleTrack 已经在 `analytics-core` 建立 `collect.Handler`、EventBus、ingestion worker、`EventWriter`、`EventReader` 和 `EventQueryBuilder` 边界，后续必须继续沿用这些契约。

## 代码审阅摘要

审阅结论：`COMMENT`。Umami 源码可作为高质量实现参考，但它的部分架构选择服务于自托管 Next.js 单体，不应直接作为 SimpleTrack 的生产边界。

### 高价值模式

- 采集 SDK 很短：`tracker/index.js` 只负责收集页面、语言、屏幕、标题、URL、referrer、事件名和事件属性，再发到统一 endpoint。SimpleTrack tracker P1 可以保持同样克制。
- 事件类型清晰：`EVENT_TYPE` 把 pageview、custom event、link event、pixel event、performance 分成整数枚举；SimpleTrack 可以保留类似逻辑枚举，但命名要贴合 `analytics-core` 的 `source_type / event_name`。
- 动态属性单独入表：`event_data` 把属性 flatten 成 `data_key + string/number/date/data_type`，避免把所有属性塞成不可查询 JSON。
- 读侧集中：Realtime、Events、Reports 都复用 filter、date range、query params 和 backend-specific SQL helpers，降低前端页面各写一套查询逻辑的风险。
- ClickHouse 设计有明确冷热分层：明细表承接事件，materialized view 生成小时级聚合，projection 优化常见 page/referrer 查询。

### 不能直接照搬

- `src/app/api/send/route.ts` 过重：HTTP route 同时做 schema 校验、网站查找、客户端识别、bot/IP 过滤、session/visit 生成、URL 解析、UTM 抽取和写库调用。SimpleTrack 必须保持 `httpapi` 与 `collect.Handler` 分层。
- 写入路径缺少统一事务边界：PostgreSQL 路径先写 `website_event`，再写 `event_data` 和 `revenue`；SimpleTrack 的 ingestion worker 应把幂等状态、事件明细和属性写入放进可验证的提交/回滚语义。
- `runQuery` 由环境变量选择存储后端，适合应用单体但不适合 `analytics-core`。SimpleTrack 应继续使用 `EventWriter`、`EventReader`、`TableRouter` 和 query builder adapter。
- Kafka 发送错误在 `src/lib/kafka.ts` 中被捕获并打印，调用侧可能无法稳定感知失败。SimpleTrack 已经有 EventBus ack / retry / dead-letter 设计，不能引入静默失败。
- 原始 SQL 拼接需要严格白名单。Umami 已有 `FILTER_COLUMNS` 和 operator mapping，但 SimpleTrack 的 Go query builder 仍应把字段、排序、分页、表路由全部限制在结构化 plan 内。
- Umami 的 Team、Boards、Links、Pixels、Replays、Revenue、Attribution 已经进入源码主干；这些能力对 SimpleTrack 是 P2/P3 参考，不应进入 P1 产品范围。

## P1 映射

| SimpleTrack P1 能力 | 参考 Umami 实现 | 落地方式 |
| --- | --- | --- |
| Tracker SDK | `src/tracker/index.js` | 先实现 pageview、自定义事件、`identify` 可选；保留 `beforeSend`、domain allowlist、DNT 作为后续增强 |
| Collect API | `src/app/api/send/route.ts` | 只借鉴 payload 口径；继续由 Go `collect.Handler` 标准化请求，再发布到 EventBus |
| EventBus / ingestion | `saveEvent.ts` 的写入分发、`kafka.ts` 的队列思路 | 不直接在 HTTP 请求内写 ClickHouse；继续使用 Redis Stream pending 优先、ack、dead-letter 和幂等 guard |
| Event storage | `prisma/schema.prisma`、`db/clickhouse/schema.sql` | P1 保留 `events` 逻辑模型，物理表按 tenant/project/source 路由；属性表可参考 `event_data` |
| Realtime | `getRealtimeData.ts`、`getRealtimeActivity.ts`、`api/realtime/[websiteId]/route.ts` | 固定短窗口，组合最近活动、pageviews、sessions，作为接入验收页 |
| Events table | `api/websites/[websiteId]/events/route.ts`、`getWebsiteEvents.ts` | 复用 `EventQueryBuilder` 和 `EventReader`，支持时间范围、事件类型、搜索、分页 |
| Website 设置与 snippet | `WebsiteAddForm`、`settings/websites`、tracker config | 在 Supastarter shell 内实现 source 创建、安装 snippet、接入状态 |
| Goal 最小闭环 | `reports/getGoal.ts`、`EVENT_TYPE` | P1 只做一个关键事件目标，不引入完整 Reports 管理 |
