# SimpleTrack 数据模型与事件字典

> 用途：把 Umami 的采集、事件、session data、Distinct ID、UTM、Revenue 口径转成 SimpleTrack 可直接使用的埋点和数据模型建议。它不是新的功能解读，而是实施时的命名约束和字段边界。

## 设计原则

1. 先让事件能回答业务问题，再增加属性。
2. 事件名描述业务动作，不描述 DOM 结构或代码实现。
3. 事件属性描述这一次动作，会话属性描述这一趟访问，Distinct ID 描述同一个业务用户。
4. 不把邮箱、手机号、姓名、cookie、token、订单明细写入 analytics payload。
5. UTM、Revenue、Attribution 共享同一套 `campaign`、`plan`、`cohort label` 口径。

## 数据层边界

| 数据层 | 放什么 | 不放什么 | 示例 |
| --- | --- | --- | --- |
| `pageview` | 页面访问、路径、标题、来源 | 按钮动作、业务状态 | `/pricing.html`、`/app/dashboard.html` |
| `event` | 一次明确业务动作 | 页面曝光、长期用户标签 | `signup_completed`、`checkout_completed` |
| `event data` | 这次动作的上下文 | 原始表单、token、完整订单明细 | `plan`、`surface`、`step`、`revenue` |
| `session data` | 这一趟访问的稳定上下文 | 永久用户身份、敏感个人信息 | `role`、`workspaceSize`、`cohort` |
| `distinct id` | 不可直接反查个人信息的稳定用户标识 | 邮箱、手机号、姓名 | `batch-user-0001`、内部 user id hash |
| `utm` | 外部流量来源标签 | 内部行为状态 | `utm_source`、`utm_medium`、`utm_campaign` |

## 命名规范

事件名建议使用：

- 小写英文。
- 下划线连接。
- 动词过去式或动作完成语义。
- 以业务动作为中心。

推荐：

- `signup_started`
- `signup_completed`
- `install_started`
- `first_event_sent`
- `checkout_completed`

不推荐：

- `button_click_01`
- `modal_submit`
- `page_btn_top_clicked`
- `foo_event`

字段名建议：

- 面向报表的字段保持稳定，例如 `plan`、`campaign`、`cohort`。
- 页面或组件位置用 `surface`。
- 漏斗步骤用 `step` 或 `goal_name`。
- 金额用 `revenue`，币种用 `currency`。
- 逻辑时间用 `logical_day`、`synthetic_date`，不要伪装成真实入库时间。

## 固定维度字典

| 字段 | 取值 | 用途 |
| --- | --- | --- |
| `plan` | `free` / `trial` / `pro_monthly` / `pro_annual` | Breakdown、Revenue、Attribution |
| `campaign` | `producthunt_launch` / `google_brand` / `google_competitor` / `docs_seo` / `linkedin_founder` / `email_nurture` | UTM、Attribution、Compare |
| `cohort` | `spring_launch` / `self_serve_wave` / `paid_pilot` | Cohort label、Retention、Filters |
| `role` | `founder` / `growth_lead` / `engineer` / `product_manager` / `ops` | Segments、Breakdown |
| `workspaceSize` | `1-10` / `11-50` / `51-200` / `201-500` | Segments、ICP 分析 |
| `currency` | `USD` | Revenue |

这些字段已经和 [真实业务数据方案](./真实业务数据方案.md)、[功能打通矩阵](./功能打通矩阵.md) 以及 `tracking-demo` 保持一致。

这里的 `cohort` 字段是一个固定维度标签，更准确地说是 `cohort label`，用于表示用户来自哪一波启动、试用或付费生命周期。它不等同于 Umami 的 Cohorts 功能对象；Cohorts 功能对象是基于时间窗口和行为条件动态定义的人群。

## 推荐事件字典

| 事件名 | 业务含义 | 建议触发点 | 必带属性 | 支撑能力 |
| --- | --- | --- | --- | --- |
| `pricing_viewed` | 查看价格页 | 进入 `pricing.html` 或价格 CTA | `plan`、`campaign`、`cohort` | Compare、Breakdown |
| `compare_opened` | 查看竞品对比 | 进入 `compare.html` 或点击 compare CTA | `campaign`、`role` | Journeys、Attribution |
| `signup_started` | 开始注册 | 打开或提交注册表单前 | `plan`、`campaign`、`cohort` | Goals、Funnels |
| `signup_completed` | 注册完成 | 注册成功页或服务端确认 | `plan`、`campaign`、`cohort` | Goals、Funnels、Attribution |
| `workspace_created` | 创建 workspace | onboarding 创建成功 | `workspaceSize`、`role`、`plan` | Activation、Segments |
| `install_started` | 开始安装 SDK | 点击安装或进入安装页 | `surface`、`plan` | Funnels、Journeys |
| `sdk_install_completed` | SDK 安装完成 | 安装校验成功 | `surface`、`plan` | Goals、Funnels |
| `first_event_sent` | 首个业务事件发送 | 用户完成首个事件上报 | `surface`、`plan` | Activation、Retention |
| `dashboard_viewed` | 查看产品内 dashboard | 进入 dashboard | `plan`、`role`、`workspaceSize` | Sessions、Retention |
| `filter_applied` | 使用过滤器 | 应用过滤条件 | `field`、`match`、`segment` | Filters、Segments |
| `segment_opened` | 打开 Segment | 进入 segment 页或应用分组 | `segment`、`cohort` | Segments、Cohorts |
| `cohort_opened` | 打开 Cohort | 进入 cohort 页或查看人群 | `cohort`、`logical_day` | Cohorts、Retention |
| `checkout_started` | 开始结账 | 点击 checkout 或进入结账页 | `plan`、`campaign`、`cohort` | Funnels、Attribution |
| `checkout_completed` | 支付完成 | 支付成功或订阅确认 | `revenue`、`currency`、`plan`、`campaign`、`cohort` | Revenue、Attribution、Goals |
| `subscription_upgraded` | 订阅升级 | 升级成功 | `revenue`、`currency`、`plan`、`campaign`、`cohort` | Revenue、Retention |
| `billing_viewed` | 查看账单 | 进入 billing 页 | `plan`、`role` | Sessions、Revenue |

## Revenue 事件口径

收入事件只建议在业务确认成功后产生：

- `checkout_completed`
- `subscription_upgraded`

必带字段：

- `revenue`
- `currency`
- `plan`
- `campaign`
- `cohort`

建议规则：

1. `revenue` 使用主币种金额，例如 `29` 或 `290`，不要混用 cents。
2. `currency` 当前固定 `USD`。
3. 不在 `checkout_started` 上记收入。
4. 如果未来接真实订单系统，收入事件应以后端订单状态为准。
5. Attribution 只消费确认后的收入事件，不消费点击或表单事件。

## UTM 口径

最小必填：

- `utm_source`
- `utm_medium`
- `utm_campaign`

可选：

- `utm_content`
- `utm_term`

命名规则：

1. 全部小写。
2. 用下划线，不用空格。
3. campaign 名称必须进入 campaign 字典。
4. 同一活动不能同时出现 `spring`、`Spring2026`、`spring_launch`。
5. 报表、链接生成器和批量脚本共用同一套 campaign 字典。

## Session Data 建议

适合放入 session data：

- `plan`
- `cohort`
- `role`
- `workspaceSize`
- `campaign`
- `source`
- `medium`

不适合放入 session data：

- 邮箱
- 手机号
- 姓名
- token
- cookie
- 原始订单号
- 原始表单内容

Session data 的目标是解释“这一趟访问是什么上下文”，不是把用户画像和业务数据库复制进分析系统。

## Distinct ID 建议

Distinct ID 应该满足：

- 稳定。
- 不超过官方限制。
- 不直接暴露个人身份。
- 可从业务系统映射，但不能让外部读者直接反查。

推荐形态：

- 内部用户 id 的哈希。
- 不含邮箱或手机号的业务用户 id。
- demo 中的 `batch-user-0001`、`persona-001` 这类合成 id。

不推荐：

- `user@example.com`
- `13800000000`
- `张三`
- session token

## Filter / Segment / Cohort 字段分层

| 字段 | 适合 Filter | 适合 Segment | 适合 Cohort | 说明 |
| --- | --- | --- | --- | --- |
| `plan` | 是 | 是 | 可选 | 稳定商业分层 |
| `campaign` | 是 | 是 | 可选 | 增长来源切片 |
| `cohort` | 是 | 是 | 是 | 当前 demo 作为 cohort label；不要把字段本身等同于 Cohort 功能对象 |
| `role` | 是 | 是 | 否 | ICP 和人群画像 |
| `workspaceSize` | 是 | 是 | 否 | B2B 分层 |
| `logical_day` | 可选 | 否 | 是 | 留存和逻辑时间窗口 |
| `synthetic_date` | 可选 | 否 | 是 | 仅用于仿真数据解释 |

关键区别：

- Filter 是临时切片。
- Segment 是保存后的过滤口径。
- Cohort 是带时间和行为条件的人群定义。

## 最小埋点版本

如果 SimpleTrack 只做第一版，建议先接这些：

1. `signup_started`
2. `signup_completed`
3. `install_started`
4. `sdk_install_completed`
5. `first_event_sent`
6. `checkout_completed`

第一版全局上下文字段：

- `plan`
- `campaign`
- `cohort`
- `role`
- `workspaceSize`

这里的“全局上下文字段”不是说每个事件的 event data 都必须重复带完，而是说每次访问、身份或关键事件至少要能通过 `session data`、`identify()` 或事件属性拿到这些业务上下文。单个事件仍以“推荐事件字典”里的必带属性为准。

第一版暂不强求：

- 复杂 attribution model。
- 多币种收入。
- 全站 session replay。
- 全量 cohort 生命周期分析。

## 实施顺序

1. 先冻结事件字典：确认事件名、触发时机、必带属性和可选属性。
2. 再冻结固定维度字典：统一 `plan / campaign / cohort / role / workspaceSize / currency` 的取值。
3. 接入 tracker、`identify()` 和 session data，确保全局上下文能进入 Umami。
4. 再上报 UTM 与收入事件：UTM 先保证 `source / medium / campaign`，Revenue 只认支付成功或订阅确认。
5. 最后进入 Cloud UI 验收：先看 Events / Properties，再看 Filters / Goals / Revenue / Attribution。

## 复验方式

1. 用 [执行与复验手册](./执行与复验手册.md) 启动仿真站。
2. 先跑 `debug.html` 验证 `track()` 和 `identify()`。
3. 再跑真实浏览器 persona。
4. 最后跑 `bulk-send.mjs` 批量事件。
5. 在 Cloud UI 里按 `Events -> Properties -> Filters -> Goals -> Revenue -> Attribution` 顺序检查。

证据写法：

- 如果只看到事件进入 Events，写“事件输入侧已验证”。
- 如果 Revenue 引用当前截图，必须说明 `$11.86k / 355 orders` 是多轮 smoke、旧 UA、修正 UA 和重跑后的累积站点结果，不等同于一次干净样本。
- 如果 Attribution 引用当前截图，可以写“checkout conversion step 已出现非零结果态”，但不能写成因果 ROI 或渠道真实贡献已被证明。
