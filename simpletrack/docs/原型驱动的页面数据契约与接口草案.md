# SimpleTrack 原型驱动的页面数据契约与接口草案

> 状态：进行中，已按 `simpletrack/prototype/simpletrack-enterprise-mvp/` 重写第一版 P1 页面 contract
> 最近更新：2026-05-06
> 目的：把当前 Next.js 企业控制台原型、`simpletrack-saas` 生产控制面和 `simpletrack-anaysitics-service` 数据面收敛到同一组页面数据契约，避免继续沿用旧 `simpletrack-umami-inspired` 的页面口径。

---

## 一、适用范围

本草案覆盖当前 P1 已确认或正在落地的页面/界面：

| 页面或界面 | 来源 | P1 定位 |
| --- | --- | --- |
| `/onboarding` | `simpletrack-enterprise-mvp` 原型 | 创建站点、复制 tracker snippet、验证第一条信号 |
| `/dashboard` | `simpletrack-enterprise-mvp` 原型 | 站点健康、核心指标、Realtime intake 摘要、Top pages / referrers |
| `/events` | `simpletrack-enterprise-mvp` 原型 + `simpletrack-saas` 生产页面 | 事件列表、事件字典、属性分布、服务端筛选和分页 |
| `/goals` | `simpletrack-enterprise-mvp` 原型 | P1 最小 Goal 定义和 conversion rate 展示 |
| `/settings` | `simpletrack-enterprise-mvp` 原型 | Website runtime source 设置、domain allowlist、事件字典、属性白名单和 ingestion rules |
| `Websites` | `simpletrack-saas` 生产控制面 | 真实 Website source 列表、创建、编辑、enable/disable、delete、write key 展示 |
| `Realtime` | `simpletrack-saas` 生产页面 | 内部 readback 的最近事件和基础实时指标 |
| `Marketing / docs / quickstart` | Supastarter `apps/marketing` / `apps/docs` | 公开产品入口、定价语义、接入文档入口 |

P1 明确不把以下旧原型页面作为页面 contract：

- `team.html`
- `funnels.html`
- `insights.html`
- revenue / attribution 页面
- replay / performance 页面
- boards / share / API key 页面

这些能力进入 P2/P3 或待评审事项，不应在 P1 后端 contract 中提前扩张。

---

## 二、契约原则

- 页面只拿自己需要的数据，不让前端直接拼接底层 ClickHouse 表名、复杂 SQL 或分析引擎内部参数。
- `simpletrack-saas` 拥有控制面配置 CRUD：organization、Website、write key、domain allowlist、bot/internal traffic 配置、quota、subscription gate。
- `simpletrack-anaysitics-service` 拥有运行时执行：`/tracker.js`、`/collect`、write key 解析、Origin/CORS、bot/internal traffic 过滤、调用 `analytics-core`、内部 Realtime / Events readback。
- `analytics-core` 保持业务无关 Go library：collect 标准化、EventBus、ingestion、ClickHouse writer/reader、query builder、property filter。
- 前端 contract 使用页面级或 BFF 级接口；内部实现可以复用 `analytics-core` 的 `EventReader` / `EventQueryBuilder`，但不直接暴露底层契约给浏览器。
- P1 contract 只覆盖“数据管道活了 + 公开产品入口”：pageview、自定义事件、Realtime、Events、Goal 最小闭环和 quickstart。

## 三、核心实体

### 1. Organization

字段：

- `id`
- `name`
- `plan_name`
- `plan_status`
- `active_source_limit`
- `created_at`

### 2. WebsiteSource

字段：

- `id`
- `organization_id`
- `name`
- `domain`
- `enabled`
- `write_key`
- `allowed_origins`
- `bot_user_agents`
- `internal_cidrs`
- `internal_ips`
- `include_client_fingerprint`
- `session_salt_configured`
- `client_hash_salt_configured`
- `last_seen_at`
- `created_at`
- `updated_at`

说明：

- `write_key` 是公开 tracker 接入凭据，不可作为 privacy salt。
- `session_salt` 和 `client_hash_salt` 只存在于服务端 runtime source config，不下发到浏览器。
- disabled source 不占用 active source limit。

### 3. InstallVerification

字段：

- `source_id`
- `tracker_status`
- `snippet_copied_at`
- `first_pageview_at`
- `first_event_at`
- `last_accepted_signal_at`
- `last_rejected_reason`

### 4. RealtimeSignal

字段：

- `id`
- `occurred_at`
- `event_name`
- `event_type`
- `path`
- `referrer`
- `distinct_id`
- `session_id`
- `visit_id`
- `status`
- `properties_summary`

### 5. EventRecord

字段：

- `id`
- `occurred_at`
- `event_name`
- `path`
- `referrer`
- `distinct_id`
- `session_id`
- `visit_id`
- `country`
- `browser`
- `os`
- `device`
- `properties`
- `user_properties`

说明：

- `visit_id` 已升级为正式写路径字段：collect 写入前确定，ClickHouse event / `_properties` 表持久化，Realtime / Events 直接读取存储字段。

### 6. EventDictionaryItem

字段：

- `event_name`
- `display_name`
- `description`
- `status`
- `required_properties`
- `last_seen_at`
- `count`
- `unique_visitors`

### 7. PropertyFilterPolicy

字段：

- `source_id`
- `scope`
- `property_name`
- `type`
- `allowed_operators`
- `allowed_values`

说明：

- Events 属性过滤必须同时经过 runtime source config 的 `allowed_property_filters` 和 `analytics-core` query builder 白名单。

### 8. GoalDefinition

字段：

- `id`
- `source_id`
- `name`
- `type`
- `rule`
- `denominator`
- `status`
- `conversions`
- `population`
- `conversion_rate`
- `updated_at`

### 9. IngestionRule

字段：

- `rule`
- `detail`
- `mode`

P1 默认规则：

- 拒绝 PII key：`email`、`phone`、`name`、`token`、`cookie`
- 拒绝 nested JSON
- 执行 domain allowlist
- unknown events 进入 review/quarantine 语义，不阻塞已允许 source 的基础收数

## 四、页面 contract

## 1. `/onboarding`

页面目的：

- 创建第一个 Website source。
- 生成 `data-write-key` tracker snippet。
- 复制 snippet 并验证第一条 pageview 或自定义事件进入 Realtime。

前端需要：

- 当前组织和套餐限制。
- 创建 Website source 表单。
- tracker snippet。
- 安装验证状态。

建议接口：

### `GET /api/app/onboarding`

响应：

```json
{
  "organization": {
    "id": "org_001",
    "name": "Acme",
    "plan_name": "Free",
    "active_source_limit": 1
  },
  "source": {
    "id": "src_001",
    "name": "Acme marketing",
    "domain": "acme.example",
    "enabled": true,
    "write_key": "stk_live_01HZX",
    "last_seen_at": null
  },
  "verification": {
    "tracker_status": "waiting",
    "first_pageview_at": null,
    "first_event_at": null,
    "last_rejected_reason": null
  }
}
```

### `POST /api/app/websites`

请求：

```json
{
  "name": "Acme marketing",
  "domain": "acme.example",
  "enabled": true
}
```

响应：

```json
{
  "source": {
    "id": "src_001",
    "name": "Acme marketing",
    "domain": "acme.example",
    "enabled": true,
    "write_key": "stk_live_01HZX",
    "allowed_origins": ["https://acme.example"]
  },
  "snippet": {
    "html": "<script defer src=\"https://cdn.simpletrack.dev/tracker.js\" data-write-key=\"stk_live_01HZX\"></script>"
  }
}
```

### `POST /api/app/websites/:source_id/snippet-copied`

用途：

- 记录用户复制 snippet 的操作，用于 onboarding 引导。

响应：

```json
{
  "ok": true
}
```

### `GET /api/app/websites/:source_id/verification`

响应：

```json
{
  "tracker_status": "verified",
```

- 管理 Website source。

## 5. `/events`

页面目的：

- 展示事件列表、事件字典和属性证据。
- 支持 P1 需要的服务端筛选、分页、排序和单条属性过滤。

前端需要：

- 当前 source。
- 时间窗口。
- Events table。
- event name / distinct id / property filter / sort / pagination 查询状态。
- selected event 的属性分布或摘要。

建议接口：

### `GET /api/app/websites/:source_id/events`

- `distinct_id`
- `property_filter`
- `limit`
- `offset`

```json

      "display_name": "First event sent",

      "required_properties": ["surface", "plan"]
    }
  ]

```

### `GET /api/app/websites/:source_id/events/:event_name/properties`

响应：

```json
{
  "event_name": "first_event_sent",
  "properties": [
    {
      "name": "plan",
      "type": "string",
      "distribution": [
        { "value": "trial", "count": 169 },
        { "value": "free", "count": 148 }
```

实现说明：

- browser 不直接持有内部 readback token。

## 五、内部服务边界

### `simpletrack-saas`

负责：

- organization / Website source CRUD
- active source limit
- write key 管理
- runtime source config 管理
- Realtime / Events 页面服务端 readback helper
- pricing / subscription gate
- marketing / docs / mail preview

不负责：

- 高频 `/collect`
- 直接写 ClickHouse
- 直接把内部 query token 下发浏览器

### `simpletrack-anaysitics-service`

负责：

- `/tracker.js`
- `OPTIONS /collect`
- `POST /collect`
- write key -> runtime source config 解析
- Origin/CORS
- bot/internal traffic 过滤
- client enrich / session resolver stage
- 调用 `analytics-core` collect / EventBus
- 可选同进程 ingestion worker
- 内部 `/v1/realtime` 和 `/v1/events`

不负责：

- 创建或修改 Website source 配置
- subscription / billing / organization 生命周期

### `analytics-core`

负责：

- collect request 标准化
- `EventEnvelope`
- `EventBus`
- ingestion processor
- ClickHouse event/property writer
- `EventQueryBuilder`
- `EventReader`
- typed property filter

不负责：

- SimpleTrack 控制面业务
- Browser SDK 静态交付
- pricing、team、billing、subscription

## 六、冻结建议

进入下一轮后端和生产页面开发前，至少冻结以下 5 件事：

1. P1 页面集合是否就是 `onboarding / dashboard / realtime / events / goals / settings / websites / marketing / docs`。
2. `Realtime` 在原型里是否继续并入 `/dashboard`，生产 SaaS 是否保留独立 Realtime 页面。
3. `visit_id` 已从 readback provisional 字段升级为正式事件写路径 schema，后续只评审 salt 轮换、cookie/no-cookie 和 Sessions 产品化。
4. Goal P1 是否只支持 event/page 简单规则，不支持 funnel rule。
5. `allowed_property_filters` 的管理入口是否放在 Website settings，还是先由服务端配置种子提供。

如果这些问题不冻结，后端 contract 会在 P1 开发中持续变动。
