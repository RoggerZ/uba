# Litlyx 视角的 SimpleTrack 落地评审清单

> 用途：把 Litlyx 这轮调研里最值得落地的能力，整理成一份可直接过会的核对清单。

## 1. 首次接入

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| 登录后默认落点是否是安装入口 | 新用户第一步是接入，不是看空图表 | 应优先实现 |
| 是否同时支持 Script / GTM / 框架入口 | 降低不同团队的接入门槛 | 应优先实现 |
| 是否有 AI prompt 辅助接入 | 能缩短“复制代码”和“改项目”的距离 | 可以先做轻量版本 |
| Workspace ID 和安装代码能否回查 | 用户很少一次配置成功后就永远不回看 | 应优先实现 |
| 是否有明确的安装验证入口 | 没有验收点，用户很难确认成功 | 应优先实现 |

## 2. 首批数据验收

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| 是否有 Raw Events 明细表 | 这是排障最快的入口 | 应优先实现 |
| 是否能在 Product 看到基础聚合结果 | 只看明细不够，用户还要看到“系统已经活了” | 应优先实现 |
| 是否区分示例数据和真实数据 | 示例数据用于教育，真实数据用于验收 | 必须明确区分 |
| 是否有最小事件验证路径 | 没有最小闭环时，排障成本会很高 | 应优先实现 |

## 3. 分析教育

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| 空态是否仍保留分析骨架 | 用户需要提前理解未来能看到什么 | 应优先实现 |
| 是否提供 `Show test data` | 能快速解释价值，减少空态挫败 | 应优先实现 |
| 是否让用户看到 Funnel / User Flow / Metadata 的未来形态 | 有助于提前建立分析心智 | 可以先做骨架和示例态 |
| `Setup events` 这类 CTA 是否说明真实去向 | 否则容易被误判成无响应 | 必须解释清楚 |

## 4. 增长闭环

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| Marketing 是否是独立分析层 | 流量和事件分析不应混成一团 | 建议独立 |
| 是否内嵌 UTM 生成器 | 看完结果后最好能直接继续投放 | 应优先实现 |
| Reports 是否是模板中心而不是单一导出按钮 | 高级能力更容易被理解和消费 | 建议优先做模板中心 |
| 是否提供 Sample / 样张预览 | 用户需要在付费前看到价值 | 应优先实现 |
| AI 是否提供任务型 prompt | AI 需要明确入口，不然很难被使用 | 应优先实现 |

## 5. 治理能力

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| 是否单独处理域名治理 | 测试域名、预览环境和脏数据会污染结果 | 应优先实现 |
| 是否支持 IP 排除 | 内网和办公流量需要隔离 | 建议尽早实现 |
| 是否支持 bot 流量治理 | 爬虫和机器人会破坏分析可信度 | 建议尽早实现 |
| 删除和 sanitization 是否有风险提示 | 这是高风险动作，不能做成普通设置 | 必须实现 |
| 是否保留治理操作的审计语义 | 高风险配置需要可追踪 | 建议在增强版补上 |

## 6. 分享与协作

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| 是否区分外部只读分享和内部成员协作 | 这是两种不同权限模型 | 必须分开 |
| Shareable links 是否独立建模 | 外部客户通常只需要只读访问 | 建议独立入口 |
| Members 的门槛是否解释清楚 | loading 态会被误解为产品故障 | 必须解释清楚 |
| 受限态是否明确是套餐限制还是权限限制 | 有助于降低客服和产品困惑 | 必须实现 |

## 7. 商业化与门槛表达

| 检查项 | 为什么重要 | 当前建议 |
| --- | --- | --- |
| 高级能力是否明确标注 premium | 不要用“空白页”伪装未解锁能力 | 应优先实现 |
| Reports 禁用态是否说明不能生成的原因 | 用户需要知道是权限问题而不是操作问题 | 必须实现 |
| Plans 和 FAQ 是否同屏解释边界 | 可以减少用户猜测和咨询成本 | 建议实现 |
| 门槛页是否带升级 CTA | 仅提示锁定不够，需要给下一步动作 | 建议实现 |

## 8. 推荐结论

### 必须先做

- 安装入口
- Workspace ID / 安装代码回查
- 最小事件验证链路
- Raw Events
- Product 骨架
- `Show test data`
- UTM 生成器
- Reports 模板中心和样张
- 清楚的受限态表达

### 可以第二阶段做

- IP / bot 治理
- AI 任务模板
- Shareable links
- Plans / FAQ 强化

### 需要更慎重推进的

- 正式 PDF 生成
- Members 邀请与权限体系
- 域名删除与 sanitization 审计

一句话判断：

- 如果某个能力不能帮助用户“更快接入、看懂数据、继续行动”，它就不该挤进最前面的版本。
