## 2. 当前可复现的两条主链

### A. 真实事件验证链

1. 在 `tracking-demo/` 启动静态服务：

```bash
python -m http.server 4174
```

2. 打开：

```text
http://localhost:4174/index.html
```

3. 在后台 `Web` 或 `Settings / General` 复制 `Workspace ID`
4. 在 demo 页面点击 `Load tracker`
5. 点击 `Fire event A`、`Fire event B`
6. 需要补量时运行：

```bash
node bulk-send.mjs --pid YOUR_WORKSPACE_ID --users 6 --sessions-per-user 2
```

7. 回到后台复核：
   - `Product`
   - `Raw Data`

### B. 页面截图复核链

1. 在运行前设置环境变量：

```powershell
$env:LITLYX_EMAIL='your_email'
$env:LITLYX_PASSWORD='your_password'
```

2. 执行：

```bash
node capture_litlyx_research.cjs
```

3. 脚本会自动：
   - 登录 Litlyx
   - 依次采集 `Phase 01` 到 `Phase 05`
   - 对截图里的邮箱和 24 位 workspace id 做可见文本遮挡

## 3. 关键脚本说明

| 脚本 | 作用 | 说明 |
| --- | --- | --- |
| `capture_litlyx_research.cjs` | 分阶段截图 | 使用 Playwright Node API 和 `msedge` channel，不是 Playwright CLI |
| `tracking-demo/app.js` | 浏览器侧事件演示 | 动态注入官方浏览器脚本，运行时粘贴 Workspace ID |
| `tracking-demo/send-event.mjs` | 单条事件发送 | 适合快速 smoke test |
| `tracking-demo/bulk-send.mjs` | 批量事件发送 | 适合把 Product 和 Raw Events 推到有数状态 |

## 4. 当前已知限制

| 限制 | 当前表现 | 处理方式 |
| --- | --- | --- |
| Browser Use 插件 | 当前只能读取 tab 元信息，DOM / screenshot 动作未稳定可用 | 截图复核改用本地 Node 脚本 |
| Reports 正式生成 | 当前 Free trial 下报告卡片和 `Generate report` disabled | 只记录样张和门槛，不触发正式生成 |
| Shareable links 创建 | 会改变外部访问状态 | 只记录空态，不点击 `Create link` |
| Domain data / Sanitize Domains | 会删除或清理数据 | 只记录页面态，不执行 |
| Members | 当前只有 loading / contact owner 态 | 记录为产品发现，不把它写成“已可用” |

## 5. 敏感信息规则

- 不把账号、密码、workspace id、token 写入仓库文件。
- `tracking-demo/` 只允许运行时粘贴 `Workspace ID`，不要写死。
- 登录截图必须遮挡邮箱和密码输入内容。
- Settings 页和安装页里出现的 24 位 workspace id 必须遮挡。
- 根目录 `snapshots/img*.png` 只作为历史临时截图保留，已经完成遮挡，不作为正式编号资产。

## 6. 当前不应执行的动作

- 不点击 `Create link`
- 不执行 Domain data 删除
- 不执行 Domain sanitization
- 不尝试 Members 邀请
- 不把 Reports disabled 态绕过成正式生成

这些动作都会改变云端状态，只有在用户明确授权后才继续。

## 7. 每次继续前的最小检查

1. 先看 [../快照进度.md](../快照进度.md) 确认当前缺口
2. 再看 [功能打通矩阵.md](./功能打通矩阵.md) 确认哪些能力是“已验证 / 页面态 / 受限态”
3. 如果要补截图，先跑 `capture_litlyx_research.cjs`
4. 如果要补数据，先跑 `tracking-demo` 和 `bulk-send.mjs`
5. 每次修改文档后，至少复查：
   - 先运行 `python validate_litlyx_assets.py`
   - 再根据输出处理相对链接、快照索引、UTF-8 或敏感信息问题

## 8. 本地静态校验

在 `simpletrack/docs/litlyx/` 下运行：

```bash
python validate_litlyx_assets.py
```

它会检查：

- Markdown 相对链接
- `快照索引.md` 中的 `.png` 引用
- `snapshots/` 和 `litlyx-reference/` 下是否有未索引图片
- 文本文档 UTF-8 解码
- 邮箱、24 位 workspace id 和已知敏感关键词

## 9. 后续最推荐的推进顺序

1. 如果账号能力变化，优先复核 `Reports` 正式生成链路
2. 如果要补增长样本，优先为 `Marketing` 构造真实 UTM 来源流量
3. 如果要补协作能力，优先搞清 `Members` 的真实限制原因
4. 如果要进入高风险动作，先单独确认授权边界，再执行创建或删除类操作
