# Litlyx 数据模型与事件字典

> 用途：把本轮 Litlyx 调研里已经真实发送的事件、字段和载荷形式整理出来，同时补一层适合继续扩展的命名边界。

## 1. 当前接入模型

本轮验证的 Litlyx 事件模型非常轻：

- 浏览器脚本通过 `<script defer data-workspace="workspace_id" ...>` 接入
- 自定义事件通过 `Lit.event(eventName, { metadata })` 发送
- 终端和批量脚本通过 `POST https://broker.litlyx.com/event` 直接上报

这意味着当前验证重点不是“多对象分析模型”，而是这三层：

1. tracker 是否成功加载
2. 事件名是否进入 Product 聚合
3. metadata 是否能成为后续 `Analyze event metadata` 的输入

## 3. 当前已验证的 payload 结构

### 浏览器侧

```js
Lit.event("demo_signup_click", {
  metadata: {
    surface: "tracking-demo",
    section: "pricing",
    variant: "hero-cta",
  },
});
```

### 终端侧

```json
{
  "pid": "WORKSPACE_ID",
  "name": "demo_terminal_event",
  "metadata": "{\"source\":\"tracking-demo\",\"surface\":\"tracking-demo\"}",
  "website": "localhost",
  "userAgent": "litlyx-research-demo/1.0"
}
```

### 批量侧

```json
{
  "pid": "WORKSPACE_ID",
  "name": "demo_report_requested",
  "metadata": "{\"source\":\"bulk-send\",\"user\":\"demo-user-01\",\"session\":1,\"step\":3,\"plan\":\"starter\",\"page\":\"http://localhost:4174/index.html?user=1&session=1\"}",
  "website": "localhost",
  "userAgent": "litlyx-bulk-send/1.0"
}
```

## 4. 字段边界

| 字段 | 当前用途 | 说明 |
| --- | --- | --- |
| `pid` | 指向当前 workspace | 必须运行时传入，不写进仓库 |
| `name` | 事件名 | Product 聚合最先消费这个字段 |
| `metadata` | 事件上下文 | 当前以 JSON 字符串上报 |
| `website` | 域名标签 | 当前统一使用 `localhost` 便于在 Raw Events 中识别 |
| `userAgent` | 调试来源标签 | 用于区分 demo、bulk-send 等来源 |

metadata 当前最常用的字段：

- `source`
- `surface`
- `section`
- `variant`
- `plan`
- `currency`
- `user`
- `session`
- `step`
- `page`

## 5. 当前命名规范

从现有样本看，Litlyx 至少支持这种比较轻量的规则：

- 小写英文
- 下划线连接
- 事件名直接描述业务动作

当前已经稳定使用的命名：

- `demo_signup_click`
- `demo_checkout_started`
- `demo_report_requested`
- `demo_upgrade_viewed`

## 6. 建议继续扩展的字段

如果后续要把 Marketing、Reports、AI 这几层样本补厚，建议优先加这些字段，而不是一开始就加大量业务杂项：

| 字段 | 用途 | 为什么先加它 |
| --- | --- | --- |
| `utm_source` | 营销来源 | 直接支撑 Marketing 分析 |
| `utm_medium` | 媒介 | 区分广告、社媒、邮件等流量 |
| `utm_campaign` | 活动 | 方便和报告模板、AI 总结对接 |
| `role` | 用户角色 | 方便后续做 metadata 分析与分群 |
| `workspace_size` | 工作区规模 | 更接近 B2B analytics 场景 |
| `report_type` | 报告意图 | 有助于把事件和 Reports 模块连起来 |

## 7. 不建议进入 payload 的信息

- 邮箱
- 密码
- token
- workspace id 原文
- 订单明细全文
- 用户可直接识别的个人信息

这些信息即使 Litlyx 能接收，也不应该进入仓库里的 demo 或脚本。

## 9. 当前最实用的结论

- Litlyx 当前这轮验证不需要复杂用户画像，也能跑通“脚本接入 -> 自定义事件 -> Product / Raw Events 出数”。
- 如果要继续深挖，它下一步最值得补的不是更多事件名，而是更有区分度的 metadata 字段，尤其是 UTM、角色和报告意图。
