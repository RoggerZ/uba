# 从门槛到升级转化

## 目标

把“当前不能用”的状态，设计成用户能理解、能判断、能继续行动的产品路径，而不是让用户以为页面坏了。

## 适用场景

- 设计 premium gate
- 设计 disabled 报告卡片
- 设计试用期限制
- 设计成员协作和分享权限
- 设计升级 CTA 与套餐说明

## 涉及证据

- `P04-S05` Reports Sample 预览
- `P04-S06` SEO premium gate
- `P04-S07` Reports 正式生成 disabled
- `P05-S09` Plans FAQ 展开态
- `P05-S10` Shareable links 空态
- `P05-S11` Members loading / contact owner 态

## 推荐顺序

1. 先让用户看到能力存在
2. 再给出样张、示例或价值说明
3. 然后明确为什么当前不能继续
4. 最后给出下一步动作

## 每步判断点

1. 如果只是灰掉按钮，用户不知道是权限问题还是操作问题
2. 如果只给锁图标，没有价值说明，用户不会知道为什么要升级
3. 如果只给 loading，用户会以为产品坏了
4. 如果创建动作会改变访问权限，需要在执行前给明确确认

## Litlyx 值得借的部分

- Reports 用 `Sample` 先展示报告样张
- SEO 直接用 premium gate 和 `Upgrade` CTA
- Plans 把套餐和 FAQ 放在同一页
- Shareable links 单独建模，不和 Members 混在一起

## Litlyx 不建议照搬的部分

- Members 的 loading / contact owner 态不够清楚
- Reports disabled 态还可以更明确解释原因
- `Setup events` 看起来像站内配置，但实际打开外部文档

## SimpleTrack 可直接采用的规则

| 场景 | 推荐表达 |
| --- | --- |
| 功能未解锁 | 明确写出当前套餐不支持，并提供升级入口 |
| 报告不能生成 | 写出缺少报告类型、权限或套餐条件 |
| 分享链接创建 | 执行前说明会创建外部只读访问入口 |
| 成员协作受限 | 明确说明是 owner 权限、套餐限制还是系统异常 |
| 外链文档入口 | CTA 文案直接说明会打开文档 |

## 常见坑

- 用 loading 代替权限说明
- 用 disabled 代替解释
- 用 Upgrade CTA 代替价值说明
- 把外部分享和团队成员协作做成一个入口
- 把文档外链包装成站内配置动作

## 推荐结论

SimpleTrack 的升级转化不应该靠“拦住用户”完成，而应该靠“让用户理解为什么现在被拦住、下一步能得到什么”完成。

第一版最应该做的是：

- Reports 样张
- 清楚的 premium gate
- disabled 原因说明
- Plans FAQ
- 成员和分享权限分层
