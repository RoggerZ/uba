# 从治理到协作边界

## 目标

把“哪些数据该进来、哪些流量该过滤、哪些人能看、哪些人能改”这条边界设计清楚。

## 适用场景

- 设计治理与权限模块
- 讨论分享链接和成员邀请的区别
- 讨论高风险操作该放在哪里

## 涉及证据

- `P02-S05`
- `P05-S01` 到 `P05-S04`
- `P05-S10` `P05-S11`
- [../../Litlyx功能矩阵.md](../../Litlyx功能矩阵.md)

## 推荐顺序

1. 先在接入链路里定义可信域名和数据来源
2. 再把 `Settings / Domains` 用作按域名清理历史数据的治理入口
3. 把 `Shields` 独立成治理层，处理 allow list、IP 排除、bot 流量
4. 把 `Shareable links` 单独做成外部只读分享能力
5. 把 `Members` 明确做成内部协作能力，并给清楚门槛说明

## 每步判断点

1. 治理入口带删除语义时，不该伪装成普通设置
2. 外部只读分享和内部成员协作是两种不同权限模型
3. 受限态必须解释清楚是权限、套餐还是系统问题
4. 如果 bot、办公 IP、预览域名不被单独治理，分析噪音会越来越大

## 常见坑

- 把域名治理、IP 排除、bot 过滤混在一个杂项设置页里
- 分享链接和成员邀请混成同一种“协作”
- 只给一个 loading 态，不告诉用户为什么不可用
- 删除和 sanitization 这种高风险操作缺少明显确认

## SimpleTrack 可直接借鉴的决策

- Settings 负责工作区信息回查，治理能力尽量单独成层
- 把 `Shareable links` 和 `Members` 分开建模
- 受限态一定要比 Litlyx 当前的 `contact owner` 更清楚
- 在治理层里优先支持域名、IP、bot 三类噪音控制
