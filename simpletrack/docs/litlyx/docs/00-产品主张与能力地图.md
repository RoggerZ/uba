# Litlyx 产品主张与能力地图

> 目标：把 Litlyx 当前这套后台能力，翻译成更容易拿来做 SimpleTrack 产品讨论的结构化语言。

## 一句话理解

Litlyx 的核心思路不是“先给一套完整 analytics 报表”，而是：

1. 先让用户接入成功
2. 再让用户看懂数据
3. 然后把增长、治理、AI、商业化逐层挂进同一个工作区壳

## 当前能力地图

| 层级 | 模块 | 回答的问题 |
| --- | --- | --- |
| 账号层 | `Workspaces` | 我有哪些项目、当前是什么套餐 |
| 接入层 | `Web / Install` | 我怎么把脚本或 GTM 接进去 |
| 回查层 | `Settings / General` | 安装代码和 Workspace ID 还能在哪看到 |
| 分析层 | `Product` | 事件现在发生了什么 |
| 明细层 | `Raw Events` | 这些事件到底有没有真正入库 |
| 增长层 | `Marketing` | 流量从哪里来、下一步怎么继续投放 |
| 报告层 | `Reports` | 这些分析能不能被包装成模板和报告产物 |
| 门槛层 | `SEO` | 哪些高级能力需要升级 |
| 治理层 | `Shields`、`Settings / Domains` | 哪些流量要保留，哪些噪音要清掉 |
| AI 层 | `Analyst` | 这些数据该怎么解释、下一步该问什么 |
| 分享层 | `Shareable links` | 如何给外部人只读看数据 |
| 协作层 | `Members` | 如何让团队成员进入同一个 workspace |
| 商业化层 | `Plans` | 能力边界和升级路径是什么 |

## 当前产品主张

从现有页面和交互看，Litlyx 更强调这几件事：

- 接入体验要短
- 空态也要能教育用户
- 高级能力可以先用样张和门槛页解释价值
- 治理、分享、协作不应混成一个杂项设置页

## 对 SimpleTrack 的启发

最值得先借的不是 UI 外观，而是这条产品顺序：

1. 先解决接入
2. 再解决首批数据可见
3. 再解决增长闭环
4. 最后补协作、治理和高级商业化能力
