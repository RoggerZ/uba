# Litlyx 事件与属性存储模型源码分析

> 目标：只回答一个问题。Litlyx 的自定义事件、metadata、visit、session 到底怎么落库，它和 Umami / analytics-core 为什么是不同路数。

## 一句话结论

Litlyx 走的是更轻的文档模型：

- `events` 集合保存事件本体
- `metadata` 直接内嵌在事件文档里
- `visits` 和 `sessions` 分开承接访问与停留时长

它没有像 Umami 那样把事件属性拆成 typed KV 行，也没有像 `xwl_bi` 那样第一次见到新字段就扩 ClickHouse 列。它更像“先把事件和 metadata 原样收进 Mongo，再在读侧用聚合管道把 metadata key/value 挖出来”。

## 证据边界

- 本文基于本轮临时解压的 Litlyx 源码，工作区路径是 `.tmp/litlyx-source/`
- 本轮抓取时记录的上游 commit：`3881c67b9350ce96f630d69a0e7bf1f49ac95876`
- 最后复验：`2026-05-10`
- `.tmp/litlyx-source/` 不作为长期入库资产；本文源码证据里的 `file:` 均按 Litlyx 上游仓库内相对路径书写。若后续需要复验，请按上述 commit 重新下载或检出 Litlyx 源码，再用这些相对路径和行号核对。

## 先看最短写入链路

Litlyx README 给出的事件接口就已经暴露了它的模型倾向：

- 浏览器侧 `Lit.event("click_on_buy_item", { metadata: {...} })`
- 服务端直接 `POST /event`
- `metadata` 在 HTTP payload 里就是一个 JSON 字符串

源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `README.md:70-103`

这说明 Litlyx 一开始就没有把“事件属性”设计成一套独立 schema，而是把它当成“跟随事件一起提交的 metadata 包”。

## Producer 做了什么

`producer/src/index.ts` 的 `/event` 路由会：

1. 从请求里拿 `website`、`pid`、`userAgent`
2. 用 `website + ip + userAgent` 计算 `sessionHash`
3. 用 `pid + ip + userAgent` 计算 `flowHash`
4. 把原始 `req.body` 连同 `_type: "event"` 一起写进 Redis Stream

源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `producer/src/index.ts:27-48`; 仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `producer/src/utils.ts:10-21`

这里最值得注意的是：Producer 没有尝试解析 `metadata` 内部字段，也没有给每个 key 做类型槽位。它只负责把请求塞进流里。

## Consumer 怎么落库

Consumer 读流后按 `_type` 分发：

- `event` -> `process_event`
- `visit` -> `process_visit`
- `keep_alive` -> `process_keep_alive`

源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `consumer/src/index.ts:55-89`

### 事件

`process_event` 会把 `metadata` 当作 JSON 字符串尝试 `JSON.parse`，然后直接写入 `EventModel.create({... metadata: metadataObject ...})`。源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `consumer/src/index.ts:181-200`

`EventSchema` 也印证了这一点：`metadata` 的类型就是 `Schema.Types.Mixed`。源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `shared_global/schema/metrics/EventSchema.ts:13-23`

### Visit

`process_visit` 负责写访问维度：browser、os、device、country、utm_*、page、referrer 等。源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `consumer/src/index.ts:102-152`; 仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `shared_global/schema/metrics/VisitSchema.ts:32-64`

### Session

`process_keep_alive` 负责 `sessions`：主要是 `session`、`flowHash`、`website`、`duration`、`updated_at`。源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `consumer/src/index.ts:154-177`; 仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `shared_global/schema/metrics/SessionSchema.ts:14-24`

## 这说明 Litlyx 的存储边界是什么

| 层 | 存什么 | 不存什么 |
| --- | --- | --- |
| `events` | 事件名、session、flowHash、website、`metadata` 对象 | 不把 metadata 再拆成 typed property rows |
| `visits` | UA、geo、UTM、page、referrer 等访问维度 | 不记录单个自定义事件的 metadata |
| `sessions` | session 生命周期和停留时长 | 不承担富用户画像职责 |

如果只从本轮读过的 ingestion path 和 metrics schema 看，Litlyx 没有出现“独立的 analytics user property / profile 表”。这里要说得很克制：

- **证据**：事件 ingestion 只看到了 `events.metadata`、`visits`、`sessions`
- **推论**：Litlyx 当前分析主线更依赖事件 metadata 和 visit/session 维度，而不是独立 profile 属性模型
- **未知**：仓库其他业务域可能存在用户/订阅/工作区对象，但它们不是这条 analytics ingestion path 的“事件属性存储模型”

## metadata 在读侧怎么被查询

Litlyx 没有提前把 metadata 拆表，但它不是完全放任不管。Dashboard API 会在 Mongo 聚合里临时把 metadata 转出来：

- `event_metadata_fields.ts` 用 `$objectToArray` 把 `metadata` 变成 key/value 数组，再汇总唯一字段名
- `event_metadata_analyze.ts` 直接按 `$metadata.<field_name>` 分组计数

源码证据：仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `dashboard/server/api/data/event_metadata_fields.ts:15-36`; 仓库: `litlyx`, commit: `3881c67b9350ce96f630d69a0e7bf1f49ac95876`, file: `dashboard/server/api/data/event_metadata_analyze.ts:16-35`

这就是 Litlyx 这套模型的关键取舍：

- 写入很轻
- 模型很灵活
- 但属性治理和高性能过滤被推迟到了读侧

## 同一个例子在 Litlyx 里会变成什么

假设上报的是：

```json
{
  "pid": "workspace_id",
  "name": "signup_clicked",
  "metadata": "{\"button\":\"hero\",\"plan\":\"pro\"}",
  "website": "demo.example",
  "userAgent": "browser"
}
```

它在 Litlyx 里最接近下面这种结果：

| 集合 | 结果 |
| --- | --- |
| `events` | 一条文档：`{ name: "signup_clicked", metadata: { button: "hero", plan: "pro" }, session, flowHash, website, created_at }` |
| `visits` | 一条或多条 visit 文档，承接浏览器、国家、UTM、page 等访问上下文 |
| `sessions` | 一条 session 文档，主要累计 `duration` 和 `updated_at` |

不会额外生成：

- `event_properties` 这样的独立属性表
- `user_properties` 这样的 typed profile 表
- 第一次见字段就自动扩列的 DDL

## 这种模型的优点

### 1. 写入心智负担很低

Producer 几乎不做 schema 工作，Consumer 只需要把 `metadata` parse 成对象再写 Mongo 文档。

### 2. 原型、demo、早期产品很容易接

对“先把 Product 页面跑起来、先让 Raw Events 有东西看”非常友好，因为不需要提前决定太多属性结构。

### 3. 访问维度和事件 metadata 被自然分层

`visits` 负责流量与访问上下文，`events.metadata` 负责动作细节，这个分工对于轻量产品分析很直接。

## 代价和限制

| 代价 | 说明 |
| --- | --- |
| metadata 治理滞后 | 同义字段、乱命名、大小写漂移不会在写入时被拦住 |
| 复杂过滤更依赖聚合管道 | 读侧要临时做 `$objectToArray`、`$metadata.field` 等处理 |
| 缺少 typed row 的稳定中间层 | 后续如果要做强约束的 Breakdown / Segment / property dictionary，会比 Umami / analytics-core 更晚补那一层 |

## 给 SimpleTrack / analytics-core 的直接启发

1. Litlyx 证明了“先把 metadata 直接收进文档”可以非常快地验证产品闭环。
2. 但如果目标是长期做稳定的 Realtime、Events、property filters、event dictionary，不能只停在 `metadata: Mixed` 这一层。
3. Litlyx 适合借鉴的更像是“接入门槛低、首屏快见数据”的产品策略，不是最终的数据面结构。
