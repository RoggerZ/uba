# Litlyx 治理、分享、协作与套餐

## 当前结论

Litlyx 把“数据是否干净”“谁能看”“谁能进 workspace”“哪些能力需要升级”拆成了几层，而不是塞进同一个设置页。

## 治理层

治理现在主要分成两部分：

### `Settings / Domains`

- `Domain data`
- `Sanitize Domains`

它负责处理已经进来的域名数据，带明显删除和清理语义。

### `Shields`

- `Domains`
- `IP addresses`
- `Bot traffic`

它更像前置治理层，负责定义哪些流量应该被允许或过滤。

## 分享与协作

### `Shareable links`

它承担的是外部只读访问能力。当前已看到：

- `0 active` 空态
- public / protected 选项
- 创建入口

但这轮没有点击 `Create link`，因为那会改变云端访问状态。

### `Members`

它显然是内部协作入口，但当前表现并不好：

- 侧栏灰态
- 直接访问 `/members` 后停在 loading / contact owner

这是一种“存在能力，但受限态表达不足”的典型案例。

## 套餐与门槛

`Plans` 和 `SEO` 一起说明 Litlyx 的商业化表达比较前置：

- `SEO` 直接用 premium gate 表达
- `Plans` 把 Personal / Business 与 FAQ 放在同一页
- `Reports` 也通过 disabled 态表达能力边界

## 对 SimpleTrack 的启发

- 治理层应该独立建模，不要埋进杂项设置
- 外部只读分享和内部成员协作必须分开
- 受限态必须比单纯 loading 更清楚
- 门槛页和套餐页都要解释“为什么你现在不能用”，而不是只给一个锁
