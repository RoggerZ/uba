# Litlyx 真实业务数据方案

## 目标

当前 Litlyx 已经验证了最小事件链路，但 Marketing、Reports、AI 的真实业务样本还不够厚。本方案用于规划下一阶段应该补什么样的数据，才能让这些页面进入更可解释的状态。

## 当前样本现状

| 项目 | 当前状态 |
| --- | --- |
| 浏览器 demo 自定义事件 | 已验证 2 个 |
| 批量脚本事件类型 | 已验证 4 个 |
| 总事件量 | 14 条 |
| Product | 已进入真实有数状态 |
| Raw Events | 已出现明细表 |
| Marketing 真实来源数据 | 还没有专门构造 |
| Reports 正式生成 | 当前账号权限受限 |
| AI 真实分析输出 | 还没有实测 |

## 下一阶段最值得补的样本

### 1. 营销来源样本

目标：

- 让 `Marketing` 不只是示例态，而是出现可解释的真实来源结构。

建议字段：

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_content`

建议来源：

- `google / cpc / brand_search`
- `linkedin / social / founder_post`
- `newsletter / email / onboarding_nudge`
- `docs / referral / custom_events_doc`

### 2. 报告意图样本

目标：

- 让 Reports、Product 和 AI 之间的关联更清楚。

建议事件：

- `report_requested`
- `report_sample_opened`
- `report_download_intent`

建议字段：

- `report_type`
- `surface`
- `role`

### 3. 升级与商业化样本

目标：

- 让 Plans、Upgrade CTA、Product 使用深度形成更像真实 SaaS 的链路。

建议事件：

- `upgrade_viewed`
- `upgrade_started`
- `upgrade_completed`

建议字段：

- `plan`
- `billing_cycle`
- `surface`

### 4. 治理与协作样本

目标：

- 让治理层和协作入口不只是静态页面，而能出现更接近真实操作意图的行为痕迹。

建议事件：

- `domain_review_opened`
- `shield_rule_started`
- `share_link_intent`
- `member_invite_intent`

注意：

- 这些事件只建议记录“意图”和“打开动作”，不要自动触发真实创建或删除。

## 推荐的样本组合

### 最小增强版

- 给 demo 页面增加带 UTM 的访问入口
- 新增 3 到 5 个营销来源事件
- 新增 2 个报告意图事件

### 增长闭环版

- 增加升级链路事件
- 增加报告类型字段
- 增加角色与工作区规模字段

### 高风险动作版

- 只有在用户明确授权后，才考虑实测：
  - `Create link`
  - Domain data 删除
  - Domain sanitization
  - Members 邀请

## 不建议现在做的事

- 为了造样本而强行绕过 Reports 的 disabled 态
- 直接创建真实分享链接
- 直接删除或清理真实域名数据
- 在 demo 里写入真实 workspace id、账号或 token

## 建议的推进顺序

1. 先补 Marketing 的真实 UTM 来源样本
2. 再补 Reports 与 AI 的意图型事件
3. 再看账号权限是否允许继续验证正式生成链路
4. 最后才考虑任何会改变云端状态的创建 / 删除动作

## 这份方案的定位

- 它是下一阶段补样本的设计输入，不代表这些数据已经在 Litlyx 当前工作区里跑通。
- 目前已跑通的真实事实仍以 [数据模型与事件字典.md](./数据模型与事件字典.md)、[功能打通矩阵.md](./功能打通矩阵.md) 和 [../快照进度.md](../快照进度.md) 为准。
