# KafkaBus 生产运行手册与故障演练

> 最近更新：2026-05-17
> 适用范围：`src/analytics-core` Kafka EventBus provider 与 `src/analytics-service` Kafka runtime 装配。
> 当前提交状态：`analytics-core` commit `2e52263`；`analytics-service` commit `0f695db`；父仓同步 commit `pending commit`。

## 1. 定位

Kafka 是 SimpleTrack / analytics-core 的生产主 EventBus。Redis Stream 只作为本地、小流量、测试实现，不应被写成 Kafka 的无损生产回滚方案。

当前第一版能力边界：

- producer 默认等待 broker `acks=all`，并保留 producer retry 默认值。
- consumer 使用 consumer group 消费主 topic。
- handler 可以并发执行，但同一 topic-partition 的 offset 只能按连续完成区间推进。
- handler 失败会按 `MaxAttempts` 重试；超过后写 DLQ。
- DLQ 写成功后，原消息才允许标记完成；DLQ 写失败时原 offset 不提交。
- provider 暴露 `Stats()` 诊断快照，service runtime 暴露 `KafkaEventBusStats()`；`analytics-service` 还提供默认关闭的内部 `GET /v1/kafka/diagnostics` route，供受控运维入口读取同一进程级快照。
- SASL/TLS 第一版支持 TLS 与 SASL/PLAIN；SCRAM、OAuth、Kerberos 暂不宣称支持。

生产安全提醒：生产部署模板必须默认启用 TLS。`SASL/PLAIN` 只有在 TLS 已启用时才推荐用于生产；`SASL/PLAIN` 明文只允许用于临时本地或测试 broker，不应复制到生产配置、Helm values、systemd unit 或 CI/CD secret 模板中。

## 2. 生产环境变量

最小生产 Kafka 配置：

```bash
export ANALYTICS_SERVICE_EVENTBUS=kafka
export ANALYTICS_SERVICE_KAFKA_BROKERS=broker-1:9093,broker-2:9093,broker-3:9093
export ANALYTICS_SERVICE_KAFKA_TOPIC=analytics.events
export ANALYTICS_SERVICE_KAFKA_DEAD_LETTER_TOPIC=analytics.events.dead
export ANALYTICS_SERVICE_KAFKA_CLIENT_ID=simpletrack-anaysitics-service
export ANALYTICS_SERVICE_KAFKA_MAX_ATTEMPTS=5
export ANALYTICS_SERVICE_KAFKA_RETRY_BACKOFF=250ms
export ANALYTICS_SERVICE_KAFKA_WORKERS=100
export ANALYTICS_SERVICE_KAFKA_QUEUE_SIZE=200
export ANALYTICS_SERVICE_KAFKA_COMMIT_INTERVAL=1s
```

启用 TLS：

```bash
export ANALYTICS_SERVICE_KAFKA_TLS_ENABLED=true
export ANALYTICS_SERVICE_KAFKA_TLS_SERVER_NAME=kafka.example.internal
export ANALYTICS_SERVICE_KAFKA_TLS_CA_FILE=/etc/simpletrack/kafka/ca.pem
```

启用 mTLS 时再加：

```bash
export ANALYTICS_SERVICE_KAFKA_TLS_CERT_FILE=/etc/simpletrack/kafka/client.pem
export ANALYTICS_SERVICE_KAFKA_TLS_KEY_FILE=/etc/simpletrack/kafka/client-key.pem
```

启用 SASL/PLAIN：

```bash
export ANALYTICS_SERVICE_KAFKA_SASL_ENABLED=true
export ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM=plain
export ANALYTICS_SERVICE_KAFKA_SASL_USERNAME=simpletrack-eventbus
export ANALYTICS_SERVICE_KAFKA_SASL_PASSWORD='<从部署密钥系统注入，不写入仓库>'
export ANALYTICS_SERVICE_KAFKA_SASL_HANDSHAKE=true
```

`ANALYTICS_SERVICE_KAFKA_TLS_INSECURE_SKIP_VERIFY=true` 只允许用于临时测试 broker，不允许进入生产部署模板。

启用内部 Kafka 诊断 route：

```bash
export ANALYTICS_SERVICE_KAFKA_DIAGNOSTICS_ENABLED=true
export ANALYTICS_SERVICE_KAFKA_DIAGNOSTICS_PATH=/v1/kafka/diagnostics
export ANALYTICS_SERVICE_QUERY_TOKEN='<从部署密钥系统注入，不写入仓库>'
export ANALYTICS_SERVICE_QUERY_TOKEN_SCOPES=kafka_diagnostics
```

也可以通过结构化 `ANALYTICS_SERVICE_QUERY_TOKENS_JSON` 给诊断专用 token 配置 `kafka_diagnostics` scope。诊断 token 不需要、也不应该绑定公开 `write_key`。

## 3. Topic 与分区建议

建议生产 topic：

```bash
kafka-topics --bootstrap-server broker-1:9093 \
  --create --if-not-exists \
  --topic analytics.events \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2

kafka-topics --bootstrap-server broker-1:9093 \
  --create --if-not-exists \
  --topic analytics.events.dead \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2
```

分区 key 当前由 `TenantID:ProjectID:SourceID` 优先组成；如果这些字段为空，再退到事件 id 或事件名。生产排查时应按 tenant/project/source 维度观察热点 partition。

## 4. ACL 矩阵

| 主体 | Resource | 必需权限 | 说明 |
| --- | --- | --- | --- |
| collect producer / service | topic `analytics.events` | `Write`, `Describe` | `/collect` 成功依赖主 topic 写入成功 |
| ingestion consumer group | topic `analytics.events` | `Read`, `Describe` | consumer group 拉取主 topic |
| ingestion consumer group | group `simpletrack-anaysitics-service` 或实际 group | `Read` | 提交 offset 与恢复消费位置 |
| DLQ producer / service | topic `analytics.events.dead` | `Write`, `Describe` | 失败消息进入 DLQ 后才提交原 offset |
| operator drill account | topic `analytics.events.dead` | `Read`, `Describe` | 只用于排查和补偿，不用于常规业务消费 |
| operator drill account | group 临时 replay group | `Read` | DLQ replay 必须使用独立临时 group |

如果启用 idempotent producer，broker 还需要为 producer 主体配置 `IdempotentWrite`。当前生产默认没有自动启用 idempotent producer。

## 5. SASL/TLS 配置矩阵

| 场景 | 当前支持 | 配置 | 备注 |
| --- | --- | --- | --- |
| 明文本地 Kafka | 支持 | 不启用 TLS/SASL | 仅用于本地集成测试 |
| TLS server auth | 支持 | `ANALYTICS_SERVICE_KAFKA_TLS_ENABLED=true` + `ANALYTICS_SERVICE_KAFKA_TLS_CA_FILE` / `ANALYTICS_SERVICE_KAFKA_TLS_SERVER_NAME` | 推荐生产最低要求 |
| mTLS | 支持 | `ANALYTICS_SERVICE_KAFKA_TLS_CERT_FILE` + `ANALYTICS_SERVICE_KAFKA_TLS_KEY_FILE` | cert/key 必须成对配置 |
| SASL/PLAIN + TLS | 支持 | `ANALYTICS_SERVICE_KAFKA_SASL_ENABLED=true` + `ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM=plain` + TLS | 推荐托管 Kafka 常见模式 |
| SASL/PLAIN 明文 | 技术上支持 | `ANALYTICS_SERVICE_KAFKA_SASL_ENABLED=true` 且 TLS disabled | 不推荐生产使用 |
| SCRAM-SHA-256/512 | 暂不支持 | `ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM=scram-sha-256/512` 当前启动会拒绝 | 需要补 SCRAM client generator 后再打开 |
| OAUTHBEARER | 暂不支持 | `ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM=oauthbearer` 当前启动会拒绝 | 需要 token provider 生命周期设计 |
| Kerberos/GSSAPI | 暂不支持 | `ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM=gssapi` 当前启动会拒绝 | 需要单独部署和密钥表规范 |

## 6. 观测入口

`analytics-core/eventbus/kafka.Bus.Stats()`、`analytics-service/internal/runtime.Runtime.KafkaEventBusStats()` 和默认关闭的内部 `GET /v1/kafka/diagnostics` route 当前提供同一类诊断快照，字段包括：

- `Metrics.ConsumedTotal`
- `Metrics.HandlerSuccessTotal`
- `Metrics.HandlerFailureTotal`
- `Metrics.HandlerRetryTotal`
- `Metrics.MalformedTotal`
- `Metrics.DeadLetterSuccessTotal`
- `Metrics.DeadLetterFailureTotal`
- `Metrics.PausedPartitions`
- `Metrics.PauseTransitionsTotal`
- `Metrics.ResumeTransitionsTotal`
- `Commits[].Topic / Partition / NextOffset / HighWaterMarkOffset / Lag`
- `Commits[].PendingCount / DoneCount / LargestPendingGap`
- `WorkerPool.Queued / QueueUsageRatio / RejectedTotal`
- `CompletionGate.InFlightMessages / WaitingTasks`

这些字段不包含原始事件 payload、broker secret、TLS/SASL material。`/v1/kafka/diagnostics` 是内部运维 route，默认关闭，只能在受控网络或内部网关后通过 `kafka_diagnostics` query token scope 访问；不要把它作为公网 HTTP 诊断入口暴露。

本 route 的 `lag_estimate` 仍是进程内估算，不是 broker 侧权威 consumer lag。生产 SLO、告警和容量判断仍应接 Kafka broker/exporter 指标；该 route 主要用于把本进程的 DLQ/retry/pause/worker/ordered-commit 状态和 broker 指标放在一起排障。

示例读取：

```bash
curl -H "Authorization: Bearer ${ANALYTICS_SERVICE_QUERY_TOKEN}" \
  http://127.0.0.1:8080/v1/kafka/diagnostics
```

## 7. DLQ 排查与补偿

DLQ topic 默认是 `analytics.events.dead`。DLQ 记录包含原始 envelope、原 topic/partition/offset、consumer group、consumer id、attempt、错误信息和 failed_at。

排查步骤：

1. 确认主消费 lag 是否仍在增长。
2. 查看 `DeadLetterSuccessTotal` 是否增长。
3. 使用临时 operator group 读取 DLQ，不要复用生产 consumer group。
4. 按错误类型分类：decode error、handler error、存储错误、配置错误。
5. 对可补偿消息生成新的补偿任务或重新写回主 topic；不要直接修改生产 consumer group offset 来“跳过”原因未明的消息。

示例读取：

```bash
kafka-console-consumer \
  --bootstrap-server broker-1:9093 \
  --topic analytics.events.dead \
  --group simpletrack-dlq-inspection-$(date +%Y%m%d%H%M%S) \
  --from-beginning \
  --max-messages 10
```

## 8. Consumer Group Reset

只在确认补偿策略后执行 reset。先 dry-run：

```bash
kafka-consumer-groups --bootstrap-server broker-1:9093 \
  --group simpletrack-anaysitics-service \
  --topic analytics.events \
  --reset-offsets --to-offset 12345 --dry-run
```

确认后执行：

```bash
kafka-consumer-groups --bootstrap-server broker-1:9093 \
  --group simpletrack-anaysitics-service \
  --topic analytics.events \
  --reset-offsets --to-offset 12345 --execute
```

不要在 DLQ 写失败或 ClickHouse/MySQL 持续失败时 reset 到更靠后的 offset；这会把仍未落库也未进入 DLQ 的消息跳过。

## 9. 故障演练

### 9.1 Broker outage

目标：确认 `/collect` 不会在 Kafka 不可写时返回假成功。

步骤：

1. 停止测试 broker 或阻断 service 到 broker 的网络。
2. 发送 `/collect` 请求。
3. 预期：`Publish` 返回错误，HTTP collect 不应被当作已持久化成功。
4. 恢复 broker 后重新发送请求，确认主 topic 收到新事件。

### 9.2 DLQ write failure

目标：确认毒消息不会在 DLQ 写失败时提交原 offset。

步骤：

1. 让 handler 对测试消息持续失败。
2. 临时撤销 service 对 DLQ topic 的 `Write` 权限，或把 DLQ topic 指向不可写 topic。
3. 等待超过 `MaxAttempts`。
4. 预期：`DeadLetterFailureTotal` 增长，原 partition offset 不推进。
5. 恢复 DLQ 写权限后，预期 DLQ 写成功并允许 ordered commit 推进连续区间。

### 9.3 Rebalance / claim close

目标：确认 rebalance 后旧 session 的 late completion 不会提交 stale offset。

步骤：

1. 启动同 group consumer A，并让一个消息的 handler sleep。
2. 启动同 group consumer B 触发 rebalance。
3. 观察旧 claim 关闭后，未完成消息不应由旧 session 标记提交。
4. 新 generation 重新消费未提交消息，完成后再提交。

### 9.4 Restart replay

目标：确认进程退出时未完成 offset 可重放。

步骤：

1. 发送一条测试事件。
2. 让 handler 收到消息后阻塞。
3. 停止 consumer 进程。
4. 启动同 group 新 consumer。
5. 预期：同一 offset 重新投递；完成后 group offset 再推进。

## 10. 回滚说明

Redis Stream 不能作为 Kafka 生产链路的无损热切回滚。Kafka 与 Redis Stream 的 consumer group、offset、DLQ topic/stream、重放命令和 ACL 模型都不同。

如果生产 Kafka 需要回滚：

1. 先暂停 `/collect` 写入或切只读维护窗口。
2. 确认 Kafka 主 topic 与 DLQ 中尚未处理的消息范围。
3. 明确是否需要把 Kafka 剩余消息补偿到 Redis Stream 或重新采集。
4. 单独启动 Redis Stream 模式，并标注为降级容量。
5. 恢复 Kafka 前再做反向补偿计划。
