# Kafka-First EventBus 执行规范

> 最近更新：2026-05-17

## 执行节奏

1. 父代理负责思考、拆分、整合、验证和最终判断。
2. 子代理只在需要并行执行时使用，例如代码切片、测试修复、文档汇总、阶段性交付评审。
3. 子代理必须按 goal 模式执行：prompt 第一条要求它先创建本子任务 goal，完成时标记 goal complete；如果未完成，必须说明 blocker 和剩余工作。
4. 子代理执行时保持轻度思考和直接执行，不扩大范围，不自行重写总计划。
5. 父代理必须整理子代理输出，关闭不再需要的子代理，并亲自核实落盘结果。
6. 全程保持 goal 模式；阶段没有真实完成时不标记完成。

## 文档频率

代码推进优先。本任务执行期间只维护本目录的轻量文档：

- 阶段开始或范围变化：更新 `分阶段实施计划.md`。
- 阶段状态变化：更新 `进度表.md`。
- 遇到真实 blocker、环境坑、设计坑：更新 `踩坑记录.md`。
- 任务全部执行完成：把精简结论同步到根目录 / AGENTS.md 指定的正式实施决策文档。

## 踩坑处理

1. 遇到环境、网络、依赖、Git、Docker、测试或工具问题时，先查根目录已有记录，例如 `docs/开发环境卡壳问题记录.md`。
2. 如果已有记录能解决问题，只在本任务 `踩坑记录.md` 中补一条任务范围内的简短引用，不复制大段原文。
3. 如果是同一类问题，更新原有任务记录的状态、证据或第一处理动作，不新增重复编号。
4. 如果是新问题，才在本任务 `踩坑记录.md` 中新增编号、根因、处理动作和后续防线。
5. 本任务执行期间的新踩坑记录只写到 `simpletrack/docs/任务计划/kafka-first-eventbus/踩坑记录.md`。

## Git 频率

1. 不再修改一点就提交。
2. 阶段完成、改动较多、或者进入高风险边界前后，再做本地 commit。
3. push 操作低频执行，优先放在阶段交付、阶段评审后或用户明确要求时。
4. push 如果出现网络问题，使用本机代理 `http://localhost:64320` 多试几次，再记录到踩坑表。

## 评审频率

阶段性交付、准备提交或核心边界变更收口时再触发评审：

- `code-review`：公共契约、Kafka provider、service runtime、DLQ/commit 语义等高风险阶段完成后使用。
- `code-simplifier`：阶段实现可用但出现重复、命名混乱或不必要抽象时使用。
- `ai-slop-cleaner`：阶段提交前按 changed-files 范围做一次反样板化清理，不扩大到无关文件。

## 当前暂停规则

Kafka provider 实现暂停期间：

1. 不继续增加 Kafka 代码。
2. 不改正式实施决策大文档。
3. 只建立和验证本任务 plan 目录。
4. 恢复代码实现时，从阶段 1 开始，并先更新进度表。
