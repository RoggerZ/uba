# Kafka 生产级认证与压测计划

> 状态：进行中
> 最近更新：2026-05-17
> 当前基线：父仓 commit `c31750e` 已推送；`analytics-core` commit `d835341`；`analytics-service` commit `c6d1139`。

## 目标

把已经跑通的本机单 broker `SASL_SSL` drill，推进成更接近生产的 Kafka 验收体系。这个计划不重做 Kafka-first EventBus v1，也不立即扩全部认证机制；先证明生产形态认证、指标源和压测口径足够清楚，再决定是否进入新代码实现。

## 范围边界

- 本轮优先覆盖 `SASL/PLAIN + TLS` 的生产形态多 broker 演练。
- 本轮优先建立认证 broker benchmark 和 Kafka broker/exporter 指标源口径。
- `SCRAM-SHA-256/512`、`OAuthBearer`、`Kerberos/GSSAPI` 先进入评审路线，不直接和本轮多 broker drill 混做。
- `analytics-service` 当前 `/v1/kafka/diagnostics` 与 `/v1/kafka/metrics` 仍是进程内诊断信号，不替代 broker 侧 exporter 指标。
- 不提交证书、私钥、JAAS、密码、token、docker volume 数据或压测原始大日志。

## 阶段 10：生产级多 broker SASL_SSL drill

状态：待完成

目标：把 `local-auth-kafka` 从单 broker disposable drill 扩成可选的 3 broker `SASL_SSL` 演练环境，验证 RF=3 / minISR=2 topic 在认证链路下仍能 publish、consume、ordered commit、restart replay 和 broker outage。

交付物：

- `local-auth-kafka` 新增 3 broker compose 或独立 `local-auth-kafka-prod` 目录。
- 每个 broker 使用 TLS server certificate，客户端用 CA PEM 校验。
- broker 间通信使用 `SASL_SSL` 或明确说明内部 listener 的安全取舍。
- 生成 topic 使用 3 partitions、replication factor 3、`min.insync.replicas=2`。
- 复用 `analytics-core` gated tests：
  - `TestKafkaIntegrationPublishConsume`
  - `TestKafkaIntegrationOrderedCommitWaitsForEarlierOffset`
  - `TestKafkaIntegrationUncommittedMessageReplaysAfterRestart`
  - `TestKafkaReplicatedIntegrationPublishConsume`
  - `TestKafkaReplicatedIntegrationSurvivesBrokerOutage`

验收标准：

- 记录 broker 地址、认证模式、topic detail、Kafka image/version。
- 五条 integration test 在认证多 broker 环境下通过。
- outage drill 只停止 disposable broker container，且演练后容器/network 清理完成。
- 失败时记录到 `踩坑记录.md`，不能把 skipped test 当通过。

## 阶段 11：认证 broker benchmark

状态：待完成

目标：在 `SASL_SSL` broker 下跑真实 broker publish benchmark，给后续生产容量评估一个可复测基线。

交付物：

- 复用或扩展 `BenchmarkKafkaIntegrationPublish`。
- 记录 benchmark 参数：broker 数、partition 数、replication factor、minISR、message size、acks、retry、flush、TLS/SASL 模式、Go 版本、Kafka 版本、运行机器。
- 对比已有 plaintext 单 broker benchmark，标注认证开销只作为本机相对观察，不写成生产 SLA。

验收标准：

- `ANALYTICS_CORE_KAFKA_BENCHMARK=1 go test ./eventbus/kafka -bench BenchmarkKafkaIntegrationPublish -benchmem -run '^$' -count=3` 在认证 broker 下完成。
- 结果写入 `生产运行手册与故障演练.md` 或单独 benchmark 记录小节。
- 如果结果波动大，记录原因和复测建议，不硬写阈值。

## 阶段 12：Kafka exporter / SLO 指标源

状态：待完成

目标：把“进程内诊断 metrics”和“broker/exporter 权威指标”分层，避免把 `lag_estimate` 当生产 SLO 事实源。

交付物：

- 选定 broker/exporter 观测入口：Kafka broker JMX、Prometheus JMX exporter、Burrow、kafka-exporter 或托管 Kafka 自带 metrics。
- 写清楚最小 SLO 候选指标：
  - consumer group lag
  - under replicated partitions
  - offline partitions
  - minISR violation / produce errors
  - request latency / error rate
  - DLQ success/failure rate
  - retry rate
- 明确哪些指标来自 broker/exporter，哪些来自 `analytics-service` `/v1/kafka/metrics`。

验收标准：

- 文档中每个 SLO 指标都有事实源归属。
- 不把进程内 `simpletrack_kafka_ordered_commit_lag_estimate` 写成 broker 权威 lag。
- 如需代码改动，另开子任务，不在本阶段文档计划里偷偷扩实现。

## 阶段 13：SCRAM / OAuth / Kerberos 认证路线评审

状态：待评审

目标：决定是否以及何时支持 `SCRAM-SHA-256/512`、`OAuthBearer`、`Kerberos/GSSAPI`，避免为了“矩阵完整”过早引入高维护成本。

评审问题：

- 目标部署环境是否真的需要 SCRAM，而不是 `SASL/PLAIN + TLS`。
- 如果使用托管 Kafka，推荐认证方式和 secret rotation 方式是什么。
- Sarama 对目标机制的支持成熟度、配置复杂度和测试环境成本如何。
- `analytics-service` env 是否继续保持简单，还是引入 credential provider / token provider。
- 是否需要先支持 SCRAM，再考虑 OAuth/Kerberos。

验收标准：

- 至少输出一个“先做 / 暂缓 / 否决”的机制矩阵。
- 每个机制写明触发条件、实现范围、测试环境和回滚策略。
- 未评审完成前，生产 runbook 继续只声明 `SASL/PLAIN + TLS` 为已支持路径。

## 推荐执行顺序

1. 阶段 10：先做 3 broker `SASL_SSL` drill，因为它直接补齐生产拓扑认证证据。
2. 阶段 11：在同一认证环境上跑 benchmark，复用已启动集群，减少环境切换成本。
3. 阶段 12：把 broker/exporter 指标源和进程内 metrics 分层，给 SLO 和告警留正确入口。
4. 阶段 13：基于真实部署需求再评审 SCRAM/OAuth/Kerberos，不作为当前阻塞项。

## 当前风险

- 多 broker `SASL_SSL` compose 比单 broker 更容易踩 listener / advertised listener / controller quorum / cert SAN 坑。
- Windows + Docker Desktop 下端口、网络和镜像源仍可能不稳定。
- 认证 benchmark 受本机资源和 Docker 网络影响大，只能作为相对基线。
- 如果直接把 SCRAM/OAuth/Kerberos 和多 broker drill 混做，范围会失控。
