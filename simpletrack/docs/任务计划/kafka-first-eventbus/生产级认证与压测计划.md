## 阶段 12：Kafka exporter / SLO 指标源

状态：已完成

目标：把“进程内诊断 metrics”和“broker/exporter 权威指标”分层，避免把 `lag_estimate` 当生产 SLO 事实源。

交付物：

- 选定 broker/exporter 观测入口：Kafka broker JMX、Prometheus JMX exporter、Burrow、kafka-exporter 或托管 Kafka 自带 metrics。
- 写清楚最小 SLO 候选指标：
  - consumer group lag
  - under replicated partitions
  - offline partitions
  - minISR violation / produce errors
  - request latency / error rate
  - DLQ success/failure rate
  - retry rate
- 明确哪些指标来自 broker/exporter，哪些来自 `analytics-service` `/v1/kafka/metrics`。

验收标准：

- 文档中每个 SLO 指标都有事实源归属。
- 不把进程内 `simpletrack_kafka_ordered_commit_lag_estimate` 写成 broker 权威 lag。
- 如需代码改动，另开子任务，不在本阶段文档计划里偷偷扩实现。

完成证据：

- 结论版本：父仓文档同步为 `848494a`；代码基线不变，仍是 `analytics-core` commit `d835341` 与 `analytics-service` commit `c6d1139`。
- 研究边界：参考 Apache Kafka Monitoring、Confluent consumer lag 文档、Prometheus JMX Exporter、Burrow、kafka-exporter 和托管 Kafka metrics 口径；本阶段只落指标源分层，不新增 `analytics-service` 代码。
- 核心判断：Kafka broker / controller / topic 健康以 broker JMX、JMX exporter 或托管 Kafka metrics 为事实源；consumer group lag 以 Burrow、kafka-exporter 或托管 Kafka consumer-lag 指标为事实源；`analytics-service` 进程内 diagnostics / metrics 只表达本进程 Kafka provider 状态。
- `simpletrack_kafka_ordered_commit_lag_estimate` 只代表当前进程按 observed high-water mark 与 ordered commit `next_offset` 估算的本地 backlog，不能覆盖 broker 侧 consumer group lag、复制退化、offline partition、minISR 或 request latency。

指标事实源表：

| 指标 / 现象 | 主事实源 | 可用出口 | `analytics-service` 本地指标用途 | 是否需要本阶段代码改动 |
| --- | --- | --- | --- | --- |
| Consumer group lag / 消费堆积 | Burrow、kafka-exporter、托管 Kafka consumer-lag 指标，或直接按 broker end offset 与 committed offset 计算 | Prometheus scrape、托管 Kafka 控制台 / API、Burrow status API | `simpletrack_kafka_ordered_commit_lag_estimate` 仅辅助定位本进程是否有 ordered commit backlog | 否 |
| Under replicated partitions | Kafka broker JMX / controller metrics，托管 Kafka metrics | JMX exporter、kafka-exporter、托管 Kafka metrics | 无；服务进程看不到完整 broker replication 状态 | 否 |
| Offline partitions | Kafka broker / controller JMX，托管 Kafka metrics | JMX exporter、kafka-exporter、托管 Kafka metrics | 无；服务进程不能替代 controller 视角 | 否 |
| minISR violation / not enough replicas | Broker topic 配置、broker request / produce error metrics、托管 Kafka metrics | JMX exporter、broker logs、托管 Kafka metrics | 后续如新增 producer error counter，只能说明 service publish 失败，不等同 broker 全局 minISR 状态 | 否 |
| Broker request latency / request error rate | Broker JMX request metrics、托管 Kafka metrics | JMX exporter、托管 Kafka metrics | 当前 `/v1/kafka/metrics` 不输出 broker request latency；只输出 provider 本地 counters / gauges | 否 |
| DLQ success/failure | `analytics-service` 进程内 Kafka provider metrics | `/v1/kafka/metrics`、`/v1/kafka/diagnostics` | 主事实源；用于判断毒消息隔离、DLQ 写失败是否阻塞原 offset 提交 | 已有，不新增 |
| Handler retry rate / handler failure rate | `analytics-service` 进程内 Kafka provider metrics | `/v1/kafka/metrics`、`/v1/kafka/diagnostics` | 主事实源；用于判断 handler、ClickHouse 或下游写入是否异常 | 已有，不新增 |
| Pause / resume / worker backpressure | `analytics-service` 进程内 Kafka provider metrics | `/v1/kafka/metrics`、`/v1/kafka/diagnostics` | 主事实源；用于判断本进程 worker pool、completion gate、ordered commit 是否形成背压 | 已有，不新增 |

后续实现边界：

- 如果要生产告警，先接 broker/exporter 或托管 Kafka 指标，再把 `analytics-service` 本地 metrics 作为附加维度拼图；不要反过来用服务本地 metrics 替代 broker 侧 SLO。
- 如果要在服务内新增 producer error、publish latency 或 Sarama client metrics，应另开代码任务，并在指标名里明确 `client` / `process` 语义。
- 如果要落 Prometheus alert rules 或 Grafana dashboard，另开运维配置任务；本阶段只确定事实源，不写阈值。

## 阶段 13：SCRAM / OAuth / Kerberos 认证路线评审

状态：已完成

目标：决定是否以及何时支持 `SCRAM-SHA-256/512`、`OAuthBearer`、`Kerberos/GSSAPI`，避免为了“矩阵完整”过早引入高维护成本。

评审问题：

- 目标部署环境是否真的需要 SCRAM，而不是 `SASL/PLAIN + TLS`。
- 如果使用托管 Kafka，推荐认证方式和 secret rotation 方式是什么。
- Sarama 对目标机制的支持成熟度、配置复杂度和测试环境成本如何。
- `analytics-service` env 是否继续保持简单，还是引入 credential provider / token provider。
- 是否需要先支持 SCRAM，再考虑 OAuth/Kerberos。

验收标准：

- 至少输出一个“先做 / 暂缓 / 否决”的机制矩阵。
- 每个机制写明触发条件、实现范围、测试环境和回滚策略。
- 未评审完成前，生产 runbook 继续只声明 `SASL/PLAIN + TLS` 为已支持路径。

评审证据：

- 当前实现边界：`analytics-core/eventbus/kafka.Options` 只公开 `SASLMechanismPlain`，`normalize()` 对非 `plain` 机制 fail fast；`newSaramaConfig()` 只把 provider option 映射为 Sarama `SASLTypePlaintext`。
- 当前 service 边界：`analytics-service/internal/config` 已暴露 TLS 与 SASL env，但 `ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM` 只接受 `plain` / `plaintext` / `sasl_plaintext`，`scram-sha-256` 等值会启动失败。
- Sarama v1.47.0 本地 module cache 证据：`config.go` 支持 `SASLTypeSCRAMSHA256` / `SASLTypeSCRAMSHA512` 并要求 `SCRAMClientGeneratorFunc`；支持 `SASLTypeOAuth` 并要求 `AccessTokenProvider`；支持 `SASLTypeGSSAPI` 并要求 `GSSAPIConfig`。
- 依赖评审子代理 `Arendt` 已按 goal-mode 完成外部评审，结论是：SCRAM 成熟度最高、优先做；OAuth 能做但需要 token/provider 生命周期，暂缓；Kerberos/GSSAPI 配置和测试成本最高，当前否决，除非企业 Kerberos 体系锁定。
- Context7 / Sarama 文档查询本轮返回 fetch failed，已改用本地 module cache 和官方链接证据，见 `踩坑记录.md` 的 `KAFKA-EVENTBUS-022`。

机制矩阵：

| 机制 | 结论 | 触发条件 | 实现范围 | 测试环境 | 回滚策略 | 主要风险 |
| --- | --- | --- | --- | --- | --- | --- |
| `SCRAM-SHA-512` / `SCRAM-SHA-256` | 先做候选；优先 SHA-512，broker 只提供 SHA-256 时降级 | 目标 Kafka 明确要求 `SASL_SSL + SCRAM`，例如 AWS MSK SCRAM、自管 Kafka 安全策略或企业托管 Kafka 只发放 SCRAM 用户 | `analytics-core` 增加 `SASLMechanismSCRAMSHA512/256`、SCRAM client generator、Sarama config 映射；`analytics-service` 放开 env 校验；runbook 增加 secret rotation 和 SCRAM drill | 本地 disposable SCRAM_SSL broker 或 staging Kafka；复用 publish/consume、ordered commit、restart replay、replicated/outage gate；补 unit test 锁定 unsupported -> supported 的机制映射 | 新机制必须 opt-in；保留当前 `plain` 路径；通过 env 切回 `ANALYTICS_SERVICE_KAFKA_SASL_MECHANISM=plain` 和旧凭据 | SCRAM 凭据轮换、broker 端用户创建方式、TLS/SAN 与 SCRAM 同时配置时的排障复杂度 |
| `OAUTHBEARER` | 暂缓 | 目标平台强制 token/IAM/OIDC，且已有稳定 IdP、STS 或 AWS MSK IAM token provider 方案 | `analytics-core` 需要 token provider 抽象，不只是增加枚举；`analytics-service` 需要 token 来源、刷新、过期、clock skew、audience/scope 配置；runbook 需要 token 轮换和失效演练 | 本地只能测 token provider 单元逻辑；完整 broker handshake 需要 OAuth-capable Kafka 或目标 managed Kafka staging | 保留 SCRAM/PLAIN 备用 listener 或备用 credential；token provider 失败时快速切回非 OAuth 机制 | token 过期、刷新失败、issuer/audience 漂移、云厂商 IAM 细节差异、故障定位跨 Kafka 和 IdP 两层 |
| `Kerberos/GSSAPI` | 当前否决；仅企业定制触发 | 客户已有 KDC/AD、keytab、service principal、DNS/SPN 和运维团队，且明确要求 Kafka 走 Kerberos | 不进入通用 SimpleTrack 第一版；若企业定制，需要独立 `GSSAPIConfig`、krb5.conf/keytab/ccache 管理、SPN/DNS/时钟同步 runbook | 需要专用 KDC + GSSAPI Kafka 演练环境；mock 不足以证明生产握手 | 必须保留 SCRAM/PLAIN/OAuth 备用路径；没有备用路径时不建议接入 | 配置面最宽，CI 难稳定，DNS/时钟/KDC 状态都会影响消费；排障成本远高于 SimpleTrack 当前阶段收益 |

完成结论：

- 当前代码和 runbook 仍只声明 `SASL/PLAIN + TLS` 为已支持生产路径；这不是把 SCRAM/OAuth/Kerberos 开关悄悄放开。
- 下一项代码任务如果目标部署环境确认需要高级认证，应先做 `SCRAM-SHA-512/256`，并保持 `SASL/PLAIN + TLS` 回滚路径。
- `OAuthBearer` 等到平台明确要求 token/IAM/OIDC，并有 token provider 生命周期设计后再做。
- `Kerberos/GSSAPI` 不进入默认产品路线，只作为企业内网 Kafka 定制需求重新评审。

## 推荐执行顺序

## 当前风险

- 多 broker `SASL_SSL` compose 比单 broker 更容易踩 listener / advertised listener / controller quorum / cert SAN 坑。
- Windows + Docker Desktop 下端口、网络和镜像源仍可能不稳定。
- 认证 benchmark 受本机资源和 Docker 网络影响大，只能作为相对基线。
- 指标源已分层，但生产阈值、Grafana dashboard、Prometheus alert rules 和托管 Kafka 具体字段映射仍未落地。
- 如果把 SCRAM/OAuth/Kerberos 直接当作“全部支持”的配置开关放开，会绕过必要的 provider 生命周期、测试环境和回滚设计。
