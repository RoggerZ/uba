## 详细记录

### KAFKA-EVENTBUS-001：实现先于任务 plan 推进

- 发现时间：2026-05-17
- 现象：Kafka provider 代码刚开始推进后，用户调整执行规范，要求先停下并建立独立 plan 目录。
- 风险：半成品 provider 容易让后续误判为“已经实现”，也可能造成工作区不可编译。
- 处理：暂停代码实现，先创建本目录，后续从阶段 1 恢复。
- 后续防线：恢复代码前先更新 `进度表.md`，明确当前阶段、改动范围和验证命令。

### KAFKA-EVENTBUS-002：Sarama 版本查询超时

- 发现时间：2026-05-17
- 现象：`go list -m -versions github.com/IBM/sarama` 访问 `proxy.golang.org` 超时。
- 新证据：`go get github.com/IBM/sarama@v1.47.0` 首次执行卡在 `goproxy.cn/sumdb/sum.golang.org/supported` 连接超时。
- 已知事实：本机 module cache 存在 `github.com/IBM/sarama@v1.47.0`，本地 `xwl_bi` 也使用 `github.com/IBM/sarama v1.47.0`。
- 处理：第一版优先使用 `v1.47.0`，与 `xwl_bi` 经验保持一致；如果后续需要确认最新版，再使用本机代理或 GitHub release 资料做单独验证。
- 当前解法：设置 `HTTP_PROXY` / `HTTPS_PROXY` 为 `http://localhost:64320`，并将 `GOPROXY` 临时切到 `https://proxy.golang.org,direct` 后，`go get github.com/IBM/sarama@v1.47.0` 成功。
- 后续防线：不要因为版本列表查询失败就切换 Kafka client；依赖选型以已确认的计划为准。

### KAFKA-EVENTBUS-003：正式文档过大

- 发现时间：2026-05-17
- 现象：`simpletrack/docs/实施决策/` 下长期文档已经很大，频繁增量会拖慢代码推进。
- 处理：本任务所有过程性记录先写入 `simpletrack/docs/任务计划/kafka-first-eventbus/`。
- 后续防线：过程性记录只写本目录；任务全部执行完成后，再同步正式实施决策文档。

### KAFKA-EVENTBUS-004：commit / push 频率过高

- 发现时间：2026-05-17
- 现象：如果每一点小改动都提交、推送，会打断 Kafka provider 这类核心链路实现节奏；push 本身也可能受网络影响。
- 处理：阶段完成、改动较多、或进入阶段性评审前后再做本地 commit；push 保持低频。
- 后续防线：push 遇到网络问题时，优先使用 `http://localhost:64320` 本地代理重试多次，并把失败证据写入本文件。

### KAFKA-EVENTBUS-005：子代理通道返回 502

- 发现时间：2026-05-17
- 现象：阶段 1 启动 service 影响面和 xwl_bi 搬迁映射两个只读子代理时，本地 responses upstream 返回 `502 Bad Gateway`。
- 根目录参考：`docs/开发环境卡壳问题记录.md` 已有“子代理失败不能算完成”的同类处理原则。
- 处理：不把这两个子代理视为完成；主代理继续本地查代码推进，阶段性交付或评审门槛需要子代理时再缩小范围重试。
- 后续防线：如果阶段评审仍遇到 502，把失败证据和替代验证写清楚，不能冒充子代理审查完成。

### KAFKA-EVENTBUS-006：子代理未显式按 goal 模式启动

- 发现时间：2026-05-17
- 现象：阶段 1 第一次启动只读 explorer 时，父代理没有在子代理 prompt 里明确写“先创建 goal，结束时 update goal”。
- 新证据：KAFKA-HARDEN-002 阶段评审子代理也按普通只读评审 prompt 启动，未显式要求子代理 goal 模式；虽然后续 code-review 和 code-simplifier 均完成并返回有效结论，但该轮不能作为“子代理 goal 模式合规样例”。
- 风险：即使子代理成功返回，也不符合用户要求的全程 goal 模式。
- 处理：已在 `执行规范.md` 补充硬规则；后续所有子代理 prompt 第一条必须要求子代理创建本子任务 goal。对已经完成但未按 goal prompt 启动的子代理，不伪造合规记录，只使用其真实代码审查结论。
- 后续防线：父代理整理子代理输出时，同时检查子代理是否报告 goal 完成或未完成原因；如果子代理平台不支持创建 goal，也必须在 prompt 中要求它声明无法创建 goal 并按子任务目标清单执行。

### KAFKA-EVENTBUS-007：子代理被 goal budget 限制打断

- 发现时间：2026-05-17
- 现象：阶段性交付 code-review 子代理只完成 preflight，随后报告被 active thread goal budget 限制拦住，未进入逐文件审查。
- 风险：如果把“已启动子代理”或“完成 preflight”误当成代码审查完成，会违反阶段评审规则。
- 处理：明确不采纳该轮为完成审查；重新派发更小范围、禁止 MCP、只读的代码审查子任务。早期宽范围评审不能声明通过；后续针对 KAFKA-HARDEN-003 到 007 的窄范围 code-review 初审返回 `REQUEST CHANGES`，指出 stale pause state 和 TLS/SASL material 静默忽略；修复后复审返回 `APPROVE`，这一次才算有效审查证据。
- 后续防线：阶段评审子代理 prompt 要限制范围和工具面，优先让它输出少量高价值 file:line 结论，而不是先做宽泛准备工作。

### KAFKA-EVENTBUS-008：Git commit 钩子要求 OmX co-author trailer

- 发现时间：2026-05-17
- 现象：`analytics-core` 注释补强提交第一次被 PreToolUse hook 拦截，提示 Lore-format commit 还必须包含 `Co-authored-by: OmX <omx@oh-my-codex.dev>`。
- 处理：保留原 Lore commit 内容，追加 OmX co-author trailer 后重新执行，commit `2b4f38d` 成功并已推送。
- 后续防线：本任务和后续 SimpleTrack 子仓提交都默认把该 co-author trailer 放进 commit message，避免到提交阶段才被钩子打断。

### KAFKA-EVENTBUS-009：本地 Kafka compose 镜像源 403

- 发现时间：2026-05-17
- 现象：在 `src/analytics-core` 执行 `docker compose up -d kafka` 时，Docker 拉取 `docker.io/bitnami/kafka:3.9` 失败，错误为当前镜像源 `https://ustc-edu-cn.mirror.aliyuncs.com/v2/bitnami/kafka/manifests/3.9?ns=docker.io` 返回 `403 Forbidden`。
- 影响：`analytics-core` 自带 compose 的单节点 Kafka 暂时没有启动，不能把 `127.0.0.1:29092` 作为本轮真实 broker 证据。
- 临时处理：不扩大到 Docker Desktop / registry 配置排障；复用本机已运行的 `confluentinc/cp-kafka:7.5.0` 容器，对外地址 `127.0.0.1:9092`，完成 KAFKA-HARDEN-001 真实 broker 集成测试。
- 验证证据：`docker exec kafka ... --version` 输出 `7.5.0-ccs`；`Test-NetConnection 127.0.0.1 -Port 9092` 成功；`ANALYTICS_CORE_KAFKA_INTEGRATION=1 ANALYTICS_CORE_KAFKA_BROKERS=127.0.0.1:9092 go test ./eventbus/kafka -run 'TestKafkaIntegration' -count=1 -v` 通过。
- 后续防线：如果必须使用 `analytics-core` compose 自带 Kafka，再单独处理镜像源、预拉镜像或评估把本地开发镜像换成当前机器可获取的 Kafka 镜像；不要把镜像源 403 误判为 Kafka provider 代码失败。

### KAFKA-EVENTBUS-010：`Stats()` 诊断快照被误读为精确事实源

- 发现时间：2026-05-17
- 现象：KAFKA-HARDEN-004 为排障新增 `Bus.Stats()` 和 `Runtime.KafkaEventBusStats()` 后，architect 评审提醒该接口容易被未来使用方误当成精确 accounting、SLO 或 broker lag 事实源。
- 根因：provider 内部只能看到本进程处理状态、ordered committer high watermark 和当前 claim 快照；它不是 broker exporter，也不读取 Kafka broker 端完整 consumer group lag。
- 处理：runbook 的“观测入口”明确 `Stats()` 是诊断快照；待办事项和进度表也标注 `lag estimate`，避免写成精确 lag。
- 后续防线：如果后续接入 Prometheus 或告警，`Stats()` 只作为进程内症状信号；精确 consumer lag、broker ISR、controller、topic health 仍以 Kafka exporter / broker metrics 为准。
