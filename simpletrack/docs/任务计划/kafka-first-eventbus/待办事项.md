## 任务细化

### KAFKA-HARDEN-001：真实 Kafka broker 集成验证（已完成）

- 完成版本：`analytics-core` commit `2d3063a`。
- 实际 broker：本机已有 `confluentinc/cp-kafka:7.5.0` 容器，对外地址 `127.0.0.1:9092`，`kafka-topics --version` 输出 `7.5.0-ccs`。
- 通过命令：`ANALYTICS_CORE_KAFKA_INTEGRATION=1 ANALYTICS_CORE_KAFKA_BROKERS=127.0.0.1:9092 go test ./eventbus/kafka -run 'TestKafkaIntegration' -count=1 -v`。
- 覆盖测试：`TestKafkaIntegrationPublishConsume` 验证 publish -> consume；`TestKafkaIntegrationOrderedCommitWaitsForEarlierOffset` 验证 offset 1 先完成时不会在 offset 0 前提交，offset 0 释放后提交到 2；`TestKafkaIntegrationUncommittedMessageReplaysAfterRestart` 验证 handler 取消后 offset 不提交，新的同组 consumer 能 replay 原 offset 0。
- 额外验证：`go test ./eventbus/kafka`、`go test -run Example ./eventbus/kafka`、`go vet ./eventbus/kafka`、`golint ./eventbus/kafka`、`git diff --check -- eventbus/kafka/integration_test.go` 均通过。
- 踩坑记录：`docker compose up -d kafka` 拉取 `bitnami/kafka:3.9` 时被当前 Docker 镜像源返回 `403 Forbidden`，详见 `踩坑记录.md` 的 `KAFKA-EVENTBUS-009`。

### KAFKA-HARDEN-002：Producer 可靠性默认值评审与落地（已完成）

- 完成版本：`analytics-core` commit `cc8797c`；父仓同步 commit `3f97360`。
- 生产默认：`ProducerAcks` 默认为 `all`，内部映射到 Sarama `WaitForAll`；`ProducerRetryMax` 默认 `5`，`ProducerRetryBackoff` 默认 `100ms`。
- API 语义：`ProducerRetryMax` 与 `ProducerRetryBackoff` 改为指针；`nil` 表示使用可靠默认值，显式 `0` 保留为调用方选择，不再静默覆盖。
- idempotent producer：仍为显式 opt-in；启用后内部设置 `Producer.Idempotent=true`、`Net.MaxOpenRequests=1`，并拒绝 `acks != all` 或 `retry max == 0`。
- 覆盖测试：默认 durable config、`leader` / `none` acks 映射、retry / flush 映射、显式 0 retry/backoff、非幂等默认 in-flight、不合法配置和 idempotent Sarama 约束。
- 验证：`go test ./eventbus/kafka`、`go test -run Example ./eventbus/kafka`、`go vet ./eventbus/kafka`、`golint ./eventbus/kafka`、`go test ./...`、`go test -run Example ./...`、`go vet ./...`、`go doc -all ./eventbus/kafka`、目标文件 `git diff --check` 均通过。
- 阶段评审：code-review 初审指出 producer retry 零值语义缺口，修复后复审 `APPROVE`；code-simplifier / ai-slop-cleaner 只读检查 `APPROVE`。

### KAFKA-HARDEN-003：Consumer rebalance 与 ordered commit 集成压测（已完成）

- 背景：ordered committer 已覆盖乱序完成，但 rebalance / claim 关闭场景仍是生产风险点。
- 完成内容：`ConsumeClaim` 使用 generation id 绑定当前 claim 生命周期；claim 退出时 abort 当前 generation，late completion 不能再通过 stale Sarama session 标记提交。
- 覆盖测试：ordered commit 乱序完成、claim 关闭后 late completion、DLQ 写失败不提交原 offset、pause 状态在 rebalance 后按当前压力重新 reconcile。
- 完成版本：`analytics-core` commit `2e52263`。
- 验证：`go test ./eventbus/kafka -count=1`、`go test ./...`、`go test -run Example ./...`、`go vet ./...`、`go doc -all ./eventbus/kafka`、`golint ./eventbus/kafka`、`git diff --check` 均已通过。

### KAFKA-HARDEN-004：DLQ / retry / consumer lag / pause-resume 可观测性（已完成）

- 背景：v1 已有 retry、DLQ 和 protector 基础逻辑，但生产排障需要可观察信号。
- 完成内容：`analytics-core/eventbus/kafka.Bus.Stats()` 暴露诊断快照；`analytics-service/internal/runtime.Runtime.KafkaEventBusStats()` 提供窄出口，后续可接入 exporter 或结构化日志。
- 指标范围：consume count、handler success/failure/retry、malformed、DLQ success/failure、paused partitions、pause/resume transition、worker queue、completion gate、ordered commit next offset 与 lag estimate。
- 边界说明：`Stats()` 是诊断快照，不作为 billing、SLO 或精确 broker lag 的唯一事实源；不包含原始事件 payload。
- 完成版本：`analytics-core` commit `2e52263`；`analytics-service` commit `9acadab`。

### KAFKA-HARDEN-005：SASL/TLS 与 broker ACL 配置矩阵（已完成）

### KAFKA-PROD-001：Kafka diagnostics 服务级观测出口（已完成）

- 完成版本：`analytics-service` commit `0f695db`；父仓同步 commit `32b4dbf`；`analytics-core` 复用 commit `2e52263` 的 `eventbus/kafka.Bus.Stats()`。

- 鉴权边界：复用 query token 生命周期和 scope 机制，新增 `kafka_diagnostics` scope；该 route 是进程级诊断，不需要 `write_key`，也不会走 source readback policy。
- 响应范围：返回 topic、DLQ topic、worker pool、completion gate、ordered commit、paused partitions、retry / DLQ / pause-resume counters；`lag_estimate` 明确是本进程诊断估算，不是 broker 权威 lag。
- 已通过验证：`go test ./internal/config ./internal/collectapi ./internal/runtime -count=1`、`go test ./...`、`go test -run Example ./...`、`go vet ./...`、`golint ./internal/collectapi ./internal/config ./internal/runtime ./internal/controlplane`、目标包 `go doc -all`、`git diff --check`。
- 审查修复：code-review 初审指出 diagnostics 会被误带 `write_key` / `AllowedWriteKeys` 误伤，以及 `paused` 可能输出 `null`；已改为 process-scoped token 校验并固定空 `paused` 为 `{}`，补回归测试。
- 复审结论：code-review follow-up 返回 `APPROVE`，确认 `write_key` 误伤和空 `paused` 漂移均已收口。
- 收口说明：该接口只暴露进程内诊断快照，不回显原始事件 payload、broker secret、TLS/SASL material，也不作为精确 broker lag / SLO / billing 事实源。
