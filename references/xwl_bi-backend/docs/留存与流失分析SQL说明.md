# 留存与流失分析 SQL 逻辑说明

本文档对系统中的留存分析与流失分析的 SQL 逻辑及前端展示逻辑进行解析，解释为何两者使用相同的 SQL 语句以及数据的具体含义。

## 1. SQL 语句分析

系统中“留存分析”与“流失分析”使用的是同一套 SQL 生成逻辑。该 SQL 基于 ClickHouse 的 `retention` 函数实现。

### 1.1 SQL 结构

```sql
SELECT
    '2026-02-23' AS dates,
    -- value 数组存储的是【留存用户数】
    array(
        sum(r[1]), -- 初始事件触发用户数 (Cohort Size)
        sum(r[2]), -- 第 0 天留存用户数 (通常等于初始用户数或完成了回访事件的用户数)
        sum(r[3]), -- 第 1 天留存用户数
        sum(r[4]), -- 第 2 天留存用户数
        ...
    ) as value,
    -- ui 数组存储的是【留存用户 ID】
    array(
        groupUniqArray(if(r[1] = 1, xwl_distinct_id, null)),
        groupUniqArray(if(r[2] = 1, xwl_distinct_id, null)),
        ...
    ) as ui
FROM (
    SELECT
        xwl_distinct_id,
        -- retention 函数返回 0/1 数组，表示用户是否满足后续日期的回访条件
        retention(
            xwl_part_event = '支付订单' and toYYYYMMDD(xwl_part_date) = '20260223', -- 初始条件
            xwl_part_event = '应用启动' and toYYYYMMDD(xwl_part_date) = '20260223', -- 第 0 天回访
            xwl_part_event = '应用启动' and toYYYYMMDD(xwl_part_date) = '20260224', -- 第 1 天回访
            ...
        ) AS r
    FROM xwl_event41
    PREWHERE ...
    GROUP BY xwl_distinct_id
)
LIMIT 1000
```

### 1.2 关键字段解释

*   **`r` (Retention Result)**: `retention` 函数返回的一个由 0 和 1 组成的数组。
    *   `r[1] = 1`: 表示该用户满足初始行为（如“支付订单”）。
    *   `r[n] = 1`: 表示该用户在第 n-2 天满足了回访行为（如“应用启动”）。
*   **`sum(r[n])`**: 统计满足第 n-2 天回访行为的**留存用户总数**。

## 2. 留存与流失的关系

### 2.1 数据源一致性

系统在进行“流失分析”时，**并没有生成专门计算流失数的 SQL**，而是直接复用了“留存分析”的 SQL。

*   **留存分析 SQL 返回的数据**：`[初始用户数, 第0天留存数, 第1天留存数, ...]`
*   **流失分析 SQL 返回的数据**：`[初始用户数, 第0天留存数, 第1天留存数, ...]` （完全相同）

### 2.2 流失指标的计算

由于 SQL 返回的是**留存用户数**，前端页面在展示“流失分析”时，是通过计算得出的：

*   **流失用户数 (Churn Count)** = `初始用户数` - `留存用户数`
*   **流失率 (Churn Rate)** = `(1 - 留存率)` * 100%

## 3. 前端展示逻辑与常见误解

### 3.1 前端展示逻辑

在前端页面 (`RetentionResult.vue`) 中，即使用户选择了“流失分析”视图，表格中的数据展示逻辑如下：

1.  **人数栏 (Number Column)**:
    *   **逻辑**: 直接展示后端 SQL 返回的 `value` 字段。
    *   **结果**: 无论是在“留存分析”还是“流失分析”模式下，这一栏显示的**始终是留存人数**。
    *   **原因**: 前端的钻取功能（点击数字查看用户列表）依赖于后端返回的留存用户 ID 列表 (`ui`)。如果展示流失人数，点击后却跳转到留存用户列表，会导致逻辑混乱。

2.  **比例栏 (Percentage Column)**:
    *   **留存分析模式**: 展示留存率 (`留存人数 / 初始人数 * 100%`)。
    *   **流失分析模式**: 展示流失率 (`100% - 留存率`)。

### 3.2 常见误解：为什么流失分析中人数显示为 0？

**现象**：
在“流失分析”视图下，第 2 天的数据显示为 **0**。

**原因解析**：
*   这个 **0** 代表的是**留存人数**（因为 SQL 返回的是留存数）。
*   这意味着在第 2 天，**没有任何用户留存**（留存率为 0%）。
*   对应的流失率应该是 **100%**（所有用户都流失了）。

**容易产生的误解**：
用户可能会直觉地认为“流失分析”下显示的数字应该是“流失人数”，因此看到 0 会误以为是“0 人流失”（即 100% 留存）。但实际上这里的 0 代表“0 人留存”（即 100% 流失）。

**如何验证**：
请查看该单元格下方的百分比数值。如果人数显示 0，百分比显示 100.00%，则说明所有用户均已流失。

### 3.3 总结

| 分析模式 | 人数栏显示内容 | 比例栏显示内容 | 点击钻取对象 |
| :--- | :--- | :--- | :--- |
| **留存分析** | **留存人数** | 留存率 | 留存用户列表 |
| **流失分析** | **留存人数** (注意！) | **流失率** | 留存用户列表 |

## 4. 日期对应关系与计算基准

### 4.1 日期对应关系

假设您在 **2026-02-24** (今天) 进行分析，选择了 **“昨天” (2026-02-23)** 作为分析日期，时间窗口为 **7天**。表格中的各列对应的实际日期如下：

| 表头名称 | 含义 | 对应实际日期 |
| :--- | :--- | :--- |
| **初始事件触发用户数** | 基准分母 (Cohort Size) | **2026-02-23** (昨天) |
| **当天** | 第 0 天留存 (Day 0 Retention) | **2026-02-23** (昨天) |
| **第 1 天** | 次日留存 (Day 1 Retention) | **2026-02-24** (今天) |
| **第 2 天** | 2日留存 (Day 2 Retention) | **2026-02-25** (明天) |
| **第 3 天** | 3日留存 (Day 3 Retention) | **2026-02-26** (后天) |
| ... | ... | ... |
| **第 7 天** | 7日留存 (Day 7 Retention) | **2026-03-02** |

### 4.2 计算基准

所有的留存率（或流失率）计算，**分母始终是“初始事件触发用户数”**。

*   **第 N 天留存率** = `(第 N 天回访用户数) / (初始事件触发用户数)`
*   **而不是**相对于前一天的数据。

**示例**：
*   初始用户数：100 人
*   第 1 天留存用户数：40 人
*   第 2 天留存用户数：30 人

*   **第 2 天留存率** = 30 / 100 = 30% (而不是 30/40 = 75%)
