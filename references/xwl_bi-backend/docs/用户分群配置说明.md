# 用户分群配置说明（规则型分群 vs 快照分群）

本文档用于回答“用户分群是否支持按规则配置（如过去30天累计充值金额>50）”以及“如何基于行为分析结果创建分群”的问题。

## 1. 结论

当前系统内的“用户分群（UserGroup）”是一个**静态的用户ID列表快照**，不是“规则型分群（Cohort/Segment）”。

*   **可做的事**：先在行为分析里算出某个结果单元格对应的一批用户（uid 列表），再“保存为分群”。
*   **不可做的事**：把“过去30天金额>50”等规则保存成配置，并让分群自动定时重算、动态变化。

更详细的链路可参考：[UserGroup逻辑链路分析.md](file:///c:/Users/admin/Documents/src/xwl_bi/docs/UserGroup逻辑链路分析.md)。

## 2. 为什么无法“按规则配置分群”

从实现上看：

*   MySQL 的 `user_group` 表只保存压缩后的 `user_list`（uid 列表）以及名称/备注/人数等元信息，没有存放规则的字段。
*   创建分群的 API 仅接收 `uids + name(+remark)`，后端将 uid 列表 gzip 压缩后落库；不接受“条件表达式/规则 JSON”。
*   分群在分析中使用时，后端从 MySQL 取出 uid 列表并注入 SQL 的 `IN (...)` 条件；没有“重新计算分群”的入口与调度逻辑。

## 3. 现有系统里“分群”怎么创建

### 3.1 创建入口

分群创建入口主要来自各类行为分析的结果表格单元格（该单元格通常已经携带对应人群的 uid 列表）：

*   漏斗结果页：[FunnelResult.vue](file:///c:/Users/admin/Documents/src/xwl_bi/vue/src/views/behavior-analysis/components/FunnelResult.vue)
*   留存结果页：[RetentionResult.vue](file:///c:/Users/admin/Documents/src/xwl_bi/vue/src/views/behavior-analysis/components/RetentionResult.vue)
*   路径结果页：[TraceResult.vue](file:///c:/Users/admin/Documents/src/xwl_bi/vue/src/views/behavior-analysis/components/TraceResult.vue)

### 3.2 基本操作方式

1. 先在行为分析页面配置时间范围、事件与过滤条件，点击“计算”得到结果。
2. 在结果表中找到你要的人群对应的单元格（通常是某个步骤/某一天/某条路径的用户数）。
3. 点击单元格旁的“保存为分群”（弹出创建分群弹窗），填写分群名并保存。

## 4. 四类用户分群如何创建（现有能力可做/不可做的边界）

下面按“能否用现有链路落地”为每类分群给出可行路径与限制。

### 4.1 剧集偏好群（可做）

**目标**：过去30天 `View Drama Detail` 中 `tags` 包含 “逆袭” 或 “豪门” 的用户。

**推荐做法**：

1. 时间范围选择最近30天。
2. 在行为分析里选择能产出 uid 单元格的分析类型（漏斗/路径/留存均可，优先选择能直观看到该事件触发用户的页面）。
3. 过滤条件：
    *   事件名：`View Drama Detail`
    *   事件属性：`tags`
    *   操作符：`match`（正则）
    *   值：`逆袭|豪门`
4. 在结果表里找到对应单元格，点击“保存为分群”创建。

**注意**：如果 `tags` 是数组而不是字符串，当前筛选控件/后端 SQL 可能仍按字符串处理，需要以实际埋点字段类型为准。

### 4.2 潜在流失用户 Risk（部分可做）

**目标**：过去7天无 `Launch App` 事件，且曾有充值记录。

**可落地的近似路径**（更贴近“无事件”判定）：

1. 使用留存分析，以“充值/购买成功”等事件作为初始事件（用于保证“曾有充值记录”）。
2. 将回访事件选择为 `Launch App`。
3. 窗口设为 7 天，切换到“流失”视角，找到“第7天流失”对应的人群单元格。
4. 在该单元格点击“保存为分群”创建。

**限制**：

*   “曾有充值记录”是否可做到“历史全量”取决于你设置的分析时间范围；当前界面一般只能选一个时间段，无法天然表达“历史任意时间发生过充值”。
*   如果“充值记录”不等同于某个事件（例如来自外部账单表），则需要先保证该行为已经以事件方式上报并可分析。

### 4.3 高价值用户 High Value（现状无法精确实现）

**目标**：过去30天 `Purchase` 事件的总金额（按用户聚合） > $50。

**核心缺口**：需要先做“按用户聚合 SUM(amount)”并按阈值筛选用户，再导出 uid 列表保存分群；当前“保存为分群”的链路依赖分析结果单元格自带 uid 列表，且不提供“按用户聚合后再按阈值切人群并输出 uid”的能力。

**可行替代（近似）**：

*   若业务能接受近似，可筛“单笔金额 > 50”（事件属性过滤），但这与“30天累计 > 50”不同。

**推荐落地方案（需要数据侧配合）**：

*   将“过去30天累计充值金额”离线计算为用户属性（profile），写回用户属性表；
*   在分析里使用“全局筛选用户维度”按该用户属性 `> 50` 筛选；
*   再通过能够产出 uid 的结果页单元格“保存为分群”。

### 4.4 沉浸式用户 Engaged（现状无法精确实现）

**目标**：过去7天 `Watch Video` 时长（按用户聚合） > 60分钟。

**结论**：同“高价值用户”的问题，本质是按用户聚合 SUM(duration) 后按阈值筛选并导出 uid，当前链路不支持精确生成该 uid 集合并保存为分群。

**推荐落地方案（需要数据侧配合）**：

*   将“过去7天观看时长”离线汇总为用户属性；
*   通过用户属性筛选后，在可导出 uid 的分析结果里“保存为分群”。

## 5. 如果要支持“规则型分群”，需要补齐哪些能力（建议）

如果要原生支持类似下面的分群：

*   过去30天累计充值金额 > 50
*   过去7天观看时长 > 60 分钟
*   过去7天未启动且历史充值过

通常需要新增：

1. **分群规则存储**：在分群实体中增加规则字段（如 JSON DSL），支持 AND/OR、事件条件、用户属性条件、聚合条件、时间窗口等。
2. **规则执行/重算**：后端实现规则到 ClickHouse SQL 的编译与执行，产出 uid 列表并落库；支持手动重算/定时重算。
3. **动态语义**：明确分群是“快照”还是“动态”，并在 UI 上区分展示与使用方式。
4. **聚合与否定条件**：支持按用户聚合（sum/avg/count distinct）以及“无事件/未发生”类条件的表达与计算。
