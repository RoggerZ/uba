# 事件分析十一年范围点击“展开侧边栏”卡顿分析

## 结论摘要

- 这次卡顿的主因在前端渲染层，不是点击“展开侧边栏”后又重新发了一次事件分析接口。
- 问题本质是：十一年范围下，事件分析默认按天展示，会生成 4000+ 时间桶；而点击“展开侧边栏”时，当前实现又会同步触发结果区重建、表格重排、图表 resize，导致主线程阻塞。
- 仅从现有代码看，卡顿是多个前端开销叠加后的结果，其中最关键的两个点是：
  - 结果区数据量和 DOM 规模过大。
  - 侧边栏展开动作不是“轻量改宽度”，而是带着“重建结果组件”的副作用。

## 现象对应代码链路

### 1. 点击“展开侧边栏”时做了什么

文件：`vue/src/views/behavior-analysis/event.vue`

- 页面使用了 `split-pane`，并且带了 `:key="splitKey"`。
- 点击右侧按钮会执行 `toggleSidebar()`。
- `toggleSidebar()` 中做了 3 件事：
  - 切换 `sidebarCollapsed`。
  - 修改 `splitPercent`。
  - `splitKey++`，强制重建整个 `split-pane`。
  - 在下一帧里再触发一次 `window.dispatchEvent(new Event('resize'))`。

对应位置：

- `split-pane :key="splitKey"`：`vue/src/views/behavior-analysis/event.vue:35`
- 展开/收起按钮：`vue/src/views/behavior-analysis/event.vue:395`
- `toggleSidebar()`：`vue/src/views/behavior-analysis/event.vue:598`

这意味着当前交互不是简单地调整左右宽度，而是会让 `split-pane` 及其内部右侧结果区重新挂载一次。

### 2. 点击侧边栏不会重新请求接口，但会重新构造结果区

文件：`vue/src/views/behavior-analysis/event.vue`

- 真正请求事件分析接口的是 `go()`。
- `toggleSidebar()` 并没有调用 `go()`，所以这次卡顿不是因为又发了一次 `EventList` 请求。
- 但是 `splitKey++` 会导致右侧 `EventResult` 重新挂载，而 `EventResult` 在 `mounted()` 时会执行 `init()`，重新构造表格和图表数据。

对应位置：

- `go()`：`vue/src/views/behavior-analysis/event.vue:904`
- `EventResult` 挂载：`vue/src/views/behavior-analysis/components/EventResult.vue:475`
- `EventResult.init()`：`vue/src/views/behavior-analysis/components/EventResult.vue:773`

## 为什么“十一年范围”会把问题放大

### 1. 默认按天展示，十一年约等于 4000+ 日期列

文件：`vue/src/views/behavior-analysis/event.vue`

- 页面默认 `windowTimeFormat` 是“按天”。

对应位置：

- 默认粒度：`vue/src/views/behavior-analysis/event.vue:488`
- 兜底粒度：`vue/src/views/behavior-analysis/event.vue:810`

文件：`vue/src/views/behavior-analysis/components/EventResult.vue`

- `init()` 会遍历 `eventRes.alldata`，把所有日期收集到 `allDateColumns`。
- 在按日期分组模式下，会把每一个日期都追加成一个表格列。

对应位置：

- 收集全部日期列：`vue/src/views/behavior-analysis/components/EventResult.vue:796`
- 保存到 `allDateColumns`：`vue/src/views/behavior-analysis/components/EventResult.vue:806`
- 将日期逐个 push 到 `tableTitle`：`vue/src/views/behavior-analysis/components/EventResult.vue:768`

如果时间范围是十一年，按天展示时日期列数量会直接进入 4000+ 级别。只要还叠加以下任一因素，体量会进一步上升：

- 多指标
- 分组维度
- 对比时间范围
- 表格搜索、翻页、重绘
- 图表多 series 展示

### 2. Element Table 渲染的是“超宽动态列”，容器宽度变化时回流成本很高

文件：`vue/src/views/behavior-analysis/components/EventResult.vue`

- 结果表通过 `v-for="(v,k,index) in tableTitle"` 动态生成列。

对应位置：

- 动态列定义：`vue/src/views/behavior-analysis/components/EventResult.vue:240`

文件：`vue/src/components/PageTable/index.vue`

- 底层仍然是 `el-table`。
- `el-table` 在 4000+ 列这种宽表场景下，本身就会非常吃布局计算和 DOM 更新。

对应位置：

- `el-table`：`vue/src/components/PageTable/index.vue:3`

当左侧展开时，右侧宽度发生变化，浏览器需要重新计算这个超宽表格的列布局、滚动区域和可见区域，成本非常高。

### 3. 右侧结果区会重新做一轮数据整理

文件：`vue/src/views/behavior-analysis/components/EventResult.vue`

- `init()` 里会重新生成：
  - `tableTitle`
  - `tableData`
  - `allDateColumns`
  - `selectedDates`
  - `chartSeries`
  - `chartXAxis`
- 这不是纯展示层的宽度变化，而是一次完整的数据转换过程。

对应位置：

- 重建表格数据：`vue/src/views/behavior-analysis/components/EventResult.vue:783`
- 生成 `tableData`：`vue/src/views/behavior-analysis/components/EventResult.vue:815`
- 刷新表格：`vue/src/views/behavior-analysis/components/EventResult.vue:898`
- 初始化图表数据：`vue/src/views/behavior-analysis/components/EventResult.vue:899`
- 图表数据重建：`vue/src/views/behavior-analysis/components/EventResult.vue:1004`
- 图表 series 更新：`vue/src/views/behavior-analysis/components/EventResult.vue:1045`

### 4. 图表还会收到一次全局 resize

文件：`vue/src/views/behavior-analysis/event.vue`

- `toggleSidebar()` 最后手动派发了全局 `resize` 事件。

对应位置：

- `window.dispatchEvent(new Event('resize'))`：`vue/src/views/behavior-analysis/event.vue:607`

文件：`vue/src/components/Charts/mixins/resize.js`

- 图表 mixin 会监听全局 `resize`，然后执行 `chart.resize()`。

对应位置：

- 监听 `resize`：`vue/src/components/Charts/mixins/resize.js:15`
- 执行 `chart.resize()`：`vue/src/components/Charts/mixins/resize.js:12`

文件：`vue/src/components/Charts/EventAnalysisChart.vue`

- 图表组件在数据变化时还会执行 `setOption(option, true)`，这会让大图表在结果区重建后再次完整刷新。

对应位置：

- `setOption(option, true)`：`vue/src/components/Charts/EventAnalysisChart.vue:241`

## 额外放大因素

以下问题不是这次卡顿的第一主因，但会继续放大卡顿体感。

### 1. `PageTable` 有额外的刷新副作用

文件：`vue/src/components/PageTable/index.vue`

- `filterList` 是个 computed，但内部会调用 `refreshPage()`。
- `refreshPage()` 又会把 `pageshow` 先置 `false` 再置回 `true`，会带来额外的组件刷新。

对应位置：

- `filterList`：`vue/src/components/PageTable/index.vue:105`
- `refreshPage()`：`vue/src/components/PageTable/index.vue:146`

这部分不是“展开侧边栏”卡顿的根因，但在大表格场景下属于明显的额外损耗。

### 2. 结果区重建会把日期筛选状态重置为全量

文件：`vue/src/views/behavior-analysis/components/EventResult.vue`

- 每次 `init()` 时都会重新设置 `selectedDates = [...this.allDateColumns]`。

对应位置：

- 全量选中日期列：`vue/src/views/behavior-analysis/components/EventResult.vue:807`

如果后续用户做了日期筛选，这种重建也会让结果区重新回到“最重状态”。

## 根因判断

按优先级排序，当前问题可以归纳为：

1. 十一年范围按天展示，导致结果表和图表的数据规模过大。
2. 侧边栏展开动作通过 `splitKey++` 强制重建了结果区。
3. 重建后又触发全局 `resize`，进一步放大图表和表格的回流/重绘成本。
4. 表格底层没有做列虚拟化，`el-table` 不适合承载 4000+ 动态列。
5. 页面内部还有一些额外刷新副作用，在大数据量下会进一步恶化卡顿。

## 解决方案建议

## 方案一：先限制时间粒度和时间桶数量

优先级：最高

这是性价比最高的止血方案，因为十一年按天本身就不适合直接渲染成前端宽表。

建议：

- 当时间范围超过阈值时，自动切换时间粒度，不再默认“按天”。
- 建议规则：
  - `<= 93 天`：允许按天
  - `94 天 ~ 365 天`：默认按周
  - `> 365 天`：默认按月
- 当用户强制选择超大范围按天时，直接给出提示，而不是无上限渲染。
- 给前端和后端同时加“最大时间桶数量”限制，例如：
  - 表格最大 366 列
  - 图表最大 730 个点
  - 超限后要求切换粒度或只支持导出

原因：

- 只要还保留“十一年按天 = 4000+ 列”的前提，即使优化侧边栏逻辑，页面依然会脆弱。

## 方案二：侧边栏展开不要重建 `EventResult`

优先级：最高

建议：

- 去掉依赖 `splitKey++` 的强制重建方案。
- 侧边栏展开/收起只做容器宽度变化，不要让右侧结果组件重新挂载。
- 如果当前 `split-pane` 组件不支持受控更新宽度，建议：
  - 改成纯 CSS `flex-basis` / `width` 控制
  - 或更换为支持受控尺寸的分栏组件

原因：

- 用户点击的是布局开关，不是重新计算结果。
- 布局动作应该只触发轻量 resize，不应该再次执行 `EventResult.init()` 这种重活。

## 方案三：表格改成“懒加载/虚拟化/分段展示”

优先级：高

建议：

- 大范围场景下默认只展示图表和汇总，不立即挂载明细表格。
- 用户切换到“表格展示”时再懒加载表格。
- 对表格做列虚拟化或分段加载，不要一次性挂 4000+ 列。
- 如果业务允许，可以只展示最近 N 个时间桶，把全量数据放到导出。

原因：

- 当前瓶颈里，`el-table` 的超宽列渲染是最重的一部分之一。
- 只靠图表 `resize` 优化，无法解决宽表 DOM 过大的根问题。

## 方案四：缩小 resize 的影响范围

优先级：中高

建议：

- 不再派发全局 `window.resize`。
- 改为只对当前图表实例调用 `resize()`。
- 把 resize 放到 `requestAnimationFrame` 或更严格的 debounce/throttle 中。
- 对于纯表格区域，不要因为图表需要 resize 就跟着整块重建。

原因：

- 全局 `resize` 容易误伤所有监听它的组件。
- 现在这个页面的目标只是调整右侧容器尺寸，不需要通知整个页面树。

## 方案五：避免结果区在布局变化时重复做数据转换

优先级：中

建议：

- 将 `eventRes -> tableTitle/tableData/chartSeries/chartXAxis` 的转换结果缓存起来。
- 缓存 key 可以基于：
  - `eventRes`
  - `groupType`
  - `windowTimeFormat`
  - `selectedDates`
- 侧边栏展开/收起时，如果数据本身没变，只做视图 resize，不重复 `init()`。

原因：

- 当前卡顿的一部分来自“数据没变，但组件重建后又重算了一遍”。

## 方案六：清理次级刷新副作用

优先级：中

建议：

- 去掉 `PageTable.filterList` 中的副作用调用，避免 computed 里触发 `refreshPage()`。
- 避免通过 `tableShow=false/true` 方式强制刷新表格，改为更细粒度更新。
- 检查是否有不必要的 deep watch 和整对象替换。

原因：

- 这些问题单独看不一定致命，但在“4000+ 列宽表”场景下会明显放大卡顿。

## 推荐执行顺序

建议按下面顺序推进：

1. 限制超大时间范围下的时间粒度和时间桶数量。
2. 去掉侧边栏展开时对 `split-pane` 和 `EventResult` 的强制重建。
3. 将表格改成懒加载或虚拟化，避免一次性渲染超宽列。
4. 收窄 resize 的影响范围，只精确刷新图表实例。
5. 清理 `PageTable` 等组件里的额外刷新副作用。

## 风险说明

- 只做方案二，不做方案一，卡顿会缓解，但超长时间范围下仍可能慢，因为 4000+ 列本身就很重。
- 只做方案一，不做方案二，日常使用会改善很多，但布局开关仍然存在不必要的重建。
- 真正稳妥的方案，是“限制时间桶规模 + 布局变化不重建结果区 + 宽表懒加载/虚拟化”组合落地。

## 最终判断

这次“点击展开侧边栏特别卡顿”的根本原因，不是某一行代码单点失误，而是：

- 事件分析页面允许“十一年按天”的超大结果规模；
- 侧边栏展开实现又把一个本该轻量的布局动作，做成了“结果组件重建 + 全局 resize”；
- 最终在 `el-table` 宽表和 ECharts 大图表上集中爆发。

因此，修复方向不应该只盯着“按钮点击慢”，而应该同时控制：

- 数据规模
- 结果区生命周期
- 宽表渲染策略
- resize 传播范围
