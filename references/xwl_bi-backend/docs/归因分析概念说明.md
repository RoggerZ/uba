# 归因分析概念说明

## 1. 什么是归因分析

归因分析，可以理解成一句话：

**当用户最终发生了“注册、登录、付费、解锁、留存”这些结果时，我们要把这份结果，尽量合理地分配给前面带来这个用户的来源。**

比如一个用户：

1. 先在 TikTok 广告里看到短剧推广
2. 点进来安装 App
3. 先以游客身份浏览
4. 第二天注册
5. 第三天充值

这时候业务上最常见的问题就是：

- 这个注册算哪个渠道带来的？
- 这笔付费应该算给哪个广告？
- 这个用户到底是自然量，还是投放量？
- 不同渠道带来的用户，后面留存和付费谁更好？

这类“把结果追溯到来源”的分析，就是归因分析。

## 2. 为什么要做归因分析

如果没有归因分析，团队通常只能看到“总注册数、总付费金额、总活跃人数”，但看不清：

- 哪个渠道真正带来了用户
- 哪个渠道只是带来了点击，却没有带来付费
- 哪种投放素材吸引来的用户更容易留存
- 游客转注册，到底是哪个入口、哪个来源推动的

归因分析的核心价值，不是单纯“记来源”，而是回答一句更有业务价值的话：

**钱花在哪，结果从哪来，后续值不值。**

## 3. 结合当前项目，归因通常看哪些字段

从当前仓库里的埋点方案来看，归因最相关的字段主要有：

| 字段 | 通俗解释 | 在归因里的作用 |
| :--- | :--- | :--- |
| `referrer` | 安装渠道、外部带量来源 | 常用来判断用户最初从哪来 |
| `source` | 更细的流量来源或业务来源 | 用来补充区分来源细节 |
| `device_id` | 设备标识 | 用户未注册前，常用它串起游客行为 |
| `guest_user_id` | 游客阶段的用户ID | 用于识别游客期行为 |
| `previous_user_id` | 登录前账号ID | 帮助识别身份切换前是谁 |
| `current_user_id` | 登录后账号ID | 帮助确认最终归属到哪个账号 |
| `provider` | 登录方式 | 可分析 Google、Apple、TikTok 等方式的转化差异 |
| `login_scene` / `register_scene` | 登录或注册发生时的业务场景 | 用来理解“为什么在这个时点发生转化” |

可以把它们理解成两层：

- **来源层**：`referrer`、`source`
- **身份串联层**：`device_id`、`guest_user_id`、`previous_user_id`、`current_user_id`

归因分析最难的地方，往往不是“看哪个渠道”，而是“怎么把同一个人前后多个身份串起来”。

## 4. 归因分析里最常见的概念

下面把常见概念用尽量直白的话解释。

### 4.1 渠道

渠道就是“用户从哪里来的”。

比如：

- TikTok 广告
- Facebook 广告
- 应用商店自然流量
- KOL 分享链接
- 老用户转发

如果把用户比作进店顾客，那渠道就是“他是从哪条街、哪个门、谁带进来的”。

### 4.2 来源（Source）

来源比“渠道”更细一点。

例如同样都是 TikTok：

- 一个是信息流广告
- 一个是达人短视频挂链
- 一个是直播间导流

它们大类都属于 TikTok，但细分来源不同。

所以通常可以这样理解：

- **渠道**：大类来源
- **来源**：更细颗粒度的子来源

### 4.3 触点

触点就是“用户和业务发生接触的一次机会”。

比如：

- 看到一条广告
- 点了一次落地页
- 打开一次 App
- 点击一次登录按钮
- 浏览一次充值弹窗

一个用户往往不是只接触一次，而是会经过很多触点。

归因分析，本质上就是在问：

**最后这次转化，应该记到哪一个触点，或者哪几个触点身上？**

### 4.4 转化

转化就是“用户做了你想要的那个动作”。

常见转化事件有：

- 安装
- 激活
- 注册
- 登录成功
- 付费
- 解锁剧集
- 次日留存

不同团队关注的转化目标不同：

- 增长团队更关心注册、激活
- 商业化团队更关心充值、解锁
- 内容团队更关心留存、完播

### 4.5 转化路径

转化路径就是用户从“第一次接触”到“最终完成目标”中间走过的过程。

例如：

`广告曝光 -> 广告点击 -> App安装 -> 游客浏览 -> 登录成功 -> 充值`

这个链路越清楚，团队越能知道：

- 卡在哪一步
- 哪一步流失最大
- 哪种来源带来的用户路径更短

### 4.6 归因对象

归因对象，就是“最终要把功劳记给谁”。

常见有三种：

- 记给渠道
- 记给广告计划或素材
- 记给某个触点事件

例如：

- 把一笔充值归给 TikTok 渠道
- 把一个注册归给某个广告组
- 把一个转化归给“最后一次点击登录按钮”

### 4.7 归因窗口

归因窗口可以理解成“追溯有效期”。

例如规定：

- 点击后 7 天内注册，算这个广告带来的
- 曝光后 1 天内付费，算这个素材有贡献

超过这个时间，就不再认这次触点。

它像超市小票的有效期：

- 在有效期内，可以追溯
- 超过有效期，就不算了

### 4.8 首次归因

首次归因，也叫首触归因，可以理解成：

**把功劳主要记给“第一次把用户带进来”的来源。**

比如一个用户：

1. 第一次通过 TikTok 广告进入
2. 后面又通过 Facebook 再回来
3. 最终完成注册

如果按首次归因，这个注册就记给 TikTok。

这种方式适合回答：

- 谁最会拉新
- 谁最先把用户带进来

### 4.9 末次归因

末次归因，也叫末触归因，可以理解成：

**把功劳主要记给“转化前最后一次影响用户的来源”。**

还是刚才那个用户：

1. 先从 TikTok 来
2. 后面又从 Facebook 回来
3. 最终注册

如果按末次归因，这个注册就记给 Facebook。

这种方式适合回答：

- 谁最擅长临门一脚
- 谁更能推动最终成交

### 4.10 多触点归因

多触点归因就是：

**不把功劳只给一个来源，而是分给多个来源。**

因为真实用户决策往往不是“一下子就成交”，而是多次接触后才转化。

例如：

- TikTok 先种草 40%
- Facebook 再提醒 30%
- 搜索回访再补一脚 30%

这样更接近真实过程，但实现和解释都更复杂。

### 4.11 线性归因

线性归因是多触点归因里最容易理解的一种：

**用户经过几个触点，就平均分配功劳。**

比如用户依次经历了：

1. 广告曝光
2. 广告点击
3. 回访打开

最终注册。

那每个触点各分三分之一的功劳。

优点是简单公平，缺点是容易“太平均”，看不出重点。

### 4.12 时间衰减归因

时间衰减归因的意思是：

**离转化越近的触点，分到的功劳越多。**

这很像现实中的印象分：

- 很久之前看过一次广告，影响可能已经淡了
- 转化前刚看到的一次促销提醒，影响更直接

所以这种方式会更偏向“最后阶段”的触点。

### 4.13 U 型归因

U 型归因是比较常见的一种折中方案。

它通常认为：

- 第一次触点很重要
- 最后一次触点也很重要
- 中间触点也有价值，但次一些

通俗理解就是：

- 第一位负责“把人带进门”
- 最后一位负责“把单签下来”
- 中间的人负责“持续影响”

### 4.14 激活

激活不是“来了就算”，而是“来了之后开始真正使用”。

比如短剧 App 里，一个用户安装后：

- 打开 App
- 浏览首页
- 看了第一集

这通常比单纯安装更像“激活”。

归因里常见的问题是：

- 某渠道安装很多，但激活很少
- 某渠道安装一般，但激活率很高

所以不能只看安装量，还要看激活质量。

### 4.15 留存

留存就是“用户后面有没有再回来”。

比如：

- 今天来了，明天又来，叫次日留存
- 今天来了，7 天后又来，叫 7 日留存

归因和留存结合后，问题就升级成：

- 哪个渠道不仅能带来注册，还能带来会回来的用户
- 哪个来源拉来的人只是首日热闹，后面很快流失

这比单看首日转化更有价值。

### 4.16 LTV

LTV 是生命周期价值，可以简单理解成：

**一个用户在后续一段时间里，总共能贡献多少钱。**

比如某渠道今天带来 100 个用户：

- 注册很多，不一定有价值
- 付费少，也不一定划算
- 但如果这些用户 30 天累计付费高，LTV 就高

所以归因分析最终通常不只看“谁带来了量”，还要看“谁带来了长期价值”。

## 5. 在当前项目里，为什么身份串联特别重要

当前项目有“游客 -> 登录/注册 -> 绑定第三方账号”的过程，所以归因会比普通网站更复杂。

最容易出问题的点有三个：

### 5.1 用户前后不是同一个ID

用户一开始可能只有：

- `device_id`
- `guest_user_id`

后面登录成功后，才变成：

- `current_user_id`

如果串不起来，就会出现：

- 渠道带来了游客，但注册记不到这个渠道头上
- 游客期看了很多剧，注册后像“换了一个人”

### 5.2 一个设备可能对应多个账号

同一台手机可能：

- 先生成一个游客账号
- 后来绑定一个正式账号
- 再切换到另一个账号

所以不能粗暴地把“设备”永远等同于“同一个用户”。

### 5.3 登录方式会影响归因解释

比如：

- Apple 登录用户付费更高
- TikTok 登录转化更快
- 游客升级绑定的留存优于直接注册

这时候 `provider`、`login_scene`、`register_scene` 就不仅是行为字段，也会影响归因结论。

## 6. 一个最容易理解的归因例子

假设用户行为如下：

1. 通过 `referrer = tiktok_ads` 第一次安装
2. 以游客身份进入 App，产生 `guest_user_id = G1001`
3. 看了 3 集短剧
4. 第二天通过 Apple 登录
5. 生成 `current_user_id = U9001`
6. 当天充值 30 元

那么可以有几种不同看法：

### 6.1 首次归因看法

- 注册记给 `tiktok_ads`
- 付费也记给 `tiktok_ads`

因为是它最早把用户带进来。

### 6.2 末次归因看法

如果用户在第二天回来时，是通过别的来源重新进入，例如 `source = push_recall`：

- 那么付费可能记给 `push_recall`

因为它更接近最终转化。

### 6.3 项目里的关键动作

要让这条链路分析成立，系统至少要能把下面几件事连上：

- `device_id`
- `guest_user_id`
- `current_user_id`
- `referrer`
- `source`
- 注册/登录/付费事件时间

只要这些链路缺一块，归因就会变得不稳定。

## 7. 看归因分析结果时，不要只盯“量”

做归因分析时，最容易犯的错误是只看“哪个渠道注册多”。

更合理的顺序通常是：

1. 看带量能力：谁带来的安装、注册多
2. 看转化效率：谁的注册率、付费率高
3. 看留存质量：谁带来的用户更愿意回来
4. 看长期价值：谁的 LTV 更高

也就是说：

**归因分析不是找“谁声音最大”，而是找“谁贡献最大”。**

## 8. 一句话总结

如果只用一句最通俗的话概括归因分析，可以这么记：

**归因分析，就是在用户走完整条路径后，判断“这份结果应该算谁的功劳”。**

在当前项目里，真正的难点不是定义渠道，而是把游客、设备、登录后账号这几段身份链路正确串起来，再去看 `referrer`、`source`、注册、付费、留存之间的关系。
