# 短剧APP埋点数据分析方案

本文档基于 `shortAAA`（用户中心）、`shortPortal`（C端接口）、`short-cms`（内容管理）项目代码分析，制定针对短剧业务的埋点数据分析方案。

用户增长相关口径已按实际后端逻辑修正。更细的代码链路、数据流和时序说明，见 [shortAAA_用户增长埋点逻辑分析.md](C:/Users/admin/Documents/src/xwl_bi/docs/埋点逻辑分析/shortAAA_用户增长埋点逻辑分析.md)。

## 1. 梳理用户属性、事件、事件属性

### 1.1 关键口径说明

1. App、Redis、Token 链路里实际使用的 `UserID` 是 `t_cust.login_name`，不是 `t_cust.id`。
2. 游客也有自己的 `UserID`，不绑定第三方账号也可以签到、分享、购买套餐、解锁剧集。
3. `register_success` 的业务含义是“首次绑定新的第三方身份成功”，不是“权限开通”。
4. 一个设备历史上可以对应多个账户，`dev:{device}:cust` 只表示当前默认指针。
5. 用户增长分析默认按“账户级”统计，不是按“自然人级”统计。

### 1.2 用户属性（User Properties）

用户属性用于描述用户的长期特征，在所有事件中通用。

| 属性名称 | 标识符 | 类型 | 描述/来源 |
| :--- | :--- | :--- | :--- |
| 账户级用户ID | `user_id` | String | 取 `t_cust.login_name`，游客和绑定用户都有 |
| 用户类型 | `cust_type` | Integer | 0: 游客, 1: 已绑定第三方账号；仅表示身份状态，不表示功能权限 |
| 是否绑定第三方账号 | `is_bound_provider` | Boolean | 当前行为发生时是否已绑定第三方账号 |
| 登录方式 | `provider` | String | Google, Apple, Facebook, TikTok, Email 等 |
| 第三方账号标识哈希 | `provider_id_hash` | String | `provider_id` 建议哈希后上报 |
| 设备ID | `device_id` | String | 对应 Header `X-Dev` |
| 设备品牌 | `device_brand` | String | 对应 Header `X-Brand` |
| 设备型号 | `device_model` | String | 对应 Header `X-model` |
| 系统版本 | `os_version` | String | 对应 Header `X-OsVersion` |
| 安装渠道 | `referrer` | String | 对应 Header `X-Referrer`，用于归因 |
| 流量来源 | `source` | String | 对应 Header `X-Source` 或游客建档时的 `source` |
| 账户余额 | `gold_coin` | Integer | 当前金币/积分余额 |
| VIP等级 | `vip_level` | Integer | 0: 普通, 1: VIP/分享达人 |
| 游客建档时间 | `visitor_created_time` | Timestamp | 首次生成游客 `UserID` 的时间 |
| 绑定注册时间 | `register_success_time` | Timestamp | 首次绑定第三方账号成功的时间 |
| 语言偏好 | `language` | String | 对应 Header `X-Lang` |

### 1.3 核心事件与属性（Events & Properties）

#### (1) 游客建档（`visitor_created`）

- 触发时机：`VisitorHandler` 新插入游客 `t_cust` 成功后。
- 典型属性：
  - `user_id`
  - `device_id`
  - `referrer`
  - `source`
  - `device_brand`
  - `device_model`
  - `os_version`
  - `country`
  - `city`

#### (2) 登录点击（`login_click`，前端可选）

- 触发时机：用户点击登录按钮，选择 Google、Facebook、TikTok、Apple 等方式时。
- 典型属性：
  - `user_id`
  - `provider`
  - `device_id`
  - `referrer`
  - `source`
  - `is_bound_provider`

#### (3) 登录成功（`login_success`）

- 触发时机：`Login4Dramactio` 登录成功并返回 token 后。
- 典型属性：
  - `guest_user_id`
  - `previous_user_id`
  - `current_user_id`
  - `provider`
  - `provider_id_hash`
  - `provider_account_exists`
  - `login_scene`
  - `user_id_changed`
  - `is_success`
  - `device_id`
  - `referrer`
  - `source`

`login_scene` 推荐值：

- `guest_bind_new_provider`
- `guest_login_existing_provider`
- `switch_to_new_provider`
- `login_existing_provider`

#### (4) 注册成功（`register_success`）

- 触发时机：本次登录对应的 `provider + provider_id` 在系统里首次出现，且写库成功。
- 典型属性：
  - `guest_user_id`
  - `previous_user_id`
  - `current_user_id`
  - `provider`
  - `provider_id_hash`
  - `register_scene`
  - `is_guest_conversion`
  - `is_new_account`
  - `user_id_changed`
  - `is_success`
  - `device_id`
  - `referrer`
  - `source`

`register_scene` 推荐值：

- `guest_bind_new_provider`
- `switch_to_new_provider`

说明：

- `register_success` 表示“新第三方身份首次建档/绑定成功”。
- 它不是前端独立的注册页面事件。
- 它也不是购买、解锁、分享、签到的前置条件。

#### (5) 应用启动（`app_launch`）

- 触发时机：用户冷启动或后台唤醒 App。
- 典型属性：
  - `launch_type`
  - `network_type`
  - `is_bound_provider`
  - `device_id`

#### (6) 浏览剧集详情（`view_drama_detail`）

- 触发时机：用户进入剧集详情页。
- 典型属性：
  - `drama_id`
  - `drama_title`
  - `drama_type`
  - `tags`
  - `is_exclusive`
  - `total_episodes`
  - `is_bound_provider`

#### (7) 观看视频（`watch_video`）

- 触发时机：用户开始播放某一集视频。
- 典型属性：
  - `drama_id`
  - `video_id`
  - `episode_index`
  - `duration`
  - `progress`
  - `is_paid_content`
  - `is_unlocked`
  - `is_bound_provider`

#### (8) 付费/充值（`purchase`）

- 触发时机：用户充值金币或购买会员。
- 典型属性：
  - `product_id`
  - `amount`
  - `currency`
  - `payment_method`
  - `pay_status`
  - `is_bound_provider`

#### (9) 解锁剧集（`unlock_episode`）

- 触发时机：用户消耗金币解锁付费集。
- 典型属性：
  - `drama_id`
  - `video_id`
  - `cost_coins`
  - `balance_after`
  - `is_bound_provider`

#### (10) 互动行为（`interaction`）

- 触发时机：点赞、收藏、分享。
- 典型属性：
  - `action_type`
  - `target_type`
  - `target_id`
  - `platform`
  - `is_bound_provider`

#### (11) 搜索（`search`）

- 触发时机：用户执行搜索。
- 典型属性：
  - `keyword`
  - `result_count`
  - `is_bound_provider`

## 2. 行为分析：事件分析（Event Analysis）

针对短剧 APP，核心关注三条线：

1. 用户增长与身份绑定
2. 内容消费与活跃
3. 商业化转化

### 2.1 推荐分析事件

1. 用户增长分析（User Growth）
   - 分析事件：`visitor_created`、`register_success`、`login_success`、`login_click`（可选）
   - 分析指标：每日新增游客数、每日新增注册账户数、每日游客转注册数、游客转注册率、登录成功用户数
   - 维度：`referrer`、`source`、`provider`、`register_scene`、`login_scene`、`os_version`
   - 目的：监控拉新质量、身份绑定意愿、登录结构

2. 视频完播率分析
   - 分析事件：`watch_video`
   - 分析指标：平均观看时长、完播次数占比
   - 维度：`episode_index`、`drama_id`、`referrer`
   - 目的：判断哪一集流失率最高，优化剧情节奏

3. 付费转化分析
   - 分析事件：`unlock_episode`
   - 分析指标：解锁次数、消耗金币总数、解锁用户数
   - 维度：`drama_id`、`is_exclusive`、`is_bound_provider`
   - 目的：识别最吸金的剧集，并比较游客与绑定用户的消费差异

4. 充值活跃度
   - 分析事件：`purchase`
   - 分析指标：充值总金额（GMV）、充值人数（PayUser）、支付成功率
   - 维度：`provider`、`referrer`、`is_bound_provider`
   - 目的：识别渠道质量和不同身份状态下的付费行为差异

说明：

- 不再使用 `Login/Register` 或 `Register/Login` 作为注册转化分析口径。
- `register_success` 代表身份绑定成功，不代表功能权限升级。

### 2.2 用户分群配置（User Segmentation）

- 高价值用户（High Value）：过去 30 天 `purchase` 总金额 > 50 美元
- 沉浸式用户（Engaged）：过去 7 天 `watch_video` 时长 > 60 分钟
- 潜在流失用户（Risk）：过去 7 天无 `app_launch` 事件，且曾有充值记录
- 游客高活跃用户：`is_bound_provider = 0` 且过去 7 天有 `watch_video` / `purchase` / `unlock_episode`
- 剧集偏好群：过去 30 天 `view_drama_detail` 中 `tags` 包含“逆袭”或“豪门”的用户

## 3. 行为分析：留存分析（Retention Analysis）

关注用户是否持续回到 APP 消费内容。

### 3.1 推荐分析场景

1. 新游客次留/七留
   - 初始事件：`visitor_created`
   - 回访事件：`app_launch` 或 `watch_video`
   - 目的：评估拉新渠道带来的游客质量

2. 新注册账户次留/七留
   - 初始事件：`register_success`
   - 回访事件：`app_launch` 或 `watch_video`
   - 目的：评估绑定第三方身份后的后续留存与内容消费

3. 付费用户留存
   - 初始事件：`purchase`
   - 回访事件：`unlock_episode` 或 `watch_video`
   - 目的：评估付费用户的生命周期价值（LTV）

说明：

- “游客是否转注册”更适合做漏斗分析，不建议放在留存分析里替代注册口径。

### 3.2 用户分群配置

- 渠道新游客：`referrer` 为指定渠道，且 `visitor_created_time` 在过去 7 天内
- 新注册账户：`register_success_time` 在过去 7 天内
- 首充用户：仅发生过 1 次 `purchase` 事件的用户

## 4. 行为分析：漏斗分析（Funnel Analysis）

分析用户从接触到核心转化的路径转化率。

### 4.1 推荐分析场景

1. 游客转注册漏斗（Guest to Register Conversion）
   - 步骤 1：`visitor_created`
   - 步骤 2：`login_click`（可选）
   - 步骤 3：`register_success`
   - 目的：分析游客到身份绑定成功的流失情况

2. 登录完成漏斗（Login Completion）
   - 步骤 1：`app_launch` 或 `view_profile`
   - 步骤 2：`login_click`
   - 步骤 3：`login_success`
   - 目的：分析登录入口、登录页和第三方授权的转化损耗

3. 核心付费漏斗（Paywall Conversion）
   - 步骤 1：`view_drama_detail`
   - 步骤 2：`watch_video`
   - 步骤 3：`unlock_episode`（触发解锁/充值意图）
   - 步骤 4：`purchase`
   - 步骤 5：`unlock_episode_success`
   - 目的：找到付费转化断点

说明：

- 不要做 `register_success -> purchase`、`register_success -> unlock_episode` 这类漏斗。
- 因为游客与绑定用户在功能权限上基本一致，`register_success` 不是消费或内容使用的门槛。

### 4.2 用户分群配置

- 卡点用户：完成了 `watch_video` 但未进入 `unlock_episode` 的用户
- 支付失败用户：触发了 `purchase` 但 `pay_status != success` 的用户

## 5. 行为分析：路径分析（Path Analysis）

探索用户的自然行为流向，发现非预期的行为路径。

### 5.1 推荐分析场景

1. 内容发现路径
   - 起始事件：`app_launch`
   - 观察路径：用户启动后是先去 `search`，还是浏览 `home_feed`，还是去 `history`
   - 目的：优化首页布局和推荐策略

2. 付费后行为路径
   - 起始事件：`purchase`
   - 观察路径：充值后是立即 `unlock_episode`，还是继续浏览其他剧集，或者直接退出
   - 目的：判断充值后的真实消费承接是否顺畅

### 5.2 用户分群配置

- 搜索型用户：起始路径主要为 `search`
- 推荐流用户：主要在 `home_feed` 产生交互

## 6. xwl_bi 用户增长推荐配置

### 6.1 每日新增游客数

- 分析模块：行为分析 -> 事件分析
- 事件：`visitor_created`
- 指标：用户数
- 时间粒度：按天
- 推荐分组：`referrer`、`source`、`os_version`

### 6.2 每日新增注册账户数

- 分析模块：行为分析 -> 事件分析
- 事件：`register_success`
- 指标：用户数
- 事件过滤：`is_success = 1`
- 时间粒度：按天
- 推荐分组：`register_scene`、`provider`、`referrer`、`source`

### 6.3 每日游客转注册数

- 分析模块：行为分析 -> 事件分析
- 事件：`register_success`
- 指标：用户数
- 事件过滤：`is_success = 1 AND is_guest_conversion = 1`
- 时间粒度：按天
- 推荐分组：`provider`、`referrer`、`source`

### 6.4 游客转注册率

推荐两种方式：

1. 快速趋势口径
   - 分子：`register_success` 用户数，过滤 `is_guest_conversion = 1`
   - 分母：`visitor_created` 用户数

2. 正式转化口径
   - 模块：漏斗分析
   - 漏斗：`visitor_created -> register_success`
   - 可选中间步骤：`login_click`

### 6.5 登录分析

- 登录成功用户数：`login_success` 用户数
- 登录结构分析：按 `login_scene`、`provider` 分组
- UserID 变化占比：按 `user_id_changed` 分组
- 游客登录已有账号规模：过滤 `login_scene = guest_login_existing_provider`

### 6.6 `xwl_distinct_id` 推荐取值

| 事件 | `xwl_distinct_id` |
| :--- | :--- |
| `visitor_created` | 当前游客 `vid` |
| `register_success` | `current_user_id` |
| `login_success` | `current_user_id` |
| `login_click` | 当前页面用户 ID；若是游客则用游客 `vid` |

说明：

- xwl_bi 用户数底层按 `COUNT(DISTINCT xwl_distinct_id)` 统计。
- 当前口径是账户级，不是自然人级。

## 7. 结论

1. 游客有自己的 `UserID`，且不绑定第三方账号也能完整使用主要功能。
2. `register_success` 的正确含义是“新第三方身份首次绑定成功”，不是权限升级。
3. `Register/Login` 不再适合作为注册转化率，应改为：
   - 每日新增游客数：`visitor_created`
   - 每日新增注册账户数：`register_success`
   - 每日游客转注册数：`register_success` 且 `is_guest_conversion = 1`
   - 游客转注册率：`visitor_created -> register_success`
4. 消费、解锁、分享、签到等功能分析应直接基于业务事件本身，并按 `is_bound_provider` 分组，不应强行挂在 `register_success` 之后。
