## 2026-04-30：Supastarter 依赖安装与验证

现象：

- `src/simpletrack-saas` 使用旧 Node 版本安装依赖时，Prisma preinstall 会失败。
- npm / pnpm 网络不稳定时，依赖下载会卡住或落到不稳定镜像源。
- `saas` type-check 在 Prisma generated client 未生成时会报 `packages/database/prisma/generated/client` 缺失。

原因：

- Supastarter 当前依赖的 Prisma 要求 Node 20.19+、22.12+ 或 24.0+，Node 22.10.0 不满足要求。
- 本地网络访问 npm registry 可能不稳定，需要走本机代理。
- `saas` app 依赖数据库包生成的 Prisma client。

处理方式：

```powershell
$env:HTTP_PROXY = "http://localhost:7897"
$env:HTTPS_PROXY = "http://localhost:7897"
$env:npm_config_proxy = "http://localhost:7897"
$env:npm_config_https_proxy = "http://localhost:7897"
$env:npm_config_registry = "https://registry.npmjs.org/"
```

```powershell
& "$env:USERPROFILE\AppData\Local\nvm\v24.1.0\npm.cmd" exec --yes pnpm@10.9.0 -- --config.manage-package-manager-versions=false --config.package-manager-strict=false --registry=https://registry.npmjs.org/ install --frozen-lockfile
```

```powershell
$env:DATABASE_URL = "postgresql://postgres:postgres@127.0.0.1:5432/simpletrack"
& "$env:USERPROFILE\AppData\Local\nvm\v24.1.0\npm.cmd" exec --yes pnpm@10.9.0 -- --config.manage-package-manager-versions=false --config.package-manager-strict=false --filter @repo/database run generate
```

验证结果：

- `pnpm install --frozen-lockfile` 已通过。
- `pnpm --filter @repo/database run generate` 已通过。
- `pnpm --filter saas --filter marketing --filter docs run type-check` 已通过。

后续规则：

- Windows 本地优先使用 Node 24.1.0 验证 `src/simpletrack-saas`。
- 网络失败先使用 `http://localhost:7897` 代理和官方 npm registry。
- 跑 `saas` type-check 前先确保 Prisma client 已生成。

## 2026-04-30：Windows 后台启动 Next dev 时环境变量引号被吃掉

现象：

- 使用 `Start-Process powershell -ArgumentList "-Command", "...$env:NEXT_PUBLIC_DOCS_URL=\"http://localhost:3002\"..."` 启动 dev server 时，日志出现：
  - `http://localhost:3002 : 无法将“http://localhost:3002”项识别为 cmdlet...`
  - `postgresql://... : 无法将“postgresql://...”项识别为 cmdlet...`

原因：

- Windows PowerShell 的多层 `-ArgumentList` / `-Command` 字符串转义容易丢失引号，导致环境变量赋值右侧被当成命令执行。

处理方式：

- 后台启动复杂 PowerShell 脚本时，优先使用 `-EncodedCommand`。
- 先构造完整脚本，再用 UTF-16LE 转 Base64：

```powershell
$script = @"
`$env:NEXT_PUBLIC_DOCS_URL = 'http://localhost:3002'
`$env:NEXT_PUBLIC_SAAS_URL = 'http://localhost:3000'
`$env:DATABASE_URL = 'postgresql://postgres:postgres@127.0.0.1:5432/simpletrack'
& '$npm' exec --yes pnpm@10.9.0 -- --filter marketing run dev
"@
$encoded = [Convert]::ToBase64String([Text.Encoding]::Unicode.GetBytes($script))
Start-Process powershell -ArgumentList @("-NoProfile", "-ExecutionPolicy", "Bypass", "-EncodedCommand", $encoded)
```

验证结果：

- marketing、docs、mail-preview 均成功监听对应端口。

后续规则：

- Windows 后台启动包含 URL、数据库连接串、引号的命令时，优先使用 `-EncodedCommand`，不要拼多层裸字符串。

## 2026-04-30：Playwright CLI 缺少浏览器二进制

现象：

- 使用 `pnpm exec playwright screenshot` 截图时报错：
  - `Executable doesn't exist at ... chromium_headless_shell...`
  - 提示需要执行 `playwright install`。

原因：

- Playwright 包已安装，但本机缺少对应 Chromium 浏览器二进制。

处理方式：

- 本轮没有直接执行 `playwright install`，因为这会下载并安装新的浏览器二进制。
- 改用 Codex in-app browser / Playwright MCP 完成本地页面截图验证。

验证结果：

- 已通过浏览器截图验证 marketing、docs quickstart、mail-preview。

后续规则：

- 如果后续需要稳定使用 shell Playwright 截图或 E2E，再安装 Playwright 浏览器二进制。
- 在安装前先确认这是当前任务必需项；非必需时优先使用已可用的 in-app browser / Playwright MCP。
