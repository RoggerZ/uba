- 如果记录里标注任务进度或验收状态，必须写明对应仓库的 git commit id；未提交时标注 `pending commit` 或 `未提交`，提交后回填真实 commit id。
- 如果问题影响 SimpleTrack 阶段计划或实施状态，再同步更新 `simpletrack/docs/实施决策/README.md`。
- 不要记录 token、cookie、私钥、账号密码等敏感信息。

## 2026-05-10：Gemini 和 Claude 在 legal 页面外部审查时失败

现象：

- 在 `src/simpletrack-saas` 为 marketing `privacy-policy` / `terms` 替换模板占位后，按 UI / 内容变更规则触发外部审查。
- Gemini CLI 产物记录到 `.omx/artifacts/gemini-legal-pages-20260510-081325.md`，失败核心信息是 `function_response.name: Name cannot be empty`。
- Claude CLI 产物记录到 `.omx/artifacts/claude-legal-pages-20260510-081325.md`，失败核心信息是 `API Error: 400 {"error":"1m 上下文已经全量可用，请启用 1m 上下文后重试","type":"error"}`。

原因：

- 当前本机 Gemini CLI / 上游 tool-call 打包链路在这类审查 prompt 上存在参数封装异常，返回了空的 `function_response.name`。
- 当前 Claude CLI / 提供方配置要求启用 `1m` 上下文能力，而本轮本机调用路径没有满足这个前置条件。
- 这两个问题都属于审查工具链问题，不是 `privacy-policy` / `terms` 文案本身的功能错误。

处理方式：

- 本轮不把 Gemini 或 Claude 的失败调用写成“审查通过”。
- 真实外部审查改由 DeepSeek 完成，并保留 `.omx/artifacts/deepseek-legal-pages-20260510-081325.md` 作为证据。
- 同时补跑 Codex native 子代理审查，实际返回 `PASS`，用于覆盖内容准确性和误导风险复核。
- 父仓修订记录和实施进度中明确写出 Gemini / Claude 未完成的真实原因。

验证结果：

- DeepSeek 已返回非阻塞建议，未指出当前 legal 页面有阻塞性错误。
- Codex native 子代理已真实返回 `PASS`。
- `simpletrack-saas` 本轮可提交证据来自 DeepSeek + Codex 子代理，而不是失败的 Gemini / Claude 调用。

后续避免方式：

- 涉及 UI / 内容外部审查前，先用最小命令验证 Gemini / Claude CLI 当前机器是否可用，再决定是否把它们纳入本轮强依赖。
- 如果 Gemini / Claude 继续失败，必须把失败原因写入卡壳记录和交付说明，并切换到仍然可执行的真实审查通道；不要把失败通道算作已完成。

## 2026-05-10：Supastarter 本地命令优先直接使用 `corepack pnpm`

现象：

- 在 `src/simpletrack-saas` 里为了启动 `saas` 子应用，曾走过 `npm exec --yes pnpm@... -- --filter saas ...` 这条路径。
- 这条路径在本机容易把 `--filter saas` 的参数层级解析错，报出 `ERR_PNPM_RECURSIVE_EXEC_FIRST_FAIL` 或 `Command "saas" not found`。

原因判断：

- `npm exec pnpm` 本身多了一层参数转发。
- 到了 monorepo + `--filter` + `exec` 这种组合时，本机命令链容易把过滤目标和子命令边界混在一起。

处理方式：

- 直接使用 `corepack pnpm`，不要再套 `npm exec pnpm`。
- 当前已验证可用的命令形态是：

```powershell
corepack pnpm --filter saas exec next dev --port 3005
```

- 需要 type-check、test、lint 时，也优先保持同样的 `corepack pnpm --filter ...` 风格。

当前结论：

- Supastarter 本地工作副本在这台机器上，`corepack pnpm` 是默认优先路径。
- `npm exec pnpm` 不是禁止使用，但不再作为本仓首选命令形态。

后续规则：

- 后续在 `simpletrack-saas`、`marketing`、`docs`、`mail-preview` 这些子应用执行命令时，优先先写成 `corepack pnpm --filter ...`。
- 如果命令涉及 route group 路径，还要同时遵守本文件已有的“不要把带括号的完整路径直接传给 pnpm/cmd”规则。

## 2026-05-10：本机 Playwright 本地截图优先使用系统 Chrome

现象：

- 为了做 `src/simpletrack-saas` 的本地 UI 截图和评审，曾先往“安装 Playwright 自带 Chromium”这条路试探。
- 这在当前机器上属于额外噪声，因为系统已经有 Chrome，可直接被 Playwright 调起。

原因判断：

- 本机任务目标只是对本地页面做截图和页面态核验，不是做完全隔离的 CI 浏览器基线。
- 直接走系统 Chrome 更贴近当前机器的真实浏览器环境，也能绕开“未安装 bundled Chromium”的额外步骤。

处理方式：

- 本机本地 UI 截图优先尝试：

```powershell
npx playwright screenshot --channel chrome http://127.0.0.1:3005 output/playwright/simpletrack-saas-3005-root.png
```

- 只有当系统 Chrome 不可用、版本不兼容，或者任务明确要求 Playwright 自带浏览器时，才考虑安装 bundled Chromium。

当前结论：

- 当前机器对本地 SimpleTrack UI 截图，优先用 Playwright + 系统 Chrome。
- 这条结论适用于本地截图、快速页面核验和 UI 评审，不等于正式 E2E 永久放弃 bundled browser。

后续规则：

- 做本地 UI 评审前，先判断是否只是截图/交互核验；如果是，优先系统 Chrome。
- 只有在 CI 对齐、跨机一致性或 Playwright browser feature 明确依赖时，再额外安装 Playwright 自带浏览器。

- 如果 Gemini 再次报 `function_response.name` 或类似上游打包错误，立即切换到可执行的真实审查通道，不要反复空转。

## 2026-05-09：子仓 `.git` 目录 ACL 阻止当前沙箱提交

现象：

- 在 `src/simpletrack-saas` 提交 Goal 并发唯一约束修复时失败：
  - `fatal: Unable to create 'C:/Users/admin/Documents/src/uba/src/simpletrack-saas/.git/index.lock': Permission denied`
- 随后检查到父仓 `.git`、`src/analytics-service/.git`、`src/analytics-core/.git` 也存在类似显式 `DENY` ACL。
- 当前进程身份为 `desktop-m23j16k\codexsandboxoffline`。
- `icacls src/simpletrack-saas/.git` 显示存在显式 `DENY` ACE，当前沙箱无法写 Git index。
- 2026-05-09 再次尝试只 stage `AGENTS.md` 时仍失败：
  - `fatal: Unable to create 'C:/Users/admin/Documents/src/uba/.git/index.lock': Permission denied`
- 当前版本标记：父仓 commit `674c1de` + 工作区未提交改动，`AGENTS.md` 进度 commit id 规则为 `pending commit`。

原因：

- 代码文件位于可写工作区内，但子仓 Git 元数据目录存在 Windows ACL 拒绝规则。
- 这会阻止父仓和子仓执行 `git add`、`git commit --amend`、`git commit` 等需要写 `.git/index.lock` 的操作。

处理方式：

- 本轮不能用管理员权限修 ACL，因为当前会话不允许提权。
- 继续完成代码修改、文档更新和验证；提交/推送需要在具备 `.git` 写权限的 PowerShell 或修复 ACL 后执行。
- 不要把这个问题误判成测试失败或代码生成失败。

验证结果：

- `go test`、`go vet`、`golint`、`go doc`、SaaS target tests、type-check 和 lint 已在当前工作区验证通过。
- 仅 Git 元数据写入被 ACL 阻塞。

后续规则：

- 如果再次出现 `index.lock: Permission denied`，先检查 `.git` 目录 ACL，而不是反复重跑提交命令。
- 修复方式应在管理员 PowerShell 中调整子仓 `.git` ACL，或用具备写权限的普通终端执行提交。

## 2026-05-09：代码审查子代理超时或 429 不能算完成

现象：

- P1 Goal 最小闭环进入代码审查时，架构审查子代理实际返回 `WATCH` 结论，但另一路代码质量审查子代理出现超时或 `429 Too Many Requests`。
- 当前版本标记：`analytics-core` commit `b820f7e`，`analytics-service` commit `f60ac59`，`simpletrack-saas` commit `e86ba88`，父仓文档 commit `b672e96`。

原因：

- 子代理启动成功不代表审查完成；只有拿到包含文件、行号、结论或明确无阻塞发现的返回，才算真实审查证据。
- 429 或 timeout 是工具执行失败，不是代码质量通过。

处理方式：

- 不能把“子代理超时、主代理本地复核”写成“子代理已审查”。
- 对超时或 429 的审查通道，必须重新发起更窄范围的审查，或明确记录“该通道未完成”。
- 已在 `AGENTS.md` 固化规则：代码审查必须真实完成才能声明完成。

验证结果：

- 架构审查通道已真实返回 `WATCH`，关注 Goal readback contract 复制、单页最多 25 个 Goal 的 count fan-out、readback policy 控制面闭环和内部读权限粒度。
- 代码质量复审后来已真实返回 `PASS`，覆盖 `analytics-core` Goal count read side、`analytics-service` `/v1/goals`、`simpletrack-saas` Goal readback 与 `P2002` duplicate-event 映射；仍保留的风险是 Goals 页面按 goal 数量线性 fan-out readback，尚未做 live browser / e2e 全链路验证。

后续避免方式：

- 提交说明、修订记录和最终回复只引用真实返回的审查结论。
- 如果审查代理不稳定，先拆小审查范围，必要时换用 verifier 子代理，但仍必须等待实际返回。

  - 如果 ClickHouse 属性行已经写入成功，但 MySQL 属性状态 `Commit` 失败，队列会重试；若下一次自动 reclaim，就可能重复写同一批属性行。

现象：

  - `Resolved identifier 'e.event_id' in parent scope to expression 'e.event_id' with correlated column 'event_id'`
- 触发 SQL 形态类似：

```sql
SELECT ...
FROM `events_xxx` AS e
WHERE EXISTS (
  SELECT 1
  FROM `events_xxx_properties` AS p
  WHERE p.event_id = e.event_id
)
```

原因：

- GORM dry-run 只能证明 SQL 被构建出来，不能证明目标 ClickHouse 版本能执行该 SQL。
- 当前本地 ClickHouse 25.3 对这种 correlated `EXISTS` 子查询和外层表 alias 的解析不兼容，子查询里引用 `e.event_id` 会被拒绝。
- 这个问题不是 property writer 写入失败，也不是数据没写进去，而是读侧属性过滤 SQL 形态不适合当前 ClickHouse。

处理方式：

- 将属性过滤从 correlated `EXISTS` 改为 ClickHouse 可执行的 tuple `IN (SELECT ...)` 半连接形态。
- 新 SQL 思路：

```sql
WHERE (tenant_id, project_id, source_id, event_id) IN (
  SELECT tenant_id, project_id, source_id, event_id
  FROM `events_xxx_properties`
  WHERE tenant_id = ?
    AND project_id = ?
    AND source_id = ?
    AND property_scope = ?
    AND property_name = ?
    AND property_type = ?
    AND string_value = ?
)
```

- 属性名、scope、value type 和属性值仍然走 allowlist + 绑定参数，只有物理属性表名由 ClickHouse `TableRouter` 派生。

验证结果：

- `go test ./internal/storage/clickhouse ./internal/e2e` 通过。
- 冷启动后执行 `ANALYTICS_CORE_E2E=1 go test -count=1 ./internal/e2e -v` 通过。
- 验证结束后已执行 `docker compose down` 停止 Redis/MySQL/ClickHouse。

后续规则：

- ClickHouse 查询 builder 的 dry-run 测试只能作为 SQL 结构检查；新增非平凡 SQL 形态时必须补真实 ClickHouse e2e。
- 避免在 ClickHouse 查询中依赖 correlated subquery 访问外层 alias；优先使用 tuple `IN`、显式 JOIN 或预聚合表方案。
- 属性过滤、漏斗、留存、分群等后续查询只要引入子查询，都要在本地 ClickHouse 版本上复验执行，而不能只看 GORM dry-run 输出。

## 2026-04-30：Windows 后台启动 Next dev 时环境变量引号被吃掉

现象：

- 使用 `Start-Process powershell -ArgumentList "-Command", "...$env:NEXT_PUBLIC_DOCS_URL=\"http://localhost:3002\"..."` 启动 dev server 时，日志出现：
  - `http://localhost:3002 : 无法将“http://localhost:3002”项识别为 cmdlet...`
  - `postgresql://... : 无法将“postgresql://...”项识别为 cmdlet...`

原因：

- Windows PowerShell 的多层 `-ArgumentList` / `-Command` 字符串转义容易丢失引号，导致环境变量赋值右侧被当成命令执行。

处理方式：

- 后台启动复杂 PowerShell 脚本时，优先使用 `-EncodedCommand`。
- 先构造完整脚本，再用 UTF-16LE 转 Base64：

```powershell
$script = @"
`$env:NEXT_PUBLIC_DOCS_URL = 'http://localhost:3002'
`$env:NEXT_PUBLIC_SAAS_URL = 'http://localhost:3000'
`$env:DATABASE_URL = 'postgresql://postgres:postgres@127.0.0.1:5432/simpletrack'
& '$npm' exec --yes pnpm@10.9.0 -- --filter marketing run dev
"@
$encoded = [Convert]::ToBase64String([Text.Encoding]::Unicode.GetBytes($script))
Start-Process powershell -ArgumentList @("-NoProfile", "-ExecutionPolicy", "Bypass", "-EncodedCommand", $encoded)
```

验证结果：

- marketing、docs、mail-preview 均成功监听对应端口。

后续规则：

- Windows 后台启动包含 URL、数据库连接串、引号的命令时，优先使用 `-EncodedCommand`，不要拼多层裸字符串。

## 2026-05-10：Next dev 在 `127.0.0.1` 下 hydration 失效，Better Auth 本地 loopback origin 不一致

现象：

- 使用 `http://127.0.0.1:3005/login` 打开 `src/simpletrack-saas/apps/saas` 时，登录页会退化成原生表单提交，URL 变成 `?email=...&password=...`。
- 密码显隐按钮、登录模式切换等 client 交互也一并失效，看起来像“整页没有 hydration”。
- 即使切回 `localhost`，如果后台重启 dev server 时没有从 monorepo 根目录带上 `.env.local`，`/api/auth/get-session` 和 `/api/auth/sign-in/email` 会直接 `500`，日志里只有 `DATABASE_URL is not set`。

原因：

- Next.js 16 dev server 默认会拦截未允许来源的 dev 资源访问。当前 `NEXT_PUBLIC_SAAS_URL` 以 `localhost` 作为 base URL 时，直接从 `127.0.0.1` 访问会触发 `/_next/webpack-hmr` 被阻断，客户端脚本没有真正接上。
- Better Auth 的 `trustedOrigins` 原先只信任 `appUrl`，本地用 `localhost` 和 `127.0.0.1` 混着访问时，auth origin 也会不一致。
- 直接在 `apps/saas` 目录后台启动 `next dev` 不会自动继承 `src/simpletrack-saas/.env.local`，需要显式用 `dotenv` 从仓库根目录喂环境变量。

处理方式：

- 在 `simpletrack-saas` commit `9ebe9f4` 中：
  - `apps/saas/next.config.ts` 改为通过共享 helper 推导 `allowedDevOrigins`，允许当前 loopback 别名的对端 hostname。
  - `packages/auth/auth.ts` 改为通过 `getLoopbackTrustedOrigins(appUrl)` 推导 Better Auth 本地 trusted origins，但在 `production` 仍只保留配置的 `appUrl`。
  - 新增 `packages/utils/lib/loopback-trusted-origins.ts` 与对应 Vitest 回归测试。
- 本地重启 dev server 时，固定使用：

```powershell
$env:NEXT_PUBLIC_SAAS_URL = "http://localhost:3005"
corepack pnpm exec dotenv -e ../../.env.local -- next dev --port 3005
```

验证结果：

- `corepack pnpm --filter saas test -- loopback-trusted-origins.test.ts` 通过。
- `corepack pnpm --filter saas type-check` 通过。
- Playwright + 系统 Chrome 验证 `http://localhost:3005/login` 和 `http://127.0.0.1:3005/login` 均可完成登录并跳转到 `/`，且没有 `>=400` 响应。

后续规则：

- 验证 `src/simpletrack-saas` 本地登录流时，默认同时覆盖 `localhost` 和 `127.0.0.1` 两条 loopback 入口。
- 后台启动 `apps/saas` 的 Next dev server 时，优先使用 `corepack pnpm exec dotenv -e ../../.env.local -- next dev --port <port>`，不要直接裸跑 `next dev`。
- 如果再次看到登录页退化成 URL query 提交，先检查 `/_next/webpack-hmr` 是否被 blocked，再检查 Better Auth `trustedOrigins` 和当前 `NEXT_PUBLIC_SAAS_URL` 是否一致。
