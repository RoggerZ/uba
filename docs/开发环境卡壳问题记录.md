## 2026-05-02：ClickHouse 属性过滤 correlated EXISTS 不支持外层 alias

现象：

- 为 `analytics-core` 的 Events 查询增加 property filter 后，dry-run query builder 单元测试通过，但真实 ClickHouse e2e 失败。
- 失败发生在 `TestPropertyBatchWriterToClickHouse` 通过 `EventReader.ListEvents` 查询属性过滤结果时。
- 典型错误：
  - `Resolved identifier 'e.event_id' in parent scope to expression 'e.event_id' with correlated column 'event_id'`
- 触发 SQL 形态类似：

```sql
SELECT ...
FROM `events_xxx` AS e
WHERE EXISTS (
  SELECT 1
  FROM `events_xxx_properties` AS p
  WHERE p.event_id = e.event_id
)
```

原因：

- GORM dry-run 只能证明 SQL 被构建出来，不能证明目标 ClickHouse 版本能执行该 SQL。
- 当前本地 ClickHouse 25.3 对这种 correlated `EXISTS` 子查询和外层表 alias 的解析不兼容，子查询里引用 `e.event_id` 会被拒绝。
- 这个问题不是 property writer 写入失败，也不是数据没写进去，而是读侧属性过滤 SQL 形态不适合当前 ClickHouse。

处理方式：

- 将属性过滤从 correlated `EXISTS` 改为 ClickHouse 可执行的 tuple `IN (SELECT ...)` 半连接形态。
- 新 SQL 思路：

```sql
WHERE (tenant_id, project_id, source_id, event_id) IN (
  SELECT tenant_id, project_id, source_id, event_id
  FROM `events_xxx_properties`
  WHERE tenant_id = ?
    AND project_id = ?
    AND source_id = ?
    AND property_scope = ?
    AND property_name = ?
    AND property_type = ?
    AND string_value = ?
)
```

- 属性名、scope、value type 和属性值仍然走 allowlist + 绑定参数，只有物理属性表名由 ClickHouse `TableRouter` 派生。

验证结果：

- `go test ./internal/storage/clickhouse ./internal/e2e` 通过。
- 冷启动后执行 `ANALYTICS_CORE_E2E=1 go test -count=1 ./internal/e2e -v` 通过。
- 验证结束后已执行 `docker compose down` 停止 Redis/MySQL/ClickHouse。

后续规则：

- ClickHouse 查询 builder 的 dry-run 测试只能作为 SQL 结构检查；新增非平凡 SQL 形态时必须补真实 ClickHouse e2e。
- 避免在 ClickHouse 查询中依赖 correlated subquery 访问外层 alias；优先使用 tuple `IN`、显式 JOIN 或预聚合表方案。
- 属性过滤、漏斗、留存、分群等后续查询只要引入子查询，都要在本地 ClickHouse 版本上复验执行，而不能只看 GORM dry-run 输出。

## 2026-05-02：analytics-core e2e 冷启动 ClickHouse native handshake EOF

现象：

- 在 `src/analytics-core` 执行依赖冷启动后立即运行 `go test ./internal/e2e -v`，测试可能失败在 ClickHouse native 连接阶段。
- 典型错误类似：`ping native clickhouse failed: EOF` 或 native handshake EOF。
- 同一轮输出里还可能看到 MySQL 驱动打印的 `[mysql] ... unexpected EOF`。这类日志出现在容器刚启动但 MySQL 尚未完成握手时，不等同于最终测试失败，最终结果以 `go test` 的退出码和 `PASS/FAIL` 为准。

原因：

- `docker compose up -d` 只表示容器已启动，不代表 Redis、MySQL、ClickHouse 已经完成协议握手、认证和数据库初始化。
- 旧版 e2e 测试在 compose 启动后马上打开 Redis/MySQL/ClickHouse，并且 ClickHouse native 只做一次 `Ping`；ClickHouse TCP 端口可能已经开放，但 native server 还没有准备好完成握手，因此出现 EOF。
- 原测试上下文只有 20 秒，覆盖不了 Windows + Docker Desktop 冷启动时偶发的慢启动窗口。

处理方式：

- 在 `src/analytics-core/internal/e2e/pipeline_test.go` 为 Redis、MySQL、ClickHouse native、ClickHouse GORM 连接增加 readiness retry loop。
- 统一使用：
  - `e2eDependencyTimeout = 90 * time.Second`
  - `e2eDependencyPollInterval = 500 * time.Millisecond`
- ClickHouse native 连接失败或 `Ping` 失败时关闭当前连接并重试，避免把启动期 EOF 当成真实写入链路失败。
- MySQL 连接先用 `database/sql` 做 readiness 探活，探活成功后再创建生产路径使用的 GORM 连接。
- 顺手补了 `TestPropertyBatchWriterToClickHouse`，用真实 ClickHouse 表验证 `PropertyBatchWriter` 能把 `event_properties` / `user_properties` 展开记录写入并查询回来。

验证方式：

```powershell
cd C:\Users\1\Documents\git\uba\src\analytics-core
docker compose down
docker compose up -d
$env:ANALYTICS_CORE_E2E = '1'
go test -count=1 ./internal/e2e -v
docker compose down
```

本次验证结果：

- 冷启动后 `TestCollectToRealtimeAndEventsPipeline` 通过。
- 冷启动后 `TestPropertyBatchWriterToClickHouse` 通过。
- 验证结束后已执行 `docker compose down` 停止 Redis/MySQL/ClickHouse。

后续规则：

- 以后排查 e2e 数据库连接失败时，先确认是否是冷启动 readiness 问题，不要把第一次 EOF 直接当成 ClickHouse 写入代码问题。
- 涉及 Docker Compose 依赖的 e2e 必须用显式 readiness retry，并用 `go test -count=1` 做冷启动复验，避免缓存掩盖问题。
- 只要启动/复验过 compose，结束时必须执行 `docker compose down`，不要让本地依赖容器长期挂着。

## 2026-05-01：analytics-core Docker Compose Redis 端口冲突

现象：

- 在 `src/analytics-core` 执行 `docker compose up -d` 时，MySQL 和 ClickHouse 镜像拉取并启动成功，但 Redis 容器启动失败。
- Docker 报错：`Bind for 0.0.0.0:6379 failed: port is already allocated`。

原因：

- 本机已有 Redis 或其他服务占用了默认 `6379` 端口。
- analytics-core 的 compose 初版直接映射 `6379:6379`，不适合在已有 Redis 的开发机上作为默认配置。

处理方式：

- 已执行 `docker compose down` 清理本次部分启动的容器。
- 已将 `src/analytics-core/docker-compose.yml` 默认端口改为更高位端口，并支持环境变量覆盖：
  - Redis：`26379`
  - RedisInsight：`28001`
  - MySQL：`23306`
  - ClickHouse HTTP：`28123`
  - ClickHouse native TCP：`29000`
- 已同步更新 `src/analytics-core/README.md` 的本地运行说明。
- 第二次启动时发现 `13306` 也被占用，因此默认端口继续上移到当前端口组。
- Redis Stream 集成测试使用 `localhost:26379` 时出现过一次 `i/o timeout`，改用 `127.0.0.1:26379` 后稳定通过。

验证结果：

- `docker compose config` 通过。
- `go test ./...`、`go vet ./...`、`git diff --check` 通过。

后续规则：

- analytics-core 本地运行依赖默认使用高位端口，避免覆盖开发机已有数据库。
- 如需使用标准端口，通过 `ANALYTICS_CORE_REDIS_PORT` 等环境变量显式覆盖。
- Windows + Docker Desktop 下运行 Redis 集成测试优先使用 `127.0.0.1:26379`，避免 `localhost` 解析带来的偶发连接问题。

## 2026-04-30：Supastarter 依赖安装与验证

现象：

- `src/simpletrack-saas` 使用旧 Node 版本安装依赖时，Prisma preinstall 会失败。
- npm / pnpm 网络不稳定时，依赖下载会卡住或落到不稳定镜像源。
- `saas` type-check 在 Prisma generated client 未生成时会报 `packages/database/prisma/generated/client` 缺失。

原因：

- Supastarter 当前依赖的 Prisma 要求 Node 20.19+、22.12+ 或 24.0+，Node 22.10.0 不满足要求。
- 本地网络访问 npm registry 可能不稳定，需要走本机代理。
- `saas` app 依赖数据库包生成的 Prisma client。

处理方式：

```powershell
$env:HTTP_PROXY = "http://localhost:7897"
$env:HTTPS_PROXY = "http://localhost:7897"
$env:npm_config_proxy = "http://localhost:7897"
$env:npm_config_https_proxy = "http://localhost:7897"
$env:npm_config_registry = "https://registry.npmjs.org/"
```

```powershell
& "$env:USERPROFILE\AppData\Local\nvm\v24.1.0\npm.cmd" exec --yes pnpm@10.9.0 -- --config.manage-package-manager-versions=false --config.package-manager-strict=false --registry=https://registry.npmjs.org/ install --frozen-lockfile
```

```powershell
$env:DATABASE_URL = "postgresql://postgres:postgres@127.0.0.1:5432/simpletrack"
& "$env:USERPROFILE\AppData\Local\nvm\v24.1.0\npm.cmd" exec --yes pnpm@10.9.0 -- --config.manage-package-manager-versions=false --config.package-manager-strict=false --filter @repo/database run generate
```

验证结果：

- `pnpm install --frozen-lockfile` 已通过。
- `pnpm --filter @repo/database run generate` 已通过。
- `pnpm --filter saas --filter marketing --filter docs run type-check` 已通过。

后续规则：

- Windows 本地优先使用 Node 24.1.0 验证 `src/simpletrack-saas`。
- 网络失败先使用 `http://localhost:7897` 代理和官方 npm registry。
- 跑 `saas` type-check 前先确保 Prisma client 已生成。

## 2026-04-30：Windows 后台启动 Next dev 时环境变量引号被吃掉

现象：

- 使用 `Start-Process powershell -ArgumentList "-Command", "...$env:NEXT_PUBLIC_DOCS_URL=\"http://localhost:3002\"..."` 启动 dev server 时，日志出现：
  - `http://localhost:3002 : 无法将“http://localhost:3002”项识别为 cmdlet...`
  - `postgresql://... : 无法将“postgresql://...”项识别为 cmdlet...`

原因：

- Windows PowerShell 的多层 `-ArgumentList` / `-Command` 字符串转义容易丢失引号，导致环境变量赋值右侧被当成命令执行。

处理方式：

- 后台启动复杂 PowerShell 脚本时，优先使用 `-EncodedCommand`。
- 先构造完整脚本，再用 UTF-16LE 转 Base64：

```powershell
$script = @"
`$env:NEXT_PUBLIC_DOCS_URL = 'http://localhost:3002'
`$env:NEXT_PUBLIC_SAAS_URL = 'http://localhost:3000'
`$env:DATABASE_URL = 'postgresql://postgres:postgres@127.0.0.1:5432/simpletrack'
& '$npm' exec --yes pnpm@10.9.0 -- --filter marketing run dev
"@
$encoded = [Convert]::ToBase64String([Text.Encoding]::Unicode.GetBytes($script))
Start-Process powershell -ArgumentList @("-NoProfile", "-ExecutionPolicy", "Bypass", "-EncodedCommand", $encoded)
```

验证结果：

- marketing、docs、mail-preview 均成功监听对应端口。

后续规则：

- Windows 后台启动包含 URL、数据库连接串、引号的命令时，优先使用 `-EncodedCommand`，不要拼多层裸字符串。

## 2026-04-30：Playwright CLI 缺少浏览器二进制

现象：

- 使用 `pnpm exec playwright screenshot` 截图时报错：
  - `Executable doesn't exist at ... chromium_headless_shell...`
  - 提示需要执行 `playwright install`。

原因：

- Playwright 包已安装，但本机缺少对应 Chromium 浏览器二进制。

处理方式：

- 本轮没有直接执行 `playwright install`，因为这会下载并安装新的浏览器二进制。
- 改用 Codex in-app browser / Playwright MCP 完成本地页面截图验证。

验证结果：

- 已通过浏览器截图验证 marketing、docs quickstart、mail-preview。

后续规则：

- 如果后续需要稳定使用 shell Playwright 截图或 E2E，再安装 Playwright 浏览器二进制。
- 在安装前先确认这是当前任务必需项；非必需时优先使用已可用的 in-app browser / Playwright MCP。
