## ⚠️ 核心限制条款

### 1. AGPL-3.0的核心要求

#### 什么是AGPL？

AGPL（Affero GPL）是GPL的加强版，专门针对**网络服务**设计。

**核心特点**：
```
GPL：只要不分发软件，可以不开源
AGPL：即使通过网络提供服务，也必须开源
```

#### AGPL的主要义务

**✅ 你可以做的**：
- 自由使用软件
- 修改源代码
- 分发软件
- 商业使用

**❌ 你必须做的**：
1. **开源修改后的代码**
   - 如果你修改了Countly并提供网络服务
   - 必须向用户提供完整的源代码
   - 包括你的所有修改

2. **保持相同许可证**
   - 修改后的版本必须使用AGPL-3.0
   - 不能改用其他许可证

3. **提供源代码访问**
   - 必须在显著位置提供源代码下载链接
   - 用户可以通过网络获取完整源代码

### 2. Countly的额外限制（Modified Section 7）

**这是最关键的部分！**

#### 品牌和Logo限制

```
❌ 禁止行为：

1. 不能移除Countly的Logo
   - 源代码中的Logo
   - 用户界面中的Logo
   - 任何公开可见的界面

2. 不能移除Countly的品牌标识
   - 启动画面
   - 仪表板
   - 软件头部
   - 所有软件界面

3. 不能使用其他品牌替代
   - 不能用自己的品牌替换Countly
   - 不能修改Countly的名称
   - 不能更改Countly的链接

4. 必须保持原样
   - 所有名称、链接、Logo必须与原始版本一致
   - 不能做任何修改
```

**官方原文**：
> "You cannot remove any Countly logo or branding either from source code or user interface and/or any other interface that can be publicly seen by end user."

## 🚨 法律风险分析

### 场景1：直接使用Countly

**你的计划**：部署Countly，不做修改

**法律风险**：✅ **无风险**

**说明**：
- 完全合法
- 无需开源
- 无需支付费用
- 可以商业使用

**限制**：
- 必须保留Countly的Logo和品牌
- 不能声称是你自己的产品

### 场景2：修改Countly代码

**你的计划**：修改Countly功能，提供服务

**法律风险**：⚠️ **高风险**

**AGPL要求**：
```
✅ 必须做：
1. 开源所有修改的代码
2. 在界面上提供源代码下载链接
3. 使用AGPL-3.0许可证
4. 保留Countly的Logo和品牌

❌ 不能做：
1. 闭源你的修改
2. 隐藏源代码
3. 移除Countly品牌
4. 声称是你的原创产品
```

**违规后果**：
- 版权侵权诉讼
- 强制开源
- 赔偿损失
- 停止服务

### 场景3：基于Countly开发新产品

**你的计划**：参考Countly，开发类似功能的产品

**法律风险**：⚠️ **中等风险**

**关键问题**：
```
❓ 是否使用了Countly的代码？
   - 是 → 必须遵守AGPL
   - 否 → 无需遵守AGPL

❓ 是否复制了Countly的代码？
   - 是 → 版权侵权
   - 否 → 合法

❓ 功能相似是否侵权？
   - 功能和想法不受版权保护
   - 但实现代码受保护
```

**合法方式**：
- ✅ 参考功能设计
- ✅ 学习架构思路
- ✅ 自己编写代码
- ❌ 复制粘贴代码
- ❌ 修改后使用代码

### 场景4：Fork Countly并商业化

**你的计划**：Fork Countly，修改后销售

**法律风险**：⚠️ **高风险**

**AGPL要求**：
```
✅ 可以商业化，但必须：
1. 开源所有代码
2. 保留Countly品牌
3. 使用AGPL许可证
4. 提供源代码下载

实际问题：
- 客户可以免费获取源代码
- 客户可以自己部署
- 难以形成商业壁垒
- 必须保留Countly品牌（影响品牌建设）
```
